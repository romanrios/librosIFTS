Introducción


Inicia una nueva semana y el tema se centrará en los elementos de un caso de uso y cómo realizar la gráfica. Veremos cómo cada escenario describe una secuencia de eventos. Cada secuencia se inicia por una persona, otro sistema, una parte del hardware o por el paso del tiempo. A las entidades que inician secuencias se les conoce como actores. El resultado de la secuencia debe ser algo utilizable, ya sea por el actor que la inició, o por otro actor. 

El Caso de uso es una excelente herramienta para estimular a que los usuarios potenciales hablen de un sistema, desde sus propios puntos de vista. No siempre es fácil para los usuarios explicar cómo pretenden utilizar un sistema.

Es posible volver a utilizar Casos de uso. Una forma (inclusión) es utilizar los pasos de un Caso de uso como parte de la secuencia de pasos de otro Caso de uso. Otra forma (extensión) es crear un nuevo Caso de uso mediante la adición de pasos a un Caso de uso existente.

Recorré los capítulos y subcapítulos a la derecha todas las veces que necesites para registrar, tomar nota, revisar las actividades propuestas.


¡Comenzamos!


¿Cómo se compone una gráfica de CdU? 


Una gráfica permitirá mostrar los Casos de uso a los usuarios para que ellos le puedan dar mayor información. Es un hecho que los usuarios, con frecuencia, saben más de lo que dicen: el Caso de uso ayuda a romper el hielo. A su vez, una representación visual le ayuda a combinar los diagramas de Casos de uso con otro tipo de diagramas.

Una de las finalidades del proceso de análisis de un sistema es generar una colección de Casos de uso. La idea es tener la posibilidad de catalogar y hacer referencia a esta colección, que sirve como el punto de vista de los usuarios acerca del sistema. Cuando llegue el momento de actualizar el sistema, el catálogo de Casos de uso funcionará como un fundamento para obtener los requerimientos de la actualización.

¿Cómo representar un Cdu y actores?


Un Caso de uso es representado por una elipse, su nombre puede ser localizado en el exterior o interior de la elipse.

Un actor es la entidad responsable de utilizar un Caso de uso. El actor representa un rol dentro del sistema, por ejemplo, un conjunto de personas, un sistema informático, una organización, etc. Derivado de que un actor representa un rol, dentro de un rol se pueden agrupar un conjunto de entidades, incluso una entidad puede jugar diferentes roles dentro de un sistema, implicando una entidad ser representada por diferentes actores. 

El actor no forma parte del sistema, pero se modela para conocer la interacción del sistema con sus usuarios. 

La representación gráfica de un actor es un muñeco con un nombre.



¿Cómo se representan las conexiones?


La asociación es la relación entre un actor, un Caso de uso, entre Casos de uso y entre actores. La asociación entre actores pocas veces se representa por no ser de importancia en la mayor parte de los análisis.

Para mostrar el uso de las asociaciones, se muestra un actor usuario con falla, el cual realiza un reporte telefónico de falla al actor recepción de fallas, el cual, a su vez, captura, en el sistema, el registro de la falla, luego la falla es enviada al módulo “aviso de fallas” para su asignación, vía correo electrónico, al actor atención de fallas. El actor de recepción de fallas se muestra con una asociación direccionada al Caso de uso de registro de falla, indicando el sentido de la asociación de interés para nosotros, el Caso de uso registro de falla devolverá un ticket electrónico de seguimiento al actor de recepción de falla, visualizando una comunicación inversa a la señalada, pero no se muestra por resultar de menor interés.



Hacia la relación entre casos uso y actores

La generalización gráfica es la relación de herencia entre un padre y un hijo, y aplica para casos de uso y actores.

El registro de inventario de equipo de cómputo en una empresa se realiza para PC e Impresoras, pero para ambas implica el capturar información común y otra parte particular derivado del tipo de hardware. La herencia entre actores nos permite modelar organigramas y perfiles de seguridad, entre otras cosas. Si la herencia modela un organigrama, este debe ser interpretado como las responsabilidades del padre son heredadas a sus hijos. En el ejemplo anterior, se entiende que un líder de proyecto hereda actividades asignadas a él, a un programador o a un diseñador.




Extend - Include


Los Casos de uso pueden relacionarse a través de dos tipos de dependencia: la extensión y la inclusión. 

La extensión nos ayuda a modelar flujos opcionales dentro de un Caso de uso, es así que siempre representa un conjunto de acciones, con un nivel de complejidad merecedor de realizar un análisis independiente, a través de un Caso de uso. La extensión se representa como una línea discontinua direccionada hacia el Caso de uso integrador del opcional y nombrado con el estereotipo <<extend>>.

El Caso de uso de venta producto es utilizado para registrar el levantamiento de pedido de un cliente; la mayor parte de las ocasiones, todos los productos vendidos tienen existencia, pero cuando algún producto hace falta, se inicia el Caso de uso opcional de falta producto, que derivado de la opcionalidad se convierte en una relación de extensión.

La inclusión representa la segmentación de un Caso de uso para su análisis independiente, derivado de la complejidad o tamaño, o bien cierta funcionalidad común a dos o más Casos de uso que merezcan ser considerados como tales. La inclusión se representa como una línea discontinua direccionada al Caso de uso que se quiere incluir dentro de la funcionalidad de otro.


Para la situación de registro de entrada de personal y registro de salida de personal, ambos presentan, de manera diferente, la información, pero ambos requieren registrar el movimiento de una misma tabla de base de datos, o sea, esto se puede manejar por el Caso de uso de registro de movimientos común a ambos, y ambos lo pueden incluir sin entrar en conflicto.

¿Cómo comenzar?

Para llevar a cabo el desarrollo de un proyecto vimos varias metodologías, entonces podemos utilizar todas o seleccionar alguna de ellas. Ahora estamos frente al Caso de uso y nos preguntamos: ¿Cómo comenzar?

Llegada esta instancia, si aplicamos en el análisis alguna de las herramientas explicadas en las semanas anteriores, simplemente transformamos los elementos en Actores, Escenarios y Relaciones para armar el Caso de uso. Pero si es la primera herramienta que seleccionamos, te sugerimos los siguientes pasos:


Identificar usuarios y actores: Retroceder en la historia de la regla de negocios para identificar a los usuarios potenciales del software. Estos usuarios se convierten en los actores clave para el Caso de uso.

Definir actores principales: Identificar actores principales según los diferentes sectores u áreas de la organización que utilizarán el software. Pueden ser designados con nombres específicos o generalizados, como "Empleado" o "Usuario".

Analizar la interacción del usuario: Evaluar cómo el usuario principal interactuará con el software, identificando sus acciones iniciales y anticipando cómo accederá a las funcionalidades del sistema.

Elaborar catálogo de acciones: Trabajar en conjunto para desarrollar un catálogo detallado de acciones que el usuario llevará a cabo en el sistema, estableciendo así un catálogo preliminar de Casos de uso.

Aplicar desagregación: Utilizar la técnica de desagregación para identificar escenarios específicos y descomponer las acciones del usuario en Casos de uso individuales.

Considerar flujos de procesos repetitivos: Identificar Casos de uso que representen flujos de procesos repetitivos, marcándolos como <<include>> en el diagrama.

Considerar flujos de procesos ocasionales: Identificar Casos de uso que representen flujos de procesos ocasionales, marcándolos como <<extend>> en el diagrama.

Crear el diagrama de Caso de uso: Utilizar un rectángulo para representar el sistema general y colocar fuera de él a los actores identificados. Dentro del rectángulo, representar cada escenario (Caso de uso) como parte integral del sistema.
