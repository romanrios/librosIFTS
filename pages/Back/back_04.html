Javascript del lado del servidor

El motor de Javascript se llama V8 y es el que utilizan Chrome y Nodejs . Nodejs se caracteriza por utilizar un modelo de operaciones E/S sin bloqueo y orientado a eventos, lo que lo hace muy liviano y eficiente. 

Este modelo es diferente al modelo tradicional de hilos, donde los servidores web crean un hilo para cada petición y se bloquean hasta que la tarea correspondiente se complete. 

El problema con este esquema es que cada hilo requiere una pequeña cantidad de memoria RAM. Esto puede representar una limitación para sistemas con un gran número de peticiones concurrentes. Sin embargo, con Nodejs podemos tener sistemas escalables con una mayor facilidad de gestión ya que no sufriríamos estas limitaciones de memoria.


El proceso de ejecución de NodeJS


Una de las primeras cosas que vamos a aprender en NodeJS es trabajar y entender el proceso de ejecución del programa, que tiene sus particularidades especiales en este lenguaje. 

Es importante

Se destaca la naturaleza de NodeJS como un lenguaje de programación de propósito general, con el que podemos hacer todo tipo de aplicaciones..

Normalmente comenzaremos con programas que se ejecutarán en la consola de comandos, o terminal, del sistema operativo, pero a partir de ahí sus aplicaciones son muy extensas.

Pensemos entonces. que NodeJS se arranca mediante la consola de comandos y entonces el proceso que se encargará de ejecutarlo es el iniciado con la propia consola.

JavaScript es un lenguaje concurrente, asíncrono, no bloqueante, interpretado, de alto nivel y de un solo hilo. Aprenderemos qué significa todo esto a continuación y cómo se relaciona con Node.


Single Thread (único hilo)


Una de las características de NodeJS es su naturaleza "Single Thread". Cuando ponemos en marcha un programa escrito en NodeJS se dispone de un único hilo de ejecución. Un hilo de ejecución es la secuencia más pequeña de instrucción programada que puede ser administrada de forma independiente por un programador.

Esto, en contraposición con otros lenguajes de programación como Java, PHP o Ruby, donde cada solicitud se atiende en un nuevo proceso, tiene sus ventajas. Una de ellas es que permite atender mayor demanda con menos recursos, uno de los puntos a favor de NodeJS. Es decir, sirve para generar una gran cantidad de conexiones simultáneas con el servidor sin incrementar el consumo de recursos por cada una de ellas. 


Ejemplo

Como observamos en la imagen a continuación, en una aplicación tradicional el servidor crea un nuevo hilo por cada petición, en cambio Node.js maneja asincrónicamente las conexiones por medio de un único hilo y un bucle de eventos.


Para poder aprovechar de manera óptima la programación Single Thread de NodeJS, es importante entender su característica no bloqueante. 

Esta característica, a diferencia de la programación secuencial, permite que NodeJS no se bloquee al realizar operaciones, sino que las libera para atender otras solicitudes.

Cabe aclarar que el single thread no significa que NodeJS no pueda disponer de varios hilos de manera interna para resolver problemas. Es decir, nuestro hilo principal por ejemplo cuando estamos desarrollando un servidor con Node podrá estar atento a solicitudes, pero una vez que se atiendan, Node podrá levantar de manera interna otros procesos para realizar todo tipo de acciones que se deban producir como respuesta a esas solicitudes. 

No bloqueante, concurrente y asincrónico


No bloqueante

Significa que las operaciones no detienen la ejecución del programa mientras esperan a que se completen. En lugar de eso, proporcionan una forma de especificar una función "callback" que se ejecutará una vez que la operación se ha completado.


Ejemplo

Una operación bloqueante es leer un archivo del disco duro. Si utilizamos una función bloqueante para leer un archivo, el programa detendrá su ejecución hasta que se haya completado la lectura. En cambio, si utilizamos una función no bloqueante, el programa continuará ejecutando otras tareas mientras se lee el archivo, y se ejecutará la función de "callback" una vez que se haya completado la lectura.



Este enfoque permite que Node maneje muchas conexiones simultáneas, ya que no tiene que esperar a que finalice una operación antes de continuar con la siguiente. Esto lo hace muy útil para aplicaciones web y de red, donde se reciben y manejan muchas peticiones simultáneamente.

Concurrente

Significa que el código se está ejecutando simultáneamente por más de un hilo.

Asincrónico

La programación asincrónica significa que el código se ejecuta en un bucle de eventos. Cuando hay una operación de bloqueo, se inicia el evento. El código de bloqueo sigue ejecutándose sin bloquear el hilo de ejecución principal. Cuando el código de bloqueo termina de ejecutarse, la cola es el resultado de las operaciones de bloqueo y las devuelve a la pila.


Motor V8


Antes de continuar, repasemos lo anterior:

JavaScript es de un solo subproceso.

JavaScript no bloquea, es decir, los procesos lentos no bloquean su ejecución.

JavaScript es concurrente, es decir, ejecuta su código en más de un hilo al mismo tiempo.

JavaScript es asincrónico, es decir, ejecuta código de bloqueo en otro lugar.

El motor V8 es un motor en tiempo de ejecución de código abierto y alto rendimiento para JavaScript escrito en C ++ por Google. La mayoría de los navegadores ejecutan JavaScript usando el motor V8, e incluso el popular entorno de tiempo de ejecución de NodeJS también lo usa.


Loop de eventos


El concepto de loop de eventos es muy simple. Existe un ciclo infinito en el que el motor de JavaScript espera por una tarea, luego ejecuta la tarea requerida y finalmente vuelve a dormir esperando por una nueva tarea.

El algoritmo general del motor funciona de la siguiente manera:


Mientras haya tareas: 

  1. Ejecutarlas, comenzando por la más antigua.

Dormir hasta que aparezca una tarea, luego volver a 1.


Eso es una formalización de lo que vemos cuando navegamos por una página. El motor JavaScript no hace nada la mayoría del tiempo y solo corre cuando un script/controlador/evento se activa.


Las tareas son programadas --> el motor las ejecuta --> espera por más tareas (mientras duerme y prácticamente no consume CPU). Puede ocurrir que una tarea llegue mientras el motor está ocupado, entonces es puesta en cola. Las tareas forman una cola, también llamada “Cola de macrotarea”.


Objeto process


El objeto process es una variable global disponible en NodeJS que nos ofrece diversas informaciones y utilidades acerca del proceso que está ejecutando un script Node. Contiene diversos métodos, eventos y propiedades que nos sirven no solo para obtener datos del proceso actual, sino también para controlarlo.

El hecho de ser un objeto global quiere decir que lo podemos usar en cualquier localización de nuestro código NodeJS, sin tener que importarlo.

El objeto process nos permite realizar varias operaciones, entre ellas:


JavaScript Copiar código 
console.log('id del proceso: ', process.pid);
console.log('título del proceso: ', process.title);
console.log('versión de node: ', process.version);
console.log('sistema operativo: ', process.platform);


Salir de la ejecución de un programa Node

A veces necesitamos salir inmediatamente de la ejecución de un programa en NodeJS. Esto lo podemos conseguir mediante la invocación del método exit() del objeto process.


JavaScript Copiar código 
process.exit();
console.log('esto no se llegará a ejecutar');


Igualmente, el comportamiento normal de un programa será salir automáticamente cuando haya terminado su trabajo, es decir, cuando se haya terminado la secuencia de ejecución de instrucciones de un script y no haya trabajo pendiente en el "event loop".

Módulos


Un módulo en Node.js es un archivo de código JavaScript que proporciona un conjunto de funcionalidades y puede ser reutilizado en diferentes partes de una aplicación. Los módulos son una forma de dividir y organizar el código de tu aplicación de manera más eficiente y mantenible.

Hay dos tipos de módulos en Node.js: 

Los módulos nativos son módulos que forman parte de la propia instalación de Node.js y proporcionan funcionalidades esenciales para el funcionamiento del sistema.

Los módulos de terceros son módulos que se han creado por terceros y se pueden instalar a través de NPM (Node Package Manager) como veremos más adelante.

Algunos de los módulos nativos más comunes son:

  • http: Proporciona funcionalidades para crear servidores y clientes HTTP.
  • fs: Proporciona funcionalidades para leer y escribir archivos en el sistema de archivos del servidor.
  • path: Proporciona funcionalidades para trabajar con rutas de archivos y directorios.
  • util: Proporciona funcionalidades de utilidad diversa, como heredar de prototipos y ejecutar funciones de manera asíncrona.
  • assert: Proporciona funcionalidades para realizar pruebas de afirmaciones en el código.
  • net: Permite trabajar con la red utilizando diferentes protocolos.
  • events: Módulo para implementar emisores de eventos (el patrón observador de node).
  • os: Permite acceder a información al nivel del sistema operativo.

Un módulo puede ser importado dentro de Node a través del método require (nombre_del_modulo) y se guarda en una variable o constante:

JavaScript Copiar código 
const http = require('http'); // Carga el módulo llamado http


Ahora, desde la constante http podemos acceder a las propiedades y métodos del módulo http .

Cuando se desea tener disponible cualquier módulo para usarlo en un programa, se invoca al método require() . 

Este método comprueba si el módulo está en la caché de módulos y en caso negativo, lo busca en el sistema de ficheros o en caso de que sea un módulo del core de Node, en el propio ejecutable de la plataforma. 

Una vez localizado, lo compila, lo mete en la caché y devuelve su exports, es decir, las funciones, variables u objetos que el módulo pone a disposición del programador.

El módulo HTTP


El módulo http es un módulo principal de Node.js, que proporciona la funcionalidad para crear un servidor HTTP utilizando el método http.createServer() .

El método createServer del módulo http se utiliza para crear un nuevo servidor web. Se registra un controlador de eventos en el servidor, que se llama cada vez que se realiza una solicitud HTTP a la dirección del servidor.

Si creamos un archivo que contenga el siguiente código JavaScript, podremos levantar un servidor HTTP en el puerto 8000. Cabe aclarar que podemos usar cualquier número de puerto a elección siempre y cuando no esté actualmente en uso por ningún otro servicio.


JavaScript Copiar código 
const http = require('http');

const server = http.createServer((req, res) => {
    res.statusCode = 200;
    res.setHeader('Content-Type', 'text/plain');
    res.end('Hello World\n');
});

server.listen(8000, '127.0.0.1', () => {
    console.log('Server running at http://127.0.0.1:8000/');
});


Guardamos el archivo con cualquier nombre de archivo. En este caso, si lo llamamos hello.js podemos ejecutar la aplicación yendo al directorio donde se encuentra el archivo y usando el siguiente comando:


Node Copiar comando 
node hello.js


Si vamos al navegador y nos dirigimos a la siguiente URL http://localhost:8000/ podremos ver una página simple con el texto Hello, World! .

La solicitud se responde con el código de estado 200, con el header Content-Type establecido en text/plain, y el contenido del sitio que se devolverá establecido en Hello World .


El módulo del sistema de archivos (fs)


Uno de los módulos incorporados más utilizados en Node es el módulo fs , que significa sistema de archivos. Este exporta funciones para trabajar con archivos y directorios. 

Si queremos leer un archivo de texto que tenemos en local simplemente usaremos ese módulo para extraer el contenido del fichero, indicando su ruta y otra serie de parámetros que ahora describiremos.

En lo que respecta a los métodos del módulo fs , hay que señalar que se entregan en dos alternativas, tanto síncrona como asíncrona, que pueden cubrir diferentes situaciones y necesidades que tengamos.

Igual que hacemos con otros módulos de Node, utilizamos require para tener disponibles las funciones de acceso al sistema de ficheros. Se encuentran en el módulo llamado 'fs' y lo vamos a importar con el siguiente código:


JavaScript Copiar código 
let fs = require('fs');


Este método accede a un fichero para su lectura y nos entrega el contenido en forma de buffer o en forma de cadena. Se usa así:


JavaScript Copiar código 
fs.readFile('archivo.txt', 'utf-8', (err, data) => {
    if (err) {
        console.log('error: ', err);
    } else {
        console.log(data);
    }
});


Es importante notar que si no creamos un archivo llamado 'archivo.txt' en la misma ruta del código, obtendremos un error:


Node Copiar código 
error: { Error: ENOENT: no such file or directory, open 'archivo-inexistente.txt'
   at Error (native)
   errno: -2,
   code: 'ENOENT',
   syscall: 'open',
   path: 'archivo-inexistente.txt' }


Método readFileSync síncrono


Como alternativa tenemos el método readFileSync() que hace básicamente lo mismo, pero bloquea la ejecución de las siguientes líneas de código hasta que la lectura haya concluido y se tenga el contenido del archivo completo. Este método equivaldría a la lectura de archivos en la mayoría de los lenguajes de programación. Al ejecutarse, bloquea el flujo de ejecución actual, de modo que el código del programa se procesará siempre de manera secuencial.

El método se usa igual, solo que dispensamos la función callback. Además, el archivo leído será devuelto por la función. Podemos ver un ejemplo a continuación:


JavaScript Copiar código 
let archivo = fs.readFileSync('archivo2.txt', 'utf-8');



El módulo fs contiene muchas otras funciones útiles:

  • - readdir: devolverá los archivos de un directorio como un array de cadenas.
  • - stat: recuperará información sobre un archivo.
  • - rename: renombrará un archivo.
  • - unlink: eliminará un archivo.

Módulos propios


Además de utilizar módulos de Node, también podemos crear nuestros propios módulos para organizar mejor nuestro proyecto. Si no lo hacemos, tendremos todo el código en un único archivo y eso suele generar mucho desorden, sobre todo cuando los proyectos y el código crecen.

Para crear un propio módulo, debemos crear un archivo separado, por ejemplo, uno que se llame operaciones_basicas.js y agregar las siguientes líneas:


JavaScript Copiar código 
exports.producto = function (a, b) {
    return a * b;
};
exports.suma = function (a, b) {
    return a + b;
};

Exports es el objeto que se va a exportar y, por lo tanto, que se asignará al momento de hacer un require('./operaciones_basicas') .

En el ejemplo previo estamos exportando producto y suma . Por lo tanto, para usarlo podremos hacer:


JavaScript Copiar código 
const operaciones_basicas = require('./operaciones_basicas');
console.log('El producto de 4 x 10 es ' + operaciones_basicas.producto(4, 10));

También podemos exportar un objeto o una función, para ello debemos exportar directamente el objeto o función al module.exports :


JavaScript Copiar código 
module.exports = {
    producto: function (a, b) {
        return a * b;
    },
    suma: function (a, b) {
        return a + b;
    }
};



Gestor de paquetes


Junto con el marco de trabajo que creamos cuando instalamos node.js, se nos ha instalado una herramienta muy útil y potente que nos permite gestionar tanto paquetes a nivel global como paquetes a nivel local en nuestro proyecto: npm.

NPM (Node Package Manager) es un sistema de gestión de paquetes de software para JavaScript. Se utiliza principalmente para instalar y gestionar módulos de código reutilizable que se utilizan en los proyectos de desarrollo con JavaScript. NPM es una parte esencial de la plataforma Node.js y se instala automáticamente junto con él. Con NPM, podemos descargar e instalar cientos de módulos de código de terceros que pueden ayudarnos a ahorrar tiempo al desarrollar nuestros proyectos.


También podemos utilizar NPM para publicar nuestros propios módulos y compartirlos con otros/as desarrolladores/as.


La herramienta dispone de sitio web propio, donde podemos encontrar manuales y muchísima documentación, no sólo de npm en sí, sino de todos los paquetes que podemos instalar junto con él. El sitio de npm es: https://www.npmjs.com/.


El comando npm


Por lo que respecta al uso de npm, es un comando que funciona desde la línea de comandos de NodeJS. Por tanto, lo tenemos que invocar con npm seguido de la operación que queramos realizar. Ejemplo:

 Node Copiar código 
npm install async


Esto instalará el paquete async dentro de mi proyecto. Lo instalará dentro de la carpeta node_modules y a partir de ese momento estará disponible en mi proyecto y podré incluirlo por medio de require:

JavaScript Copiar código 
require('async');

Cabe aclarar que no hace falta darle la ruta al paquete async porque npm lo ha instalado dentro de node_modules y todo lo que está en esa carpeta se encuentra disponible para require() sin necesidad de decirle la ruta exacta para llegar.

Cuando queramos usar npm, a continuación del nombre del comando siempre indicaremos el subcomando que le dirá lo que queremos hacer. Algunos de los subcomandos más comunes son:



  • - help: nos muestra una ayuda de npm junto con el conjunto de subcomandos que podemos usar. Para obtener información de un subcomando de npm, hacemos npm subcomando -h . Por ejemplo npm install -h.
  • - info: nos muestra información de un paquete en concreto.
  • - init: permite generar un archivo package.json que contendrá, entre otras cosas, las dependencias de nuestro proyecto.
  • - install: nos permite instalar paquetes, ya sea de manera global o de manera local, como ya se ha comentado.

El archivo package.json


El archivo package.json es un archivo esencial en un proyecto de Node.js que contiene información importante acerca del proyecto, como su nombre, versión, dependencias y scripts de inicio.


Vemos que se indica el nombre de la aplicación (project), la versión (0 indica que está en desarrollo, cuando la saquemos a producción la versión debería ser la 1.0), el atributo private (que indica que npm no deberá de publicar nuestro proyecto, es decir, no lo deberá hacer público). A continuación viene el conjunto de scripts que se definen para la aplicación. En este caso se está definiendo un único script denominado "test", que muestra un mensaje de error ya que no tenemos tests en la aplicación. Y a continuación viene el conjunto de dependencias que ya se utilizan en el proyecto y se instalar al ejecutar el comando npm install .

 Con ese archivo, NPM puede instalar todas las dependencias de un proyecto y dejar la aplicación lista para su ejecución. 

En general, package.json es un archivo clave en un proyecto Node.js, ya que permite a los desarrolladores compartir y reutilizar código fácilmente mediante el uso de dependencias. Es también una forma de llevar el control de las versiones de las librerías que utiliza un proyecto.


Para crear un package.json se puede usar el comando npm init en la línea de comandos en el directorio del proyecto, y sigue los pasos. Se recomienda crear dicho archivo para cada proyecto, ya sea manualmente o ejecutando npm init . 

Cuando ejecutamos “ npm instal ” sin nombrar un paquete específico a instalar, NPM instalará las dependencias listadas en package.json. Cuando instalemos un paquete específico que no esté ya listado como una dependencia, NPM lo añadirá al archivo package.json .


Control de versiones


El modelo de control de versiones utilizado en npm se denomina control de versiones semántico. El control de versiones semántico es una convención formal para determinar el número de versión de las nuevas versiones de software. El estándar ayuda a los usuarios de software a comprender la gravedad de los cambios en cada nueva distribución.

Dependencias de Desarrollo

Si una dependencia solo se usa durante el desarrollo, debemos guardarla como una "dependencia de desarrollo". Por dependencias de desarrollo, nos referimos a herramientas que son necesarias solo durante el desarrollo de la aplicación, por ejemplo, para probar o reiniciar automáticamente la aplicación, como nodemon (una herramienta que veremos más adelante). Estas dependencias de desarrollo no son necesarias cuando la aplicación se ejecuta en el servidor de producción.

Por ejemplo, para usar la popular herramienta de Linting ESLint (https://eslint.org/) podemos llamar a NPM como se muestra a continuación:

Node Copiar código 
npm install eslint --save-dev


La siguiente entrada se agregaría al package.json de su aplicación:


JSON Copiar código 
"devDependencies": {
   "eslint": "^8.31.0"
 }


Nota

Un linter es una herramienta de análisis de código que se utiliza para verificar que el código sigue un conjunto predefinido de reglas y estilos. Los linters son útiles para detectar problemas comunes, como errores sintácticos, problemas de rendimiento, y violaciones de estilo de codificación. ESLint es un linter de JavaScript específico, es una herramienta de código abierto para detectar problemas en el código JavaScript.


¿Qué significa el signo de intercalación (^) delante del número de versión en package.json ?

El signo de intercalación al frente de ^8.31.0 significa que cuando se actualizan las dependencias de un proyecto, la versión de eslint que se instalará será al menos la 8.31.0.

Podemos actualizar las dependencias del proyecto con el comando:


Node Copiar código 
npm update

Otros gestores de paquetes


Un gestor de paquetes es una herramienta que nos ayuda a instalar, actualizar y desinstalar las dependencias de un proyecto de Node.js. Los paquetes son librerías y herramientas de código abierto que pueden ser utilizadas para agregar funcionalidad adicional a un proyecto. Así como vimos el gestor de paquetes NPM, existen otros, uno de los maś conocidos es YARN. Este es un gestor de paquetes alternativo a npm. Fue desarrollado por Facebook y tiene algunas características adicionales, como una instalación más rápida y una mejor gestión de las dependencias. Sin embargo, el número de paquetes disponibles es menor que en npm. Podemos encontrar más info en https://yarnpkg.com/


Modelo de eventos


El flujo de ejecución de JavaScript en el navegador, así como en Node.js, está basado en un event loop (loop de eventos). Entender como este loop de eventos funciona es importante para optimizaciones y en algunos casos para utilizar la arquitectura correcta. 

Una de las características más potentes de node.js es el trabajo con eventos. El trabajo con eventos es la asincronía por naturaleza, ya que nunca se sabe cuándo ocurrirá un evento. Normalmente en nuestras aplicaciones, comenzaremos la ejecución de servidores y los servicios que éstos lancen, y el sistema se quedará esperando que ocurra algún evento. 


APIs Asíncronas


Como vimos en clases anteriores, en JavaScript se usan frecuentemente APIs asíncronas en lugar de sincrónicas para operaciones que tardan algún tiempo en completarse. En una API sincrónica, cada operación debe completarse antes de que la siguiente pueda comenzar. Por ejemplo, la siguiente función de registro es síncrona y escribirá en orden el texto en la consola (Primero , Segundo).


JavaScript Copiar código 
console.log('Primero');
console.log('Segundo');


En contraste, en una API asincrónica, la API comenzará una operación e inmediatamente retornará (antes de que la operación se complete). Una vez que la operación finalice, la API usará algún mecanismo para realizar operaciones adicionales. Por ejemplo, el código de abajo imprimirá "Segundo, Primero" porque aunque el método setTimeout() es llamado primero y retorna inmediatamente, la operación no se completa por varios segundos.


JavaScript Copiar código 
setTimeout(function () {
    console.log('Primero');
}, 3000);
console.log('Segundo');


Usar APIs asíncronas sin bloques es aún más importante en Node que en el navegador, porque Node es un entorno de ejecución controlado por eventos de un solo hilo. "Un solo hilo" quiere decir que todas las peticiones al servidor son ejecutadas en el mismo hilo (en vez de dividirse en procesos separados). Este modelo es extremadamente eficiente en términos de velocidad y recursos del servidor, pero eso significa que si alguna de sus funciones llama a métodos sincrónicos que tomen demasiado tiempo en completarse, bloquearán no solo la solicitud actual, sino también cualquier otra petición que esté siendo manejada por tu aplicación web.



Eventos en NodeJS


A continuación explicaremos qué son los eventos del lado del servidor y cómo se implementan en NodeJS.

Eventos del lado del servidor 

No tienen nada que ver con los eventos Javascript que conocemos y utilizamos en las aplicaciones web del lado del cliente. Aquí los eventos se producen en el servidor y pueden ser de diversos tipos dependiendo de las librerías o clases que estemos trabajando.


Para hacernos una idea más exacta, pensemos por ejemplo en un servidor HTTP, donde tendríamos el evento de recibir una solicitud. Por poner otro ejemplo, en un stream de datos tendríamos un evento cuando se recibe un dato como una parte del flujo.


Módulo de eventos


Los eventos se encuentran en un módulo independiente que tenemos que requerir en nuestros programas creados con Node JS. Lo hacemos con la sentencia require que conocimos en artículos anteriores cuando hablábamos de módulos.


JavaScript Copiar código 
var eventos = require('events');


Dentro de esta librería o módulo tenemos una serie de utilidades para trabajar con eventos. Veamos primero el emisor de eventos, que se encuentra en la propiedad EventEmitter .


JavaScript Copiar código 
var EmisorEventos = eventos.EventEmitter;


Nota: Este EmisorEventos es una clase de programación orientada a objetos (POO), por eso se le ha puesto en el nombre de la clase la primera letra en mayúscula. Por convención se hace así con los nombres de las clases en POO.

Cómo definir un evento en NodeJS


En Node existe un bucle de eventos, de modo que cuando declaramos un evento, el sistema se queda escuchando en el momento que se produce, para ejecutar entonces una función. Esa función se conoce como callback o como manejador de eventos y contiene el código que queremos que se ejecute en el momento que se produzca el evento al que la hemos asociado.

Primero tendremos que "instanciar" un objeto de la clase EventEmitter, que hemos guardado en la variable EmisorEventos en el punto anterior de este artículo.

JavaScript Copiar código 
var ee = new EmisorEventos();








Luego tendremos que usar el método on() para definir las funciones manejadoras de eventos, o su equivalente addEventListener() . Para emitir un evento mediante código JavaScript usamos el método emit(). Por ejemplo, vamos a emitir un evento llamado "datos", con este código:

JavaScript Copiar código 
ee.emit('datos', Date.now());

Ahora vamos a hacer una función manejadora de eventos que se asocie al evento definido en "datos".

JavaScript Copiar código 
ee.on('datos', function (fecha) {
 console.log(fecha);
});

Si deseamos aprovechar algunas de las características más interesantes de aplicaciones NodeJS quizás nos venga bien usar setInterval() y así podremos estar emitiendo datos cada cierto tiempo. La función setInterval() es utilizada para ejecutar una función o un bloque de código en un intervalo de tiempo especificado (en milisegundos). Se ejecutará de manera repetitiva hasta que sea detenida mediante la función clearInterval() .

JavaScript Copiar código 
setInterval(function () {
 ee.emit('datos', Date.now());
}, 500);

Este código lo podemos guardar en un archivo como eventos.js. Para ponerlo en ejecución nos vamos en línea de comandos hasta la carpeta donde hayamos colocado el archivo eventos.js y ejecutamos el comando:

Node Copiar comando 
node eventos.js


Como resultado, veremos que empiezan a aparecer líneas en la ventana del terminal del sistema operativo, con un número, que es el "timestamp" de la fecha de cada instante inicial. Nota: podemos salir del programa usando las teclas CTRL + C.

Entornos de ejecución


Un entorno de ejecución es el contexto en el que se desarrolla y ejecuta una aplicación. 

Los escenarios más comunes son desarrollo, pruebas y producción: en el primero se escribe y ajusta el código, mientras que en el último, este se ejecuta para los usuarios finales. 

El entorno de pruebas se utiliza para verificar el funcionamiento antes del lanzamiento oficial. 

En Node.js, es posible definir estos diferentes contextos mediante una variable de entorno del Sistema Operativo, concretamente la variable NODE_ENV. Antes de entrar en detalles, revisemos qué son las variables de entorno.


Variables de entorno


Las variables de entorno sirven para definir parámetros sencillos de configuración de los programas, de modo que éstos puedan ejecutarse en diferentes ambientes sin necesidad de modificar el código fuente de un script.

Ejemplos habituales de las variables de entorno serían el puerto donde escucha un servidor web, el servidor de base de datos, junto con el usuario y clave para conexión, llaves de APIs, etc. Todos esos valores generalmente pueden cambiar cuando se escribe un programa (al ejecutarlo en local) y cuando se publica el script en un servidor en producción.


Cómo leer variables de entorno en NodeJS


Gracias al objeto process, disponible en todo programa NodeJS (que nos da información sobre el proceso que está ejecutando el script), tenemos acceso a las variables de entorno de una manera sencilla. Simplemente usamos la propiedad env y luego el nombre de la variable que necesitamos acceder.

Por ejemplo, una variable de entorno llamada PORT se accedería a través de esta referencia:


JavaScript Copiar código 
process.env.PORT;


Lo que puede ocurrir es que las variables de entorno no siempre se hayan definido, por lo que es útil que en nuestro programa les asignemos unos valores predeterminados. Esto lo podemos conseguir con un código como este:


JavaScript Copiar código 
let puerto;
if (process.env.PORT) {
 puerto = process.env.PORT;
} else {
 puerto = 3000;
}


Al invocar un script Node desde el terminal podemos definir si lo deseamos los valores de las variables de entorno que necesitemos. Esto se hace antes de ejecutar el programa en sí, asignando los valores a las variables necesarias.

Como vimos, una aplicación Node se ejecuta así: (si el archivo de arranque se llama simple-env.js):


JavaScript Copiar código 
node simple-env.js


Pero si queremos que el puerto en el que corra sea el puerto 8000, podemos definir la variable de entorno PORT  en la misma línea de la siguiente forma:


JavaScript Copiar código 
PORT=8000 node simple-env.js


También puedes definir múltiples variables de entorno en la misma línea si es necesario.


JavaScript Copiar código 
PORT=8000 NODE_ENV=development node simple-env.js

NODE_ENV


Es normal que en una aplicación NodeJS mantengamos variables de entorno con valores diferentes cuando estamos en desarrollo y cuando estamos en producción o en cualquier otro ambiente que tengamos. A continuación explicaremos una posible manera de estructurar nuestras variables de entorno, en grupos, de modo que las podamos cambiar todas de una única vez dependiendo de si estamos en Producción, Desarrollo, Test, etc.

La idea o el objetivo que perseguimos en esta práctica es doble:

Que no necesitemos especificar toda la lista de variables de entorno posibles al ejecutar un programa, sino simplemente informar si estamos en desarrollo, producción, test, etc.

Que podamos mantener todas las variables de entorno en un archivo de texto, de modo que podamos editarlo cómodamente como cualquier otro archivo de código de la aplicación.

En la implementación que vamos a realizar, podremos cambiar todo el conjunto de variables de entorno de una única vez, diciendo al ejecutar la aplicación si estamos en desarrollo o producción.

Por ejemplo, para usar el conjunto de variables de entorno predeterminado, que podrían ser los valores necesarios en la etapa de desarrollo, podríamos ejecutar nuestra aplicación sin definir ninguna variable de entorno:


Node Copiar comando 
node group-env.js


Si luego queremos ejecutar esta aplicación con los valores de todas las variables de entorno necesarias para producción, entonces tendríamos que ejecutar la aplicación indicando que estamos en el entorno de producción:


Node Copiar comando 
NODE_ENV=production node group-env.js


La variable NODE_ENV es una variable de entorno cuyo valor predeterminado es "development" , pero se puede establecer en "test" o "production" según sea necesario. Esta variable se utiliza para indicar en qué entorno se está ejecutando la aplicación y, por lo tanto, cambiar el comportamiento de la aplicación en consecuencia.


Definición de las variables de entorno en un JSON


Podemos usar un literal de objeto Javascript (JSON) escrito en un archivo independiente para mantener todas nuestras variables de entorno. La clave aquí es que vamos a definir diversos conjuntos de variables de entorno.

JSON Copiar código 
{
   "development": {
     "SERVERURL": "http://localhost:3001/",
     "PORT": 3001
   },
   "production": {
     "SERVERURL": "https://app.enproduccion.com/",
     "PORT": 5000
   }
 }


Este código lo colocaremos en un archivo a parte, que requeriremos cuando sea necesario. El nombre del archivo es indiferente, en nuestro caso lo hemos guardado dentro de "env.variables.json" .


Ejemplo

Tenemos dos conjuntos de valores de entorno, "development" y "production" , pero podríamos tener tantos como necesitemos y por supuesto, también podemos tener tantas variables de entorno como nos hagan falta, y no solamente PORT y SERVERURL como en el JSON anterior.

Require de las variables de entorno


En todo lugar donde necesitemos acceder a las variables de entorno haremos el correspondiente require del JSON anterior.

JavaScript Copiar código 
const envJSON = require('./env.variables.json');


En el caso anterior, todos los posibles juegos de variables de entorno se volcarán en un objeto al que hemos nombrado envJSON . Ya solo nos queda definir qué conjunto de variables vamos a usar, para lo que vamos a recibir una única variable de entorno NODE_ENV. Además, si no se encuentra esa variable vamos a especificar que su valor sea development .


JavaScript Copiar código 
const node_env = process.env.NODE_ENV || 'development';


Con esa variable de entorno podemos tomar los valores necesarios del JSON, por ejemplo:


JavaScript Copiar código 
let variables_de_entorno = envJSON[node_env];


Y si queremos acceder a una única variable de entorno, podríamos hacer algo como esto:


JavaScript Copiar código 
let puerto = envJSON[node_env].PORT;

Definir un módulo para la gestión de variables de entorno


Si queremos ir un poco más allá, podríamos definir el acceso a las variables de entorno en un módulo aparte, de modo que siempre que las necesitamos sea solo realizar el require de ese módulo de configuración. El código de nuestro módulo podría ser algo como esto:

JavaScript Copiar código 
exports.config = function () {
    const envJSON = require('./env.variables.json');
    const node_env = process.env.NODE_ENV || 'development';
    return envJSON[node_env];
};


Ahora, para acceder a las variables de entorno hacemos el require del módulo anterior (y ejecutamos la función que nos exporta, llamada config ) que realiza el acceso al grupo de valores de entorno que necesitamos en cada caso.

JavaScript Copiar código 
const entorno = require('./config-module.js').config();

Middlewares


El concepto de Middlewares es popular en muchos Frameworks de Desarrollo Web, y son elementos muy frecuentes en un servidor creado con Node.js . Cuando tengamos una instancia de un servidor en Node.js y llegue a nuestro servidor una petición http , el servidor va a empezar a usar los Middlewares definidos, antes de mandar la respuesta al cliente. Digamos que es un proceso intermedio entre la llegada de la request a un servidor http y el envío de la respuesta a un/a cliente. 

Un Middleware tiene como propósito tomar dos piezas de la aplicación y conectarlas, como un puente en el que fluye la información. Normalmente decimos que es una rutina de código que tiene como propósito recibir información y retornarla transformada.



Definición

Los Middlewares son funciones que se ejecutan en una solicitud-respuesta de una aplicación, y tienen acceso a la solicitud, la respuesta y el siguient Middleware en la cadena. A menudo se utilizan para realizar tareas como la autenticación, la gestión de sesiones, la validación de datos, la gestión de errores, entre otras.


Cuando estudiemos Express en la semana siguiente, vamos a usar Middlewares continuamente mediante la instrucción app.use() . La variable app va a contener una instancia de la aplicación que está corriendo en nuestro servidor node . Con el método use le indicamos que antes de enviar la respuesta al/la cliente use dicho Middleware dentro del conjunto de Middlewares que hayamos definido secuencialmente .

Arquitectura MVC


El MVC o Modelo-Vista-Controlador es un patrón de arquitectura de software que, utilizando 3 componentes (Vistas, Models y Controladores) separa la lógica de la aplicación de la lógica de la vista en una aplicación. Este patrón es uno de los más usados, en la actualidad se puede encontrar tanto en pequeños como en grandes sistemas.

¿Por qué se utiliza el MVC?

 La razón principal es que nos permite separar los componentes de nuestra aplicación dependiendo de la responsabilidad que tienen.

Esto significa que cuando hacemos un cambio en alguna parte de nuestro código, esto no afecta a otra parte del mismo. Por ejemplo, si modificamos nuestra Base de Datos, sólo deberíamos modificar el modelo que es quién se encarga de los datos y el resto de la aplicación debería permanecer intacta.

Esto respeta el principio de la responsabilidad única, es decir, cada parte de nuestro código sólo debe de tener una responsabilidad sin necesidad de conocer qué es lo que hace toda la aplicación.

   

Modelo

El modelo se encarga de manipular, gestionar y actualizar los datos. Si se utiliza una base de datos aquí es donde se realizan las consultas, búsquedas, filtros y actualizaciones. Es conjunto de clases que definen la lógica de negocio de nuestra aplicación. También conocido como modelo del dominio, es una representación en clases de la funcionalidad de nuestra aplicación.

Controlador

Es el encargado de procesar las peticiones del usuario y controlar todo lo que pasa en la aplicación. Como su nombre lo indica, se encarga de controlar, es decir, de gestionar las instrucciones que se reciben, atenderlas y procesarlas. Recibe las órdenes del usuario y se encarga de solicitar los datos al modelo y de comunicárselos a la vista.

Vistas

Las vistas utilizan el modelo y presentan los datos en la forma que el usuario desea. Consisten en páginas estáticas y dinámicas que se representan o envían al usuario cuando éste las solicita.

Por lo general, el usuario interactúa con la Vista, que a su vez genera la solicitud adecuada, esta solicitud será manejada por un controlador. El controlador representa la vista adecuada con los datos del modelo como respuesta.


La arquitectura MVC es una simplificación arquitectónica. El propio esquema de funcionamiento de MVC es variable de una aplicación a otra, y de un programador a otro. Normalmente, en las aplicaciones Node suele haber 5 capas:


Modelo: son los objetos del dominio que únicamente viven en memoria, ya sea en el servidor a nivel de aplicación o en el cliente a nivel de sesión.

Vista: Mostramos al usuario la información, ya sea en aplicaciones web, aplicaciones móviles, de escritorio, de consola.

Controlador: Es el motor de la aplicación. Simplemente tomará decisiones en función de lo devuelto por la capa de servicio.

Servicio: Presenta una API a las aplicaciones Front-End. Suele proporcionar una API que será usada por la aplicación Web, así como por otro tipo de aplicaciones (por ejemplo móviles) a través de servicios Web.

Acceso a Datos: Nos permite acceder a datos. Se encarga de la persistencia de los objetos del modelo del dominio.
