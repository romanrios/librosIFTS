Tipos de aplicaciones


No todas las aplicaciones tienen las mismas características. En función del entorno en el que se ejecutan, podemos distinguir dos grandes grupos: 

Aplicaciones de escritorio

 

Son las aplicaciones más tradicionales que podemos instalar en nuestro equipo. 

Características
  • Tienen, en mayor o menor grado, una buena parte del procesamiento de los datos en el propio dispositivo. 
  • Poseen un lado backend instalado. 
  • La interfaz puede generarse de muchas formas. 
  • Pueden usar componentes del sistema operativo, interfaces web, renderizar componentes propios, etc.


Aplicaciones web

 

Son la piedra angular de internet. Se encargan de gestionar las peticiones de millones de clientes a lo largo de todo el mundo. Incluían una interfaz web para acceder y manipular la información. Sin embargo, evolucionaron hacia aplicaciones distribuidas a lo largo de varios servidores o instancias en la nube.

Características
  • Mantienen la coherencia y la seguridad de los datos.
  • Intercambian información con otras aplicaciones para ofrecer servicios de interés.
  • Inicialmente se basaban en una arquitectura cliente-servidor.
  • Se enfocan en actuar como backend para atender peticiones de múltiples clientes.
  • Utilizan servicios SOAP o REST para la comunicación.

Desarollo web: Front vs Back


El desarrollo web ha transformado nuestro mundo. Cada día accedemos a internet para encontrar información, aprender, comprar productos, compartir opiniones y comunicarnos con familiares y amigos. Los desarrolladores que crean estas aplicaciones web son llamados desarrolladores web. Pueden especializarse para desarrollar partes específicas de una aplicación web:

Tipo de desarrolladores web Características
Desarrolladores web front-end
  • Implementan la parte de la aplicación web con la cual los usuarios interactúan directamente. 
  • Crean la parte visible de las plataformas que usamos cada día. 
  • En la analogía de la tienda que podés observar en la imagen, el front-end estaría representado por la parte de la tienda que los clientes pueden ver.
Desarrolladores web back-end 
  • Implementan toda la funcionalidad que los usuarios no ven, tales como los servidores, bases de datos y todas sus interacciones con el front-end de la aplicación.
Desarrolladores web full-stack 
  • Se encargan de ambas áreas. 
  • Tienen conocimiento extenso tanto sobre desarrollo web front-end y back-end.


¿Qué es frontend?


Frontend es la parte de un programa o dispositivo a la que un usuario puede acceder directamente. Son todas las tecnologías de diseño y desarrollo web que corren en el navegador y que se encargan de la interactividad con los usuarios. 

Los desarrolladores frontend son aquellos que crean los componentes visuales de un sitio web. Dan formato a las imágenes, la animación y la interactividad de un sitio web.


¿Qué es backend?


Backend es la capa de acceso a datos de un software o cualquier dispositivo, que no es directamente accesible por los usuarios. Además, contiene la lógica de la aplicación que maneja dichos datos. El backend también accede al servidor, que es una aplicación especializada que entiende la forma en la que el navegador hace solicitudes.

Los desarrolladores backend se encargan de hacer que la lógica del sitio funcione correctamente, la información se transmita de manera segura y el desempeño de la aplicación no entorpezca la experiencia del usuario.

Desarrolladores de Backend


Un desarrollador backend es un programador que se encarga del diseño y desarrollo de la lógica del servidor de una aplicación web o un sistema. Utilizan lenguajes de programación del lado del servidor , como PHP, Java, Python o Node.js, para crear la funcionalidad que se ejecuta en el servidor y que permite a las aplicaciones del lado del cliente acceder a los datos y las funcionalidades del sistema.

Principales responsabilidades:

Diseño y desarrollo de la lógica del servidor de una aplicación web o un sistema, utilizando lenguajes de programación del lado del servidor.

Creación de APIs (interfaces de programación de aplicaciones) que permitan a las aplicaciones del lado del cliente comunicarse con el servidor y acceder a los datos y las funcionalidades del sistema.

Implementación y gestión de bases de datos, incluyendo la definición de esquemas, la ejecución de consultas y la optimización del rendimiento.

Realización de pruebas y depuración de código para garantizar la calidad y la estabilidad del sistema.

Gestión de versiones y control de cambios en el código fuente mediante herramientas de sistemas de gestión de versiones, como Git.

Colaboración con otros miembros del equipo de desarrollo, como desarrolladores  frontend y diseñadores/as, para garantizar una integración eficiente y una experiencia de usuario consistente.

Participación en la planificación y el diseño del sistema, así como en la toma de decisiones técnicas importantes.

Responsabilidad por el despliegue y el mantenimiento del sistema en entornos de producción y pruebas.

Manejo, documentación y actualización de las librerías del servidor.


¿Cómo funciona Internet?


Internet es una red global de computadoras interconectadas que proporciona una variedad de servicios y contenido. Está compuesta por muchas partes, y entenderlas puede ayudar a entender mejor cómo funciona Internet.


Protocolos y funcionamiento


Los protocolos, son los estándares que permiten que los diferentes dispositivos se conecten entre sí de forma segura y estable. 

Definición

Son un conjunto de reglas estandarizadas que describen cómo los dispositivos se conectan y comunican entre sí a través de una red. Estas reglas dictan qué tipos de mensajes pueden ser enviados entre dispositivos, el formato de los mensajes, cómo los dispositivos se identifican entre sí, cómo los errores son manejados, y cómo los datos se transfieren entre dispositivos. 


Algunos de los protocolos más conocidos incluyen:

  • Protocolo HTTP, 
  • Protocolo de Control de Transmisión (TCP), 
  • Protocolo de Internet (IP), 
  • Protocolo de Transferencia de Archivos (FTP) y,
  • Protocolo de Mensajería de Internet (IMAP).

Flujo de funcionamiento


De manera simple, el flujo de funcionamiento de un servidor web es el siguiente: 

El usuario solicita un archivo a través de un cliente, que puede ser un navegador web.

El navegador web construye una solicitud para dicho archivo.

La solicitud se envía a través del protocolo HTTP a la computadora.

La petición llega a la computadora que contiene el programa; esta computadora también se le conoce como el servidor.

La computadora, a través del programa (servidor web), genera un archivo de respuesta para la petición correspondiente.

El archivo de respuesta se envía de nuevo a través del protocolo HTTP, de vuelta al cliente que hizo la solicitud.

El cliente recibe el archivo de respuesta y se lo muestra al usuario.


Servidores, software y hardware


A continuación vamos a referirnos a tres componentes centrales que conforman la infraestructura de funcionamiento de Internet.

Servidores web

 

Un servidor es una computadora especialmente diseñada para alojar aplicaciones, contenido y servicios para usuarios conectados a través de la web. Estos dispositivos conectados a la red proporcionan contenido y servicios, como alojamiento web, servicios de correo electrónico, bases de datos y almacenamiento en línea.


Características
  • Conexión de Red: Se conectan a la red mediante una conexión de banda ancha de alta velocidad.
  • Alojamiento de Servicios: Ofrecen una variedad de servicios, incluyendo alojamiento web, correo electrónico, bases de datos y almacenamiento en línea.
  • Protocolo HTTP: Utilizan el protocolo HTTP para recibir y responder a las solicitudes de los usuarios.
  • Funciones de Respuesta: Construyen y envían páginas web en formato HTML, además de archivos de utilidad como imágenes, hojas de estilo, archivos JavaScript y PDFs.
  • Servicios Adicionales: Pueden proporcionar seguridad, almacenamiento en caché y análisis de datos.
  • Administración de Red: Son responsables de la administración de la red, facilitando el acceso de los usuarios a los recursos.


Cuando un usuario solicita un recurso, el servidor web recibe la solicitud a través del protocolo HTTP, construye una respuesta y la envía de vuelta al usuario.

Ejemplo

Si un usuario accede a una página web, el servidor web responde enviando el archivo HTML de la página, así como cualquier imagen o archivo asociado, permitiendo que el navegador del usuario muestre el contenido solicitado.


Hardware


Un servidor web puede referirse tanto a hardware como a software, o a ambos trabajando juntos. En términos de hardware, se trata de una computadora que almacena el software de servidor web y los archivos que componen un sitio web, como documentos HTML, imágenes, hojas de estilo CSS y archivos JavaScript.


Ejemplo

Si un usuario accede a una página web, el servidor web proporciona los documentos HTML, las imágenes y los archivos JavaScript necesarios para mostrar el contenido en el navegador del usuario.


Software


En el contexto del software, un servidor web gestiona cómo los usuarios de la web acceden a los archivos alojados en el servidor. Esencialmente, se trata de un servidor HTTP, que es un tipo de software capaz de interpretar URLs (direcciones web) y el protocolo HTTP (el protocolo utilizado por los navegadores para obtener páginas web).

Ejemplo

Cuando un usuario ingresa una URL en su navegador, el servidor HTTP interpreta la URL, procesa la solicitud usando el protocolo HTTP y entrega el contenido del sitio web solicitado al navegador del usuario, permitiendo que la página se visualice correctamente en el dispositivo del usuario final.



Requerimientos


Al nivel más básico, el requerimiento de archivos en un servidor web comienza cuando un navegador necesita acceder a un archivo específico almacenado en el servidor. El proceso se desarrolla de la siguiente manera:

El navegador envía una solicitud de archivo al servidor web mediante el protocolo HTTP.

El servidor web, compuesto por hardware y software, recibe la petición. El hardware se encarga de recibir físicamente la solicitud, mientras que el software del servidor HTTP gestiona la solicitud.

El servidor HTTP busca el archivo solicitado en su almacenamiento. Si encuentra el documento, lo prepara para enviarlo de vuelta al navegador.

El servidor web envía el archivo solicitado al navegador a través del protocolo HTTP. Si el servidor no encuentra el archivo, devuelve una respuesta de error 404, indicando que el documento no se encuentra disponible.

Redes, enrutadores y nodos


Comprende la infraestructura básica de internet, que incluye redes, enrutadores y nodos, esenciales para la comunicación eficiente entre dispositivos.

Redes


Las redes son la columna vertebral de la comunicación digital y se dividen en varias categorías según su alcance y tecnología.

Redes de Área Local (LAN): Actúan como puntos de intersección en la red, conectando diferentes segmentos.

Redes de Área Amplia (WAN):: Conectan dispositivos dentro de un área geográfica limitada, como una oficina o un hogar.

Redes inalámbricas: Utilizan tecnologías inalámbricas para conectar dispositivos sin necesidad de cables, como Wi-Fi y redes móviles.

Enrutadores


Los enrutadores son dispositivos clave que gestionan el tráfico de datos entre redes y determinan la mejor ruta para los paquetes de datos.

Características


Función de los enrutadores

Recepción de paquetes

 Reciben paquetes de datos provenientes de diversas redes.

Determinación de ruta

 Deciden la mejor ruta para cada paquete basándose en la información de enrutamiento.
Transmisión de datos Envía los paquetes al destino correcto, manteniendo la comunicación fluida entre dispositivos conectados.

Protocolos de enrutamiento		 Intercambio de información Utilizan protocolos para intercambiar información con enrutadores vecinos.

Cálculo de ruta

 Determinan la ruta más eficiente basándose en la dirección IP, el número de saltos y el tiempo de tránsito.
Control del tráfico de red Filtrado de tráfico Pueden prevenir el tráfico no deseado o malicioso para mantener la red operativa y segura.

Nodos


Los nodos son los puntos de conexión en una red que proporcionan acceso y conectividad entre diferentes dispositivos. Estos pueden ser:

Nodos de red: Actúan como puntos de intersección en la red, conectando diferentes segmentos.

Nodos de usuario: Dispositivos finales, como computadoras y teléfonos, que se conectan a la red para enviar y recibir datos.

Nodos de acceso: Puntos de entrada a la red, como módems y puntos de acceso Wi-Fi.


Contenido


Esta es la información que los usuarios solicitan cuando usan internet. Esto incluye archivos, imágenes, videos, páginas web y otros.

Tipos de contenidos

Contenido estático Contenidos dinámico
  • Se refiere a contenido que es "servido como está". 
  • Esto significa que el contenido no cambia y se entrega tal como está almacenado en el servidor.
  • Significa que el servidor procesa el contenido o incluso lo genera desde una base de datos antes de entregarlo al usuario. 
  • Esta solución proporciona más flexibilidad, pero también se vuelve más difícil de manejar, haciendo que sea mucho más complejo desarrollar el sitio web.


Ejemplo

Una página web estática puede ser una página de perfil de una empresa que rara vez cambia, mientras que una página web dinámica podría ser un sitio de comercio electrónico donde los productos y precios cambian constantemente en función de la base de datos del servidor.




Navegadores web


Un navegador web es un programa que permite tener acceso a internet, el software cliente que se utiliza para buscar en la red, recuperar y mostrar copias de archivos en un formato que sea fácil de leer. 

Con el navegador se puede enviar y recibir mensajes por correo electrónico y navegar en la World Wide Web para buscar información de toda clase.


Función principal

Un navegador web es una aplicación cuyo principal fin consiste en interpretar documentos HTML, analizando el flujo de marcas que recibe de un servidor o un archivo local, generando una visualización acorde con el contenido y las indicaciones de formato.

Proceso de petición y respuesta

Los navegadores web reciben peticiones HTTP del usuario y las procesan.

Una vez que el navegador recibe la petición, la envía al servidor web que contiene el recurso solicitado.

El servidor web, a su vez, procesa la petición y envía una respuesta al navegador.

Esta respuesta puede contener una variedad de contenido, como HTML, CSS, JavaScript, imágenes y otros tipos de contenido.

El navegador recibe esta respuesta y la procesa para mostrar el contenido correctamente.

Compatibilidad

Existen navegadores para diferentes dispositivos y plataformas. Cada uno con sus propias características, pero todos tienen la misma finalidad: ofrecer al navegante una representación lo más fiel posible del recurso al que está accediendo, que, por lo general, será un sitio web o una aplicación web.

Ejemplo

Cuando un usuario ingresa una URL en la barra de direcciones del navegador, este envía una petición HTTP al servidor que aloja el sitio web correspondiente. El servidor procesa la petición y devuelve el contenido solicitado, que el navegador interpreta y muestra en la pantalla del usuario.



Algunos navegadores web


Algunos de estos navegadores son:

Google Chrome: es un navegador web de código propietario desarrollado por Google. Chrome es actualmente el navegador más utilizado de internet. Está disponible gratuitamente para diversas plataformas (Windows, MacOS, Android, iOS, Linux...) 

Safari: es un navegador web de código cerrado desarrollado por Apple Inc. Está disponible para Mac OS X, iOS (el sistema usado por iPhone, iPod Touch y iPad) y Microsoft Windows. 

Microsoft Internet Explorer: conocido comúnmente como IE, es un navegador web desarrollado por Microsoft para el sistema operativo Microsoft Windows desde 1995. Ha sido el navegador web más utilizado durante años, con un pico máximo de cuota de utilización del 95% entre el 2002 y 2003. Sin embargo, dicha cuota de mercado ha disminuido paulatinamente con los años. Microsoft anunció que a partir de Windows 10 dejará de publicar versiones de este navegador para sustituirlo por Edge. 

Mozilla Firefox: es un navegador web libre y de código abierto descendiente de Mozilla Application Suite y desarrollado por la Fundación Mozilla. Firefox puede ser utilizado en varios sistemas operativos (Windows, MacOS, Android, iOS, Linux...)

Direcciones web


Una dirección web para hacer una solicitud con Http puede verse de las siguientes maneras:

http://localhost:8000

https://www.google.com

https://web.whatsapp.com

https://twitter.com/explore


Todas son direcciones válidas y cada una tiene elementos distintos. Lo que aparece al principio (luego de http:// o https://) se trata del nombre de dominio. 

Nombre de dominio

El nombre de dominio nos permite identificar la computadora en la que se encuentra el recurso que estamos solicitando. Es importante notar cómo en lugar de página estamos usando el concepto de recurso, porque así como podemos solicitar una página, también puede ser otro tipo de archivo como una imagen.

Puerto

Luego del nombre de dominio viene el puerto. En el primer ejemplo, el puerto es el 8000, que aparece luego de los dos puntos. En el resto de los ejemplos se asume el puerto por defecto, porque no hay otro especificado. El puerto por defecto para el protocolo HTTP es el puerto 80, y es el que precisamente se usa cuando no se especifica otro, como en los ejemplos mencionados.


Ruta


Además del dominio y el puerto, una dirección web contiene la ruta o el path en donde se encuentra el recurso, dentro del servidor. 

Pensá en que si el dominio fuera una ciudad, el path sería la ruta para llegar a una casa en específico, ya que si bien el dominio representa al servidor, el path representa el recurso en específico que queremos. 


El path por defecto es la /, y puede omitirse (en esos casos asumimos que, si no hay path, estamos tratando de acceder al inicio, la /). En nuestros ejemplos de direcciones, el path es siempre / con excepción del último donde se indica que buscamos el recurso con la ruta / explore.



El modelo cliente-servidor


Internet está basado en el modelo o la arquitectura cliente-servidor, en la cual dos dispositivos (el cliente y el servidor) se comunican entre sí.

El cliente

Cuando accedemos a una página web en el navegador, el cliente (navegador) envía una solicitud HTTP al servidor.

El servidor

El servidor es un programa que escucha solicitudes y genera respuestas apropiadas. Estas respuestas incluyen:

Enviar información al cliente.

Ejecutar tareas o procesos.

Trabajar con la información de una base de datos y/o actualizarla.


Ejemplo

Podemos agregar un usuario nuevo a una base de datos al enviar una solicitud al servidor y el servidor debe realizar las actualizaciones correspondientes en la base de datos.


Desarrollar y mantener servidores es una de las tareas principales de los desarrolladores backend.


Ventajas


El modelo cliente-servidor tiene varias ventajas en el desarrollo de aplicaciones web, como:

Escalabilidad: el modelo cliente-servidor permite que el servidor web pueda manejar un gran número de solicitudes de manera simultánea, lo que facilita la creación de aplicaciones web que puedan soportar un alto tráfico y un alto número de usuarios.

Mantenibilidad: el modelo cliente-servidor permite a los/las desarrolladores/as actualizar y mejorar el lado del servidor de la aplicación de manera aislada, sin afectar al funcionamiento del lado del cliente. Esto facilita la actualización y el mantenimiento de la aplicación.

Seguridad: el modelo cliente-servidor permite a los desarrolladores implementar medidas de seguridad en el lado del servidor, como autenticación y autorización de usuarios, cifrado de datos y protección contra ataques, lo que mejora la seguridad de la aplicación.

Interoperabilidad: el uso de protocolos estandarizados como HTTP en el modelo cliente-servidor facilita la interoperabilidad entre diferentes aplicaciones y sistemas, lo que permite a las aplicaciones web interactuar y compartir datos con otros sistemas de manera sencilla y consistente.

Desarrollo basado en servidores


El desarrollo de aplicaciones web basadas en servidor se refiere a la creación de sitios web y aplicaciones que se ejecutan en un servidor web y se acceden a través de un navegador web. Estas aplicaciones utilizan lenguajes de programación del lado del servidor para generar contenido dinámico que se muestra en el navegador del usuario.


Beneficios de las aplicaciones web basadas en servidor

Uno de los principales beneficios de las aplicaciones web basadas en servidor es que el código se ejecuta en el servidor. Esto significa que sólo se envían los resultados al navegador del usuario, lo que reduce la carga en el dispositivo del usuario y aumenta la velocidad y el rendimiento de la aplicación.


Accesibilidad y facilidad de uso

Otra ventaja de este tipo de desarrollo es que las aplicaciones se pueden acceder desde cualquier lugar con una conexión a internet y un navegador web, lo que las hace altamente accesibles y fáciles de usar.


Protocolo HTTP


Hypertext Transfer Protocol (HTTP) (o Protocolo de Transferencia de Hipertexto, en español) es un protocolo para la transmisión de documentos hipermedia, como HTML. 

Fue diseñado para la comunicación entre los navegadores y servidores web, aunque se puede utilizar para otros propósitos también.


HTTP sigue el clásico modelo cliente-servidor, en el que un cliente establece una conexión con el servidor, realiza una petición y espera hasta que recibe una respuesta del mismo. Un servidor web brinda soporte para HTTP. 

Además, HTTP proporciona reglas claras sobre cómo se comunican un cliente y un servidor. Sólo los clientes pueden hacer solicitudes HTTP a los servidores. Los servidores solo pueden responder a la solicitud HTTP de un cliente.

Como su nombre lo indica, HTTP especifica cómo transferir hypertext (es decir, documentos web vinculados) entre dos computadoras. HTTP es un protocolo textual, sin estado.


Características HTTP
Textual Eje de prácticas de habilidades socio-productivas.
  • Todos los comandos son de texto plano y legible para ser leído por humanos.
  • Ni el servidor ni el cliente recuerdan las comunicaciones anteriores. 
  • Por ejemplo, al confiar sólo en HTTP, un servidor no puede recordar la contraseña que ingresó ni el paso que está realizando en una transacción. Necesita un servidor de aplicaciones para tareas como esa.

Al solicitar un archivo a través de HTTP, los clientes deben proporcionar la URL del archivo. El servidor web debe responder a todas las solicitudes HTTP, al menos con un mensaje de error.



Códigos y verbos en HTTP


Antes de comenzar a hacer peticiones con Http a un servidor, primero debemos saber que hay distintos tipos de peticiones que podemos hacer hacia el servidor mismo. 

Verbos

El protocolo HTTP utiliza diferentes verbos para las diferentes operaciones que se realizan y dependiendo el resultado de estas operaciones vienen con diferentes códigos. Estos verbos indican qué acción queremos realizar sobre el servidor y los más populares son GET y POST. Cada verbo indica una acción diferente a la que el servidor debe responder.

Los verbos más utilizados por defecto en la Web son:

GET: Para operaciones de consulta de datos. Podemos consultar un listado de usuarios o consultar una usuario determinado. Ambas operaciones serían GET, solo que la consulta al listado se realizaría sobre el recurso /users y la consulta a un usuario particular /beer/7437857 o algo parecido, siendo el número “7437857” el ID o identificador de ese usuario.

POST: Operaciones de inserción.

PUT: Para realizar modificaciones.

PATCH: Las partes están dinámicamente interrelacionadas o son interdependientes.

DELETE: Para eliminar un recurso dado.

OPTIONS: Se utiliza como petición auxiliar para comprobar si el servidor soporta el método que se va a utilizar en general cualquiera diferente a GET, esta comprobación se llama preflight.

HEAD: Indica que únicamente queremos que se responda con los encabezados de la respuesta, y se ignore el cuerpo de datos.


Codificación adicional


Se puede codificar información adicional al enviar una petición (por ejemplo, datos de un formulario HTML). La información puede ser codificada como:

Parámetros URL: Las peticiones GET codifican los datos en la URL enviada al servidor añadiendo al final pares nombre/valor — por ejemplo, http://mysite.com?name=Fred&age=11. Siempre encontrarás un signo de interrogación (?) separando los parámetros URL del resto de la misma, un signo igual (=) separando cada nombre de su valor asociado y un ampersand (&) separando cada par. Los parámetros URL son inherentemente inseguros ya que pueden ser modificados por los usuarios y ser enviados de nuevo. Como consecuencia los parámetros URL/peticiones GET no se usan para peticiones de actualización de datos en el servidor.

Datos POST: Las peticiones POST añaden nuevos recursos, cuyos datos están codificados dentro del cuerpo de la petición.

Cookies de lado cliente: Las Cookies contienen datos de sesión acerca del cliente, incluyendo las claves que el servidor puede usar para determinar su estado de inicio de sesión y los permisos/accesos a los recursos.


Códigos de respuesta


Los códigos de respuesta HTTP son mensajes de estado de código de tres dígitos que envían los servidores web a los navegadores web para indicar el resultado de la solicitud de un recurso. Estos códigos de estado se usan para informar al navegador web del resultado de la solicitud para que el navegador sepa qué acción debe tomar a continuación.

Dentro del listado de códigos tenemos:

2xx: Este grupo se utiliza para operaciones exitosas.

  • 200: Cuando la operación es exitosa.
  • 201: Lo mismo que el 200 pero en este caso cuando además se crea un recurso de manera exitosa.
  • 204: Lo mismo que el 201 pero esta vez no fue necesario enviar ningún tipo de información de vuelta

3xx: Este grupo se utiliza para indicar redirecciones.

4xx: Este grupo se utiliza para enumerar los errores del lado del cliente.

  • 400: Indica que la petición no fue bien realiza normalmente cuando los datos se envían mal formados.
  • 401: No está autorizado para acceder al recurso, normalmente se utiliza cuando no se está autenticado.

  • 403: Puede que esté autenticado pero no tiene permisos para acceder a ese recurso.
  • 404: Recurso no encontrado.

  • 422: Indica que la petición no se puede procesar debido a que faltan datos requeridos o están en un formato incorrecto.

5xx: Este conjunto está asociado a errores del lado del servidor, se usan para indicar que el servidor ha encontrado un error al procesar la solicitud. 



HTTP Caching


Una de las formas más efectivas de mejorar el rendimiento de una aplicación es mediante un buen sistema de caché. 

Definiciones

  • La caché HTTP almacena una respuesta asociada con una solicitud y reutiliza la respuesta almacenada para solicitudes posteriores.
  • Caching es el proceso de guardar respuestas reusables para después poder cargar requests sucesivos de forma más rápida. 


Función de HTTP Cache

Todas las solicitudes HTTP que realiza el navegador se enrutan primero al caché del navegador para comprobar si hay una respuesta almacenada en caché que sea válida que se pueda utilizar para cumplir con la solicitud. Si hay una, la respuesta se lee de la caché, lo que elimina tanto la latencia de la red como los costos de datos en los que incurre la transferencia.


HTTP cookies


HTTP es un protocolo sin estado, es decir: no guarda ningún dato entre dos peticiones en la misma sesión. Esto crea problemáticas, en caso de que los usuarios requieran interactuar con determinadas páginas Web de forma ordenada y coherente, por ejemplo, para el uso de "carritos de compra" en páginas que utilizan el comercio electrónico. Pero, mientras HTTP ciertamente es un protocolo sin estado, el uso de HTTP cookies, si permite guardar datos con respecto a la sesión de comunicación. Usando la capacidad de ampliación del protocolo HTTP, las cookies permiten crear un contexto común para cada sesión de comunicación.

Nuevas

Una cookie HTTP, cookie web o cookie de navegador es una pequeña pieza de datos que un servidor envía al navegador web del usuario. El navegador guarda estos datos y los envía de regreso junto con la nueva petición al mismo servidor. Las cookies se usan generalmente para decirle al servidor que dos peticiones tienen su origen en el mismo navegador web lo que permite, por ejemplo, mantener la sesión de un usuario abierta. Las cookies permiten recordar la información de estado en vista a que el protocolo HTTP es un protocolo sin estado.

Las cookies se utilizan principalmente con tres propósitos:

Gestión de Sesiones: Inicios de sesión, carritos de compras, puntajes de juegos o cualquier otra cosa que el servidor deba recordar.

Personalización: Preferencias de usuario, temas y otras configuraciones.

Rastreo: Guardar y analizar el comportamiento del usuario.

Las cookies se envían con cada solicitud, por lo que pueden empeorar el rendimiento (especialmente para las conexiones de datos móviles).

Sitio web estático


Un sitio estático es aquél que devuelve desde el servidor el mismo contenido establecido de forma fija en el código cada vez que se solicita una página en particular. De manera que si por ejemplo tienes una página sobre un producto en /static/myproduct1.html, a todos los usuarios se les devolverá la misma página. Si añadimos otro producto similar a nuestro sitio necesitaremos añadir otra página (ej. myproduct2.html) etc... Esto puede llegar a ser realmente muy poco eficiente.


¿Qué sucede si tenemos miles de productos?

Deberíamos repetir un montón de código a lo largo de cada página (la plantilla básica de la página, la estructura, etc), y si quisieras cambiar cualquier cosa de la estructura de la página (cómo añadir una nueva sección de "productos relacionados"), tendríamos que cambiar cada página individualmente.

Los sitios estáticos son excelentes cuando tenemos un pequeño número de páginas y queremos enviar el mismo contenido a todos los usuarios. Sin embargo pueden tener un coste de mantenimiento significante a medida que el número de páginas se hace grande.


Cuando un usuario quiere navegar a una página, el explorador envía una petición HTTP GET especificando la URL de la página HTML. El servidor recupera el documento solicitado de su sistema de ficheros y devuelve una respuesta HTTP conteniendo el documento y un código de estado de respuesta HTTP "200 OK" (indicando éxito). El servidor podría devolver un código de estado diferente, por ejemplo "404 Not Found" si el fichero no está presente en el servidor.

El servidor de un sitio estático sólo tendrá que procesar peticiones GET, ya que el servidor no almacena ningún dato modificable. Tampoco cambia sus respuestas basándose en los datos de la petición HTTP (ej. parámetros URL o cookies).

Sitio web dinámico


Un sitio dinámico es aquél que puede generar y devolver contenido basándose en la URL y los datos específicos de la petición (en vez de devolver siempre para una URL en particular el mismo fichero especificado en el código de forma fija). Usando el ejemplo de un sitio de productos, el servidor almacenará datos del producto en una base de datos en vez de ficheros HTML individuales. Cuando se recibe una petición HTTP GET para un producto, el servidor determina el ID del mismo, extrae los datos de la base de datos y construye la página HTML de la respuesta insertando los datos dentro de la plantilla HTML.

Esto tiene una ventaja primordial sobre un sitio estático: Usar una base de datos permite que la información del producto se almacene de forma eficiente y que se pueda ampliar, modificar y buscar fácilmente.


Usar plantillas HTML hace que sea muy fácil cambiar la estructura HTML, porque sólo se necesita hacer en un sólo lugar, en una única plantilla y no a lo largo de miles de páginas estáticas.

API


Una API (Application Programming Interface) es un conjunto de protocolos, herramientas y subrutinas que permiten a los/las desarrolladores/as crear aplicaciones que se comunican entre sí. Una API ofrece una interfaz para que los usuarios interactúen con esas aplicaciones.

Características

Las APIs permiten a los desarrolladores crear aplicaciones usando lenguajes de programación existentes, como Java, Python, etc.

Se utilizan ampliamente en el desarrollo de back-end para permitir que los desarrolladores creen aplicaciones web, bases de datos y otros servicios.

Ayudan a ahorrar tiempo y esfuerzo al permitir que los desarrolladores se concentren en la lógica de la aplicación en lugar de en la implementación de la base de datos.

Las APIs también se pueden utilizar para conectar aplicaciones de terceros.


 

Ejemplo

Un/a desarrollador/a de back-end puede utilizar una API para acceder a una base de datos y recuperar datos específicos para una aplicación. Además, una aplicación de gestión de contenido puede conectarse a una API de redes sociales para permitir a los usuarios compartir contenido en las redes sociales.


API REST


API REST es un conjunto de principios arquitectónicos para crear redes de servicios web que permiten la comunicación entre sistemas. Estas APIs son aplicables a una amplia variedad de protocolos y tecnologías web, y su diseño se basa en el intercambio de datos en formato JSON o XML. 

Concepto de REST

REST deriva de "REpresentational State Transfer", que significa “transferencia de representación de estado”.

Un servicio REST no tiene estado (stateless), lo que implica que, entre dos llamadas cualesquiera, el servicio pierde todos sus datos. Esto es, que no se puede llamar a un servicio REST y pasarle unos datos (p. ej. un usuario y una contraseña) y esperar que “nos recuerde” en la siguiente petición.

El estado lo mantiene el cliente, quien debe pasar el estado en cada llamada, como usuario y contraseña, un token, u otras credenciales. Y lo mismo aplica para el resto de información. El no tener estado es una desventaja clara: tener que pasar el estado en cada llamada es, como mínimo, tedioso, pero la contrapartida es clara: escalabilidad.

. 

Una API REST funciona mediante el uso de peticiones HTTP para realizar tareas básicas, como leer o escribir datos. Estas peticiones se pueden realizar mediante el uso de métodos HTTP, como GET, POST, PUT, DELETE, etc. Estos métodos también los conocemos como los verbos HTTP.

    ¿Qué es Node.js?


    Node.js es un runtime de JavaScript asíncrono y basado en eventos desarrollado en base al motor V8 de JavaScript de Chrome. Nos da todas las herramientas que necesitamos para ejecutar JavaScript en el terminal sin un navegador web (browser). 

    Node.js es la forma de ejecutar Javascript en un servidor. Con Node podemos aprovechar grandes ventajas de Javascript como los eventos y la ejecución asíncrona para crear aplicaciones en tiempo real.

    JavaScript se introdujo en 1995 como una forma de agregar programas a páginas web en el navegador Netscape Navigator. El lenguaje ha sido desde entonces adoptado por todos los otros navegadores web principales. Ha hecho que las aplicaciones web modernas sean posibles: aplicaciones con las que puedes interactuar directamente, sin hacer una recarga de página para cada acción. JavaScript también es utilizado en sitios web más tradicionales para proporcionar diversas formas de interactividad e ingenio.

    NPM


    NPM (Node Package Manager) es un administrador de paquetes para el lenguaje de programación JavaScript y el entorno de ejecución Node.js. NPM proporciona una plataforma en la nube y una herramienta de línea de comandos que facilitan la descarga, instalación, actualización y gestión de paquetes de código reutilizable para Node.js. 

    Los paquetes de NPM se pueden utilizar para agregar funcionalidades a las aplicaciones de Node.js, como librerías de utilidades, frameworks de desarrollo o herramientas de línea de comandos. NPM almacena una gran cantidad de paquetes de código abierto, que pueden ser utilizados de forma gratuita por cualquier desarrollador/a. 


    Además, NPM también proporciona un sistema de publicación y distribución para que los/las desarrolladores/as puedan compartir sus propios paquetes con la comunidad. Esto hace que sea fácil colaborar en proyectos de código abierto y reutilizar código entre diferentes aplicaciones de Node.js.

    El comando Node


    Cuando Node.js se instala en un sistema, proporciona un programa llamado node, que se utiliza para ejecutar archivos JavaScript. A continuación, veremos algunos ejemplos básicos de cómo usar Node.js.

    Digamos que tenemos un archivo hello.js, que contiene este código:

    HTML Copiar código 
    let message = 'Hello world';
    console.log(message);


    Entonces podrás lanzar node desde la línea de comandos de esta manera para ejecutar el programa:

    CSS Copiar código 
    $ node hello.js
    Hello world


    El método console.log en Node hace algo similar a lo que realiza en el navegador. Imprime una porción de texto. Pero en Node, el texto irá al flujo de la salida estándar del proceso, en lugar de la consola de JavaScript del navegador. Cuando ejecutamos Node desde la línea de comandos, eso significa que veremos los valores escritos en nuestro terminal.

    Si ejecutamos node sin indicar un archivo, se nos proporciona un prompt en el cual podremos escribir código JavaScript e inmediatamente ver el resultado.

    JavaScript Copiar código 
    $ node
    > 1 + 1
    2
    > [-1, -2, -3].map(Math.abs)
    [1, 2, 3]
    > process.exit(0)
    $


    La referencia process, al igual que la referencia console, está disponible globalmente en Node. Esta proporciona varias formas de inspeccionar y manipular el programa actual. El método exit finaliza el proceso y se le puede pasar un código de estado de salida, esto le dice al programa que inició node (en este caso, el shell de la línea de comandos) si el programa finalizó con éxito (código cero) o si se encontró con un error (cualquier otro código).

    Para obtener los argumentos de la línea de comandos proporcionados a tu script, podés leer process.argv, que es un conjunto de cadenas. Tené en cuenta que también incluye el nombre del comando node y el nombre de tu script, por lo que los argumentos reales comienzan en el índice 2. Si showargv.js contiene la sentencia:

    console.log(process.argv);

    podrías ejecutarlo así:

    CSS Copiar código 
    $ node showargv.js one --and two


    Todas las referencias globales JavaScript estándar, como Array, Math y JSON, también están presentes en el entorno de Node. La funcionalidad relacionada con el navegador, como el documento o el prompt, no lo son.
