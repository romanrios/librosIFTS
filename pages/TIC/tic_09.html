Conectá saberes técnicos


La importancia de los saberes técnicos de un/a desarrollador/a de software se manifiesta al abordar preguntas cruciales sobre la gestión de dominios y la seguridad del sistema de nombres de dominio (DNS). El registro de dominio es esencial para la identificación única en Internet y su correcto funcionamiento. 

¿Qué deben comprender los/as desarrolladores/as?

Los desarrolladores deben comprender el papel del DNS, que traduce nombres de dominio en direcciones IP, y su relación con los servidores de nombres para facilitar la navegación web. Los desafíos actuales en la seguridad del DNS, como ataques de envenenamiento de caché, destacan la necesidad de medidas de protección avanzadas. 

La vulnerabilidad del DNS tiene implicaciones significativas para la ciberseguridad y la infraestructura crítica, subrayando la importancia de que los desarrolladores fortalezcan la seguridad en esta área vital de la conectividad digital.


Y..., ¿para qué estos saberes técnicos?

  • Proceso de registro y gestión de dominios: Comprender el proceso de registro de dominios es crucial, desde la selección y compra del dominio hasta su configuración y mantenimiento. Los/as desarrolladores/as deben conocer cómo gestionar registros DNS, ajustar configuraciones como los registros MX y CNAME, y entender las políticas de registro para garantizar una gestión eficiente y precisa.
  • Relación entre DNS y servidores de nombres: Los/as desarrolladores/as deben comprender la estrecha relación entre el DNS y los servidores de nombres. Entender cómo funciona la resolución de nombres de dominio y cómo se organizan los servidores de nombres es esencial para optimizar la eficiencia y la velocidad de las operaciones DNS.
  • Desafíos y medidas de seguridad del DNS: Conocer los desafíos actuales en la seguridad del DNS, como ataques de envenenamiento de caché o suplantación de identidad (spoofing), permite a los/as desarrolladores/as implementar medidas de seguridad efectivas. Estar al tanto de las vulnerabilidades y emplear técnicas como DNSSEC (DNS Security Extensions) es esencial para proteger la integridad del sistema de nombres de dominio.

DNS: El sistema de nombres de dominio


El DNS (Sistema de Nombres de Dominio) es definitivamente un aspecto importante de la administración de redes. Se utiliza para traducir nombres de dominio legibles por humanos, en direcciones IP numéricas que las computadoras y dispositivos utilizan para comunicarse en la red. Veamos de qué se trata.

Una "guía telefónica" de direcciones

Los programas pueden hacer referencia a páginas web, buzones de correo y otros recursos mediante las direcciones de red (por ejemplo, IP) de las computadoras en las que se almacenan. Sin embargo, a las personas se les dificulta recordar estas direcciones. Además, navegar en las páginas web de una empresa desde un servidor con la dirección 128.111.24.41, significa que si la compañía mueve el servidor web a una máquina diferente con una dirección IP distinta, hay que avisar a todos sobre la nueva dirección IP. 


Por este motivo, se introdujeron nombres legibles de alto nivel con el fin de separar los nombres de máquina de las direcciones de máquina. 

De esta manera, el servidor web de la empresa podría conocerse como www.cs.washington.edu sin importar cuál sea su dirección IP. Sin embargo, como la red sólo comprende direcciones numéricas, se requiere algún mecanismo para convertir los nombres en direcciones de red.



Historia de los nombres de dominio 


Hace mucho, en los tiempos de ARPANET (Advanced Research Projects Agency Network) precursora de internet, sólo había un archivo, hosts.txt, en el que se listaban todos los nombres de computadoras y sus direcciones IP. Cada noche, todos los hosts obtenían este archivo del sitio en el que se almacenaba. En una red conformada por unos cuantos cientos de grandes máquinas de tiempo compartido, este método funcionaba razonablemente bien.

Sin embargo, antes de que se conectaran millones de computadoras PC a Internet, los involucrados se dieron cuenta de que este método no funcionaría eternamente. El tamaño de cada archivo crecería de manera considerable. No obstante, un problema aún mayor eran los constantes conflictos con los nombres de los hosts, a menos que dichos nombres se administraran en forma central (algo impensable en una enorme red internacional debido a su carga y latencia).

Para resolver estos problemas, en 1983 se inventó el DNS (Sistema de Nombres de Dominio, del inglés Domain Name System), el cual ha sido una parte clave de Internet desde entonces.

La esencia del DNS es la invención de un esquema jerárquico de nombres basado en dominios y un sistema de base de datos distribuido para implementar este esquema de nombres. 


El DNS se usa principalmente para asociar los nombres de host con las direcciones IP, pero también se puede usar para otros fines.

Dicho de otro modo, la forma en que se utiliza el DNS es la siguiente:


Para asociar un nombre con una dirección IP, un programa de aplicación invoca un procedimiento de biblioteca conocido como resolvedor y le proporciona el nombre como parámetro.

El resolvedor envía una consulta que contiene el nombre a un servidor DNS local.

El servidor DNS local busca el nombre dentro de su base de datos y devuelve una respuesta que contiene la dirección IP al resolvedor.

Posteriormente, el resolvedor transfiere la dirección IP al solicitante del programa.

Tanto los mensajes de solicitud como las respuestas se envían como paquetes UDP (Protocolo de Datagrama de Usuario).

Con la dirección IP en su poder, el programa puede establecer una conexión TCP con el host correspondiente o enviarle paquetes UDP.



El espacio de nombres del DNS


La administración de un conjunto grande de nombres que cambian en forma continua no es un problema sencillo. En el sistema postal, para administrar los nombres se requieren letras que especifiquen (de manera implícita o explícita) el país, estado o provincia, ciudad, calle y nombre del/la destinatario/a. Con este tipo de direccionamiento jerárquico, se asegura que no haya confusión entre el Alberto Rodriguez, de Av. Libertador 456, en Merlo, San Luis y el Alberto Rodriguez, de Av. Castro Barros 1345, Córdoba, Córdoba. El DNS funciona de la misma manera.

ICANN 

En Internet, el nivel superior de la jerarquía de nombres se administra mediante una organización llamada ICANN (Corporación de Internet para la Asignación de Nombres y Números, del inglés Internet Corporation for Assigned Names and Numbers), la cual se creó para este fin en 1998 como parte del proceso de maduración de Internet, que se convirtió en un asunto económico a nivel mundial. 

En concepto, Internet se divide en más de 250 dominios de nivel superior, cada uno de los cuales abarca muchos hosts. 

Cada dominio se divide en subdominios, los que a su vez también se dividen, y así en lo sucesivo. 

Todos estos dominios se pueden representar mediante un árbol, como se muestra en la figura 1; las hojas del árbol representan los dominios que no tienen subdominios (pero que, por supuesto, contienen máquinas). 

Un dominio de hoja puede contener un solo host, o puede representar a una compañía y contener miles de hosts.

Polémicas en torno a los nombres


Sin embargo, a medida que Internet se ha vuelto más comercial e internacional, también ha generado más polémica, en especial en cuestiones relacionadas con los nombres. Esta controversia incluye a la misma ICANN. 

Ejemplo

La creación del dominio "xxx" tardó varios años y juicios para resolverse. ¿Es bueno o malo colocar de manera voluntaria el contenido para adultos en su propio dominio? (Algunas personas no querían ningún contenido para adultos disponible en Internet, mientras que otros querían ponerlo todo en un solo dominio, de modo que los filtros de protección pudieran encontrar esos sitios fácilmente y apartarlos de los niños). 


Algunos de los dominios se organizan de manera automática, mientras que otros tienen restricciones en cuanto a quién pueden obtener un nombre, como se observa en la figura 2. Pero, ¿qué restricciones son apropiadas? 

Ejemplo

Tomemos como ejemplo el dominio pro. Es para profesionales calificados/as; pero, ¿a quién se puede designar profesional? Es evidente que los/las doctores/as y abogados/as son profesionales. Y, ¿qué hay sobre los/las fotógrafos/as independientes, profesores/as de piano, magos, plomeros, peluqueros, exterminadores, tatuadores, mercenarios y prostitutas? ¿Son elegibles estas ocupaciones? ¿De acuerdo con quién?


También hay dinero en los nombres. Tuvalu (el país de Oceanía) otorgó un contrato de arrendamiento de su domino tv por $50 millones, todo porque el código del país se adapta bien para promocionar los sitios de televisión. Casi cualquier palabra común (en inglés) está ocupada en el dominio .com, además de los errores ortográficos más comunes. Igualmente con artículos para el hogar, animales, plantas, partes del cuerpo, etc. La práctica de registrar un dominio con miras a venderlo después a una parte interesada a un precio mucho mayor tiene incluso su propio nombre: ciberocupación (cybersquatting). Muchas empresas que tuvieron un arranque lento cuando empezó la era de Internet, descubrieron que sus nombres de dominio ya estaban ocupados para cuando trataron de adquirirlos. En general, mientras no se violen marcas registradas y no haya fraude, el primero que llega es el primero que obtiene el nombre. Sin embargo, aún se están refinando políticas para resolver las disputas de nombres.



Características de cada dominio


Los nombres de dominio pueden ser absolutos o relativos, lo que significa que pueden proporcionarse con o sin un punto al final. Por ejemplo:

  • Absoluto: eng.cisco.com.
  • Relativo: eng.cisco.com

Un nombre de dominio absoluto siempre termina con un punto, mientras que uno relativo no lo hace. Esto permite una interpretación clara de los nombres de dominio en cualquier contexto. Por ejemplo:

  • Absoluto: eng.cisco.com.
  • Relativo: eng.cisco.com

Los nombres relativos se interpretan en un contexto específico para determinar su significado verdadero. Por ejemplo, dentro del dominio cisco.com, el nombre eng se refiere a un departamento de ingeniería, mientras que fuera de este contexto, su significado podría ser diferente.

Un dominio nombrado hace referencia a un nodo específico y a todos los nodos por debajo de él en la jerarquía de nombres de dominio. Por ejemplo, eng.cisco.com. hace referencia al departamento de ingeniería en la organización Cisco, así como a todos los subdominios que puedan existir bajo él.

No hay distinción entre mayúsculas y minúsculas en los nombres de dominio, lo que significa que "edu", "Edu" y "EDU" se consideran iguales. Esto simplifica la interpretación y el manejo de los nombres de dominio en sistemas de nombres de dominio (DNS).

Los nombres de componentes en un nombre de dominio pueden tener hasta 63 caracteres de longitud. Esto proporciona suficiente flexibilidad para nombres descriptivos y específicos. Por ejemplo, en cs.washington.edu, "washington" es el componente más largo.

Las rutas completas de los nombres de dominio no deben exceder los 255 caracteres, lo que garantiza que los nombres de dominio sean manejables y se ajusten a las limitaciones de los sistemas DNS y otros sistemas relacionados con nombres de dominio.

Los dominios pueden insertarse en el árbol de nombres de dominio en dominios genéricos o de países. Por ejemplo, cs.washington.edu también podría estar listado bajo el dominio de país us como cs.washington.wa.us, lo que muestra la flexibilidad y la estructura jerárquica del sistema de nombres de dominio.

Grandes empresas a menudo se registran bajo múltiples dominios de nivel superior para diferenciar entre sus divisiones, sucursales o servicios. Por ejemplo, sony.com, sony.net y sony.nl representan diferentes divisiones o localizaciones de la empresa Sony Corporation.

Cada dominio controla cómo se asignan los dominios debajo de él, lo que permite una gestión eficaz de los recursos y la organización de los nombres de dominio. Por ejemplo, Japón tiene dominios como ac.jp y co.jp, mientras que los Países Bajos colocan todas las organizaciones directamente bajo nl.



Registro de recursos de dominio


Cada dominio, sea un host individual o un dominio de nivel superior, puede tener un grupo de registros de recursos asociados a él. Estos registros son la base de datos del DNS. 

En un host individual, el registro de recursos más común es simplemente su dirección IP, pero también existen muchos otros tipos de registros de recursos.

 Cuando un resolvedor asigna un nombre de dominio al DNS, lo que recibe son los registros de recursos asociados a ese nombre. Por lo tanto, la función principal del DNS es relacionar los nombres de dominios con los registros de recursos.

Elementos

Un registro de recursos es una 5-tupla, es decir, un conjunto de cinco elementos que describen distintos aspectos del registro, los cuales proporcionan detalles específicos sobre cómo se debe manejar el nombre de dominio en cuestión. Aunque éstas se codifican en binario por cuestión de eficiencia, en la mayoría de las presentaciones, los registros de recursos se presentan como texto ASCII, una línea por registro de recursos. 


El formato que usaremos es el siguiente: 

Nombre_dominio Tiempo_de_vida Clase Tipo Valor

  • - El Nombre_dominio indica el dominio al que pertenece este registro. Por lo general, existen muchos registros por dominio y cada copia de la base de datos contiene información sobre múltiples dominios. Por lo tanto, este campo es la clave primaria de búsqueda que se utiliza para atender las consultas. El orden de los registros de la base de datos no es importante.
  • - El campo de Tiempo_de_vida es una indicación de la estabilidad del registro. La información altamente estable recibe un valor grande, como 86 400 (la cantidad de segundos en 1 día). A la información que es altamente volátil se le asigna un valor pequeño, como 60 (1 minuto). 
    - El tercer campo de cada registro de recursos es la Clase. Para información de Internet, siempre es IN. Para información que no es de Internet se pueden utilizar otros códigos, pero en la práctica, éstos raras veces se ven.
  • - El campo Tipo indica el tipo de registro del que se trata. Hay muchos tipos de registros de DNS. En la figura 3 se listan los más importantes.


Figura 3. Los principales tipos de registros de recursos de DNS.



Tipos de registro


Tipo Descripción
Registro SOA Proporciona el nombre de la fuente primaria de información sobre la zona del servidor de nombres, la dirección de correo electrónico de su administrador, un número de serie único, varias banderas y temporizadores.
Registro A El tipo de registro más importante, que contiene una dirección IPv4 de 32 bits de una interfaz para algún host.
Registro AAAA Contiene una dirección IPv6 de 128 bits. Cada host de Internet debe tener al menos una dirección IP, para que otras máquinas se puedan comunicar con él.
Registro MX Especifica el nombre del host que está preparado para aceptar correo electrónico del dominio especificado. Se utiliza porque no todas las máquinas están preparadas para aceptar correo electrónico.  Si alguien desea enviar correo electrónico a, por ejemplo, bill@microsoft.com, el host emisor necesita encontrar un servidor de correo en microsoft.com que esté dispuesto a aceptar correo electrónico. El registro MX puede proporcionar esta información.
Registro NS Especifica un servidor de nombres para el dominio o subdominio. Es un host que tiene una copia de la base de datos para un dominio. Se utiliza como parte del proceso para buscar nombres.
Registro CNAME Permite la creación de alias. Por ejemplo, una persona familiarizada con los nombres de Internet en general, que desee enviar un mensaje al usuario Paul en el Departamento de Ciencias Computacionales del MIT, podría suponer que paul@cs.mit.edu funcionará. No obstante, esta dirección no funcionará, puesto que el dominio del Departamento de Ciencias Computacionales del MIT es csail.mit.edu. Sin embargo, como servicio para la gente que no sabe esto, el MIT podría crear una entrada CNAME para encaminar a la gente y a los programas en la dirección correcta. La entrada podría ser como la siguiente: cs.mit.edu 86400 IN CNAME csail.mit.edu
Registro PTR Apunta a otro nombre y se usa para asociar un nombre a una dirección IP con el fin de permitir búsquedas de la dirección IP y devolver el nombre de la máquina correspondiente. Se utiliza para búsquedas inversas.
Registro SRV Identifica a un host para un servicio dado en un dominio. Generaliza el registro MX que realiza la misma tarea, pero que es sólo para servidores de correo.
Registro SPF Permite a un dominio codificar información sobre qué máquinas en el dominio enviarán correo al resto de Internet. Ayuda a las máquinas receptoras a verificar que el correo sea válido.
Registro TXT Originalmente permitía que los dominios se identificaran a sí mismos en formas arbitrarias, pero en la actualidad usualmente codifica información legible por máquina, por lo general la información SPF.
Campo Valor Puede ser un número, un nombre de dominio o una cadena ASCII. La semántica depende del tipo de registro. La semántica depende del tipo de registro. En la figura 3 se proporciona una descripción corta de los campos Valor para cada uno de los tipos principales de registro.

Servidores de nombres


Definición:

Los servidores de nombres son componentes fundamentales del sistema DNS. Son dispositivos o software que almacenan y gestionan bases de datos de nombres de dominio y sus correspondientes direcciones IP.


En teoría, un solo servidor de nombres podría contener toda la base de datos DNS y responder a todas las consultas relacionadas con ella. En la práctica, este servidor estaría tan sobrecargado que sería inservible. Además, si llegara a fallar, Internet completa quedaría inutilizable.

División del espacio de nombres DNS

Para evitar los problemas asociados al hecho de tener una sola fuente de información, el espacio de nombres DNS se divide en zonas que no se sobreponen. La figura 5 muestra una manera posible de dividir el espacio de nombres de la figura 1. Cada zona circulada contiene una parte del árbol.

El lugar donde se colocan los límites dentro de una zona es responsabilidad del administrador de esa zona. Esta decisión se toma en gran parte con base en la cantidad de servidores de nombres deseados y su ubicación. 

Por ejemplo

En la figura 5 la Universidad de Washington tiene una zona para washington.edu que maneja eng.washington.edu, pero no maneja cs.washington.edu. Ésta es una zona separada con sus propios servidores de nombres. Tal decisión podría tomarse cuando un Departamento, como el de Inglés, no desee operar su propio servidor de nombres, pero un Departamento como el de Ciencias Computacionales sí.


Figura 5: Parte del espacio de nombres DNS dividido en zonas (las cuales están circuladas).


Cada zona se asocia también con uno o más servidores de nombres. Éstos son hosts que contienen la base de datos de la zona. Por lo general, una zona debe tener un servidor de nombres primario, el cual obtiene su información de un archivo en su disco, y uno o más servidores secundarios, los cuales obtienen su información del servidor de nombres primario. 


Para mejorar la confiabilidad, algunos de los servidores de nombres se pueden ubicar fuera de la zona. Al proceso de buscar un nombre y encontrar una dirección se le conoce como resolución de nombres. 

Cuando un resolvedor tiene una consulta referente a un nombre de dominio, la pasa a uno de los servidores de nombres locales. Si el dominio que se busca está bajo la jurisdicción del servidor de nombres, como top.cs.vu-nl que está bajo cs.vu.nl, devuelve los registros de recursos autorizados. 

Un registro autorizado es uno que proviene de la autoridad que administra a ese registro y, por ende, siempre es correcto. Los registros autorizados contrastan con los registros en caché, que tal vez no estén actualizados.


Búsqueda de un nombre remoto: un caso


¿Qué pasa cuando el dominio es remoto, como por ejemplo, cuando flits.cs.vu.nl desea encontrar la dirección IP de robot.cs.washington.edu en la UW (Universidad de Washington)? 

En este caso y si no hay información en la caché sobre el dominio disponible en forma local, el servidor de nombres empieza una consulta remota. Esta consulta sigue el proceso que se muestra en la figura 6. 


El paso 1 muestra la consulta que se envía al servidor de nombres local. Esta consulta contiene el nombre de dominio buscado, el tipo (A) y la clase (IN).

El siguiente paso es empezar en la parte superior de la jerarquía de nombres y preguntar a cada uno de los servidores raíz de nombres. Estos servidores tienen información sobre cada dominio de nivel superior. Esto se muestra como el paso 2 en la figura 6. Para contactarse con un servidor raíz, cada servidor de nombres debe tener información sobre uno o más servidores raíz de nombres. Esta información por lo general está presente en un archivo de configuración del sistema que se carga en la caché DNS al momento de iniciar el servidor DNS. En esencia, es una lista de registros NS para la raíz y los registros A correspondientes.

Hay 13 servidores raíz DNS, que se llaman a-root-servers.net hasta m.root-ser- vers.net. Cada servidor raíz podría ser lógicamente una sola computadora. Sin embargo, como toda Internet depende de los servidores raíz, éstos son computadoras poderosas con un alto grado de replicación.
La mayoría de los servidores están presentes en varias ubicaciones geográficas y se puede acceder a ellos mediante el enrutamiento anycast, en donde un paquete se entrega a la instancia más cercana de una dirección de destino. Es decir, la misma dirección IP se asigna a varios dispositivos ubicados en diferentes ubicaciones geográficas. La replicación mejora la confiabilidad y el desempeño.
Es improbable que el servidor raíz de nombres conozca la dirección de una máquina en la UW, y tal vez tampoco conozca el servidor de nombres para la UW. Pero debe conocer el servidor de nombres del dominio edu, en donde se encuentra cs.washington.edu. 

En el paso 3 devuelve el nombre y la dirección IP para esa parte de la respuesta.

Después, el servidor de nombres local continúa su búsqueda y envía la consulta completa al servidor de nombres edu (a.edu-servers.net). 

Ese servidor de nombres devuelve el servidor de nombres para la UW. Esto se muestra en los pasos 4 y 5.

Si el nombre de dominio buscado estaba en el Departamento de Inglés se hubiera encontrado la respuesta, puesto que la zona de la UW incluye a dicho Departamento. Pero el Departamento de Ciencias Computacionales decidió implementar su propio servidor de nombres. Los dominios pueden insertarse en el árbol de nombres de dominio en dominios genéricos o de países. Por ejemplo, cs.washington.edu también podría estar listado bajo el dominio de país us como cs.washington.wa.us, lo que muestra la flexibilidad y la estructura jerárquica del sistema de nombres de dominio.

El servidor de nombres local reenvía como respuesta a flits.cs.vu.nl (paso 10). Así termina la resolución del nombre.



Explorar proceso con herramientas estándar


Podemos explorar este proceso mediante el uso de herramientas estándar, como el programa dig que está instalado en la mayoría de los sistemas UNIX.

Por ejemplo

Al escribir: dig @a.edu-servers-net robot.cs.washington.edu, se enviará una consulta sobre robot.cs.washington.edu al servidor de nombres a.edu-servers.net y se imprimirá el resultado en pantalla. 


Esto nos mostrará la información obtenida en el paso 4 del ejemplo anterior, y conoceremos el nombre y la dirección IP de los servidores de nombres de la UW.

Hay tres puntos técnicos a discutir sobre este gran escenario:

Punto Técnico Descripción
Mecanismos de Consulta

En el escenario, se utilizan dos mecanismos de consulta distintos: el recursivo y el iterativo.

  • La consulta recursiva implica que el servidor de nombres local maneje la resolución en nombre del host hasta que pueda devolver la respuesta deseada sin retornar respuestas parciales. Ejemplo: Cuando un host envía una consulta al servidor de nombres local, este maneja la resolución internamente y devuelve la respuesta final sin retornar respuestas parciales.
  • Por otro lado, la consulta iterativa implica que el servidor raíz de nombres (y cada servidor subsiguiente) no continúe recursivamente la consulta hacia el servidor de nombres local, sino que devuelva una respuesta parcial y avance a la siguiente consulta, dejando al servidor de nombres local la responsabilidad de continuar la resolución mediante consultas adicionales.
Caché

Las respuestas, incluyendo las devueltas, se colocan en la caché para su futura referencia. Esto agiliza el proceso de resolución, ya que si otro host en el mismo dominio realiza una consulta similar, la respuesta ya estará disponible. Sin embargo, las respuestas en caché no son autoritativas y deben ser actualizadas regularmente, ya que los cambios en los dominios pueden no propagarse a todas las cachés del mundo. El tiempo de vida de una entrada en la caché se especifica mediante el campo "Tiempo_de_vida" en cada registro de recursos. 

Ejemplo: Cuando un host genera una consulta sobre un host en el mismo dominio que ya ha sido resuelto y se encuentra en la caché, la respuesta se obtiene directamente de la caché, evitando la necesidad de realizar una consulta adicional al servidor de nombres.
Protocolo de Transporte

DNS utiliza el protocolo de transporte UDP para sus consultas y respuestas. Los mensajes DNS se envían en paquetes UDP, con un formato simple para las consultas, respuestas y servidores de nombres. Cada consulta contiene un identificador de 16 bits que se copia en la respuesta correspondiente, permitiendo al servidor de nombres relacionar las respuestas con las consultas. Este protocolo maneja situaciones donde el servidor está fallando o se pierden paquetes de consulta o respuesta. 

Ejemplo: Si no se recibe una respuesta dentro de un intervalo corto, el cliente DNS repite la consulta e intenta con otro servidor para ese dominio después de unos cuantos reintentos.
