En esta sección vamos a introducirnos en el mundo del testing en React. Cubriremos los conceptos fundamentales del testing, sus diferentes tipos y los principios que guían el proceso de testing en aplicaciones React. Comenzaremos analizando los diferentes tipos de testing que se aplican a las aplicaciones React, incluyendo el testing unitario, el testing de integración y el testing E2E (End-to-End). A continuación, nos centraremos en la configuración del entorno de testing. 

Aquí, aprenderemos a preparar nuestras aplicaciones React para poder ejecutar pruebas de manera efectiva y eficiente. A lo largo de la semana, también exploraremos herramientas y bibliotecas populares para el testing de aplicaciones React. Estudiaremos Jest, un marco de pruebas para JavaScript ampliamente utilizado, y React Testing Library, que nos permite probar componentes React de una manera fácil y accesible. 

Profundizaremos en el testing unitario de componentes React y en cómo utilizar Jest y React Testing Library para llevar a cabo pruebas efectivas y significativas en nuestros componentes. También abordaremos Cypress, una herramienta integral para pruebas E2E que nos permitirá evaluar el funcionamiento completo de nuestra aplicación desde la perspectiva del usuario final. 

Finalmente, cerraremos la sección de aprendizaje con una introducción al TDD (Test-Driven Development) en aplicaciones React.

Introducción al Testing en React



En el caso de las aplicaciones React, el testing nos permite asegurarnos de que los componentes funcionan correctamente bajo diferentes condiciones y que interactúan adecuadamente con otros elementos de la aplicación.

Definición: Testing

Es el proceso mediante el cual verificamos y validamos el correcto funcionamiento de una aplicación o componente de software.


El testing es fundamental por varias razones:

Ayuda a detectar y prevenir errores antes de que lleguen a producción.

Facilita el mantenimiento y la evolución del código.

Aumenta la confianza en la calidad y estabilidad de la aplicación.

Mejora la colaboración entre los desarrolladores, ya que facilita el entendimiento y la verificación del código.


Tipos de testing


Existen diferentes tipos de testing que se aplican en distintos niveles de la aplicación.

Vamos a describir brevemente los tres tipos principales:

Testing unitario

En React, el testing unitario consiste en verificar el comportamiento y la lógica de un único componente, asegurándonos de que funcione correctamente de manera aislada.

Definición: Testing unitario

Este tipo de testing se enfoca en probar pequeñas unidades de código, como funciones o componentes individuales.

Para realizar el testing unitario en React, se utilizan herramientas como Jest y React Testing Library, que permiten escribir pruebas automatizadas que simulan diferentes escenarios para el componente, como datos de entrada diferentes, interacciones de usuario, etc.

Al escribir las pruebas, se define un conjunto de expectativas o afirmaciones que representan el comportamiento esperado del componente.

Estas afirmaciones se comparan con el resultado real del componente al ejecutar las pruebas, y se emiten informes de errores o fallas en caso de que el resultado no coincida con las expectativas definidas.


El testing unitario también ayuda a documentar el comportamiento de los componentes y su interacción con otros elementos de la aplicación. Al escribir pruebas unitarias para cada componente, se proporciona una descripción detallada de su comportamiento y su funcionalidad.

Testing de integración

En el contexto de React, el testing de integración implica probar la interacción entre varios componentes y cómo se comunican mediante propiedades, contextos o servicios externos.

Definición: Testing de integración

Es una técnica de prueba de software que se enfoca en evaluar cómo diferentes componentes o módulos de una aplicación trabajan juntos, asegurándose de que la interacción entre ellos sea correcta.

El objetivo principal del testing de integración es garantizar que los diferentes componentes de la aplicación se integren adecuadamente y que la funcionalidad de la aplicación en su conjunto sea coherente y completa.

Para realizar el testing de integración en React, se utilizan herramientas como Jest, React Testing Library y Enzyme, que permiten escribir pruebas automatizadas que simulan diferentes escenarios para los componentes y cómo se relacionan entre sí.

Al escribir las pruebas de integración, se simula el comportamiento de la aplicación en un ambiente controlado, utilizando datos y escenarios que representan situaciones reales.


Se pueden probar diferentes combinaciones de componentes, propiedades y estados, para asegurarse de que la interacción entre ellos funcione correctamente.

Además, el testing de integración también implica probar cómo los componentes interactúan con servicios externos, como APIs o bases de datos, y cómo manejan errores o situaciones imprevistas. Esto ayuda a garantizar que la aplicación tenga un comportamiento estable y predecible en diferentes entornos.

Testing E2E (End-to-End)

En el contexto de una aplicación React, el testing E2E implica automatizar acciones del usuario, como hacer clic en botones, completar formularios y navegar entre páginas, para asegurarse de que la aplicación funcione como se espera en un entorno real.

Definición: Testing E2E (End-to-End)

Es una técnica de prueba de software que simula el flujo completo de una aplicación desde la perspectiva del usuario final.

Para realizar el testing E2E en React, se utilizan herramientas como Cypress o Selenium, que permiten automatizar interacciones de usuario y realizar pruebas en diferentes navegadores y sistemas operativos.

El objetivo principal del testing E2E es garantizar que la aplicación funcione correctamente en su conjunto, desde la perspectiva del usuario final. Esto incluye verificar la funcionalidad, la interacción entre componentes, el flujo de la aplicación y la respuesta a situaciones imprevistas.


Al escribir pruebas E2E, se definen una serie de pasos que simulan el comportamiento del usuario, desde la carga inicial de la aplicación hasta la realización de acciones específicas. Por ejemplo, se puede simular que un usuario ingresa a la aplicación, navega a una página específica, completa un formulario y envía los datos.

Durante la ejecución de las pruebas E2E, se verifican las respuestas de la aplicación a cada acción y se emiten informes de errores o fallas en caso de que algo no funcione como se espera.

Además, se pueden simular situaciones imprevistas, como una conexión a Internet lenta o una interrupción en la comunicación con un servicio externo, para asegurarse de que la aplicación maneje estos escenarios de manera adecuada.


Principios de testing en React


Ahora que conocemos los conceptos básicos del testing y sus diferentes tipos, vamos a repasar algunos de los principios que debemos tener en cuenta al realizar pruebas en aplicaciones React:

Mantener las pruebas legibles y simples: Las pruebas deben ser fáciles de leer y entender para que otros desarrolladores puedan comprender rápidamente qué se está probando y por qué. Es importante utilizar nombres descriptivos para las pruebas y estructurarlas de manera clara y concisa.

Enfocarse en el comportamiento, no en la implementación: Al escribir pruebas, es importante centrarse en cómo se espera que funcione un componente, en lugar de cómo está implementado internamente. Esto permite que las pruebas sean más resistentes a los cambios en el código y facilita la refactorización.

Utilizar mocks y stubs: Al realizar pruebas unitarias e incluso de integración, es común utilizar mocks y stubs para simular el comportamiento de servicios externos.

Priorizar la cobertura de casos críticos y comunes: Aunque sería ideal tener una cobertura del 100% en nuestras pruebas, en la práctica, esto puede ser difícil de lograr. Por lo tanto, es importante priorizar y centrarse en los casos de uso más críticos y comunes de nuestra aplicación.

Mantener un equilibrio entre los diferentes tipos de pruebas: Es importante tener una combinación equilibrada de pruebas unitarias, de integración y E2E en nuestro proyecto. Las pruebas unitarias son rápidas de ejecutar y nos ayudan a detectar errores en unidades de código individuales. Las pruebas de integración nos permiten verificar cómo se combinan las unidades de código, mientras que las pruebas E2E nos brindan una visión más amplia del funcionamiento de la aplicación en su totalidad.

Jest


Jest es fácil de configurar, tiene una gran cantidad de características útiles y se integra perfectamente con otras librerías de testing como React Testing Library y Enzym.

Definición: Jest

Es un marco de testing desarrollado por Facebook que nos permite escribir pruebas para nuestras aplicaciones JavaScript, incluidas las aplicaciones React.


Algunas características clave de Jest incluyen:

Ejecución rápida de pruebas con soporte para pruebas en paralelo.

Generación automática de "snapshots" para verificar el resultado de las pruebas.

Soporte integrado para pruebas asíncronas.

Mocking sencillo y potente para funciones y módulos.


Configuración de Jest en un proyecto React


Si estamos utilizando create-react-app para iniciar nuestro proyecto, Jest ya estará configurado y listo para usar. Si no es así, podemos seguir los siguientes pasos para configurar Jest:

1

Instalar Jest y las dependencias necesarias:

2

Crear un archivo de configuración de Babel en la raíz del proyecto llamado .babelrc con el siguiente contenido:

        "@babel/preset-react"

3

Agregar un script en el archivo package.json para ejecutar Jest:

Escribiendo pruebas


Para escribir pruebas con Jest, creamos archivos con la extensión .test.js o .spec.js. Jest reconocerá automáticamente estos archivos y los ejecutará como pruebas.

Describe() y it()

En Jest, las pruebas se definen mediante la función describe() y la función it(). La función describe() se utiliza para agrupar pruebas relacionadas y la función it() se utiliza para definir cada prueba individual. En conjunto, estas dos funciones proporcionan una estructura clara y organizada para escribir pruebas.

Ejemplo

Cómo utilizar describe() y it() para escribir pruebas básicas para una función simple de suma:

JavaScript Copiar código 
 
                            
describe("suma()", () => {

  it("debe sumar dos números positivos", () => {

    expect(suma(2, 3)).toBe(5);

  });

  it("debe sumar dos números negativos", () => {

    expect(suma(-2, -3)).toBe(-5);

  });

  it("debe devolver cero cuando se suman cero y cualquier otro número", () => {

    expect(suma(0, 5)).toBe(0);

    expect(suma(5, 0)).toBe(0);

  });

});

Se utiliza describe() para agrupar las pruebas relacionadas con la función suma(). Dentro de describe(), se definen tres pruebas individuales utilizando it(). Cada prueba define una entrada diferente para la función suma() y se espera un resultado específico.

Se utiliza expect() para definir una expectativa, es decir, el resultado esperado de la prueba. En este ejemplo, se utiliza el matcher toBe() para comparar el resultado real de la función suma() con el valor esperado.

Es importante tener en cuenta que Jest utiliza el patrón de diseño "Arrange, Act, Assert" para escribir pruebas.


Esto significa que cada prueba debe seguir los siguientes pasos:

Arrange: Preparar los datos necesarios para la prueba.

Act: Ejecutar la función o el código que se está probando.

Assert: Verificar el resultado de la prueba.

Veamos cómo se aplicaría el patrón "Arrange, Act, Assert" a la prueba debe sumar dos números positivos:

Arrange: Definimos dos números positivos (2 y 3).

Act: Ejecutamos la función suma(2, 3).

Assert: Verificamos que el resultado de la función sea igual a 5.


Verificar resultados


En Jest, las expectations son funciones que se utilizan para verificar el resultado de una prueba. Estas expectations se encargan de comparar el resultado real de una función con un valor esperado, utilizando diferentes matchers (igualdad, desigualdad, tipo de dato, etc.).

Si queremos verificar que una función suma() devuelve el resultado correcto, podemos utilizar la expectativa toBe(). La expectativa toBe() verifica si dos valores son iguales utilizando el operador de igualdad.

Ejemplo

Cómo utilizar la expectativa toBe() para verificar el resultado de una función suma():

JavaScript Copiar código 
 
                            
it("debe sumar dos números y devolver el resultado", () => {

  expect(suma(2, 3)).toBe(5);

});

En este ejemplo:

Se espera que la función suma(2, 3) devuelva el valor 5.

Para verificar esto, utilizamos la expectation toBe(5).

Si el resultado real de la función es igual a 5, la prueba se considera exitosa.

De lo contrario, Jest emitirá un informe de error.

Jest proporciona una variedad de matchers que se pueden utilizar para comparar valores durante las pruebas. Algunos de los matchers más comunes son toBe(), toEqual(), toContain(), toMatch(), toHaveLength(), toBeNull(), toBeTruthy() y toBeFalsy().

ToBe(): Verifica si dos valores son iguales utilizando el operador de igualdad estricta (===). Por ejemplo: expect(resultado).toBe(5);.

ToEqual(): Verifica si dos valores son iguales utilizando una comparación profunda (deep equality). Esto significa que, si los valores son objetos o arrays, Jest verifica si los valores de todas las propiedades son iguales. Por ejemplo: expect(resultado).toEqual(5);.

ToContain(): Verifica si un valor está contenido en un array o en una cadena. Por ejemplo: expect(['manzana', 'naranja']).toContain('manzana');.

ToMatch(): Verifica si una cadena coincide con una expresión regular. Por ejemplo: expect('Mi correo es email@example.com').toMatch(/email@example\.com/);.

ToHaveLength(): Verifica si un objeto tiene una propiedad length con un valor específico. Por ejemplo: expect('hello').toHaveLength(5);.

ToBeNull(): Verifica si un valor es nulo. Por ejemplo: expect(resultado).toBeNull();.

ToBeTruthy(): Verifica si un valor es verdadero. Por ejemplo: expect(resultado).toBeTruthy();.

ToBeFalsy(): Verifica si un valor es falso. Por ejemplo: expect(resultado).toBeFalsy();.


Además de estos matchers, Jest también proporciona otros matchers para verificar valores más específicos. Por ejemplo, toHaveProperty() se utiliza para verificar si un objeto tiene una propiedad específica, y toThrow() se utiliza para verificar si una función lanza una excepción.

Es importante tener en cuenta que cada expectation debe estar encadenada después de la función expect(). Además, las expectations pueden ser encadenadas para verificar múltiples condiciones. Por ejemplo:

JavaScript Copiar código 
                            

it("debe devolver un objeto con las propiedades correctas", () => {

  const resultado = obtenerUsuario(123);

  expect(resultado).not.toBeNull();

  expect(resultado).toHaveProperty("nombre", "Juan");

  expect(resultado).toHaveProperty("edad", 30);

});

En este ejemplo:

Se espera que la función obtenerUsuario(123) devuelva un objeto con las propiedades nombre y edad definidas.

Utilizamos las expectativas not.toBeNull(), toHaveProperty() y toHaveProperty() para verificar estas condiciones.

Ejemplo

Cómo escribir una prueba básica con Jest para un componente React:

JavaScript Copiar código 
 
                            
import React from "react";

import { render } from "@testing-library/react";

import Greeting from "./Greeting";



test("debe mostrar un saludo", () => {

  const { getByText } = render(<Greeting />);

  const greetingElement = getByText(/hola, mundo/i);

  expect(greetingElement).toBeInTheDocument();

});

En este ejemplo, utilizamos la función test de Jest para definir una prueba y la función expect para realizar una aserción sobre el resultado de la prueba. También utilizamos React Testing Library para renderizar el componente y buscar elementos en el DOM.


Ejecución y seguimiento


Una vez que hemos escrito nuestras pruebas utilizando Jest, es importante saber cómo ejecutarlas y realizar un seguimiento de su progreso. Jest ofrece varias herramientas y opciones para facilitar la ejecución y el monitoreo de nuestras pruebas.

Ejecución

Para ejecutar todas las pruebas en nuestro proyecto, simplemente podemos utilizar el comando npm test (o yarn test si utilizamos Yarn). Esto ejecutará todas las pruebas en nuestros archivos de prueba y nos mostrará un resumen de los resultados.

Si queremos ejecutar un conjunto específico de pruebas, podemos usar el comando npm test -- seguido del nombre del archivo o una expresión regular que coincida con los nombres de los archivos de prueba que queremos ejecutar.

Ejemplo

Si queremos ejecutar solo las pruebas en los archivos que contienen la palabra "component", podríamos usar el comando:
npm test -- component.

Jest también ofrece una herramienta llamada "watch mode", que monitorea automáticamente los cambios en nuestro código y ejecuta las pruebas relevantes cada vez que guardamos un archivo.

Para habilitar el watch mode, podemos utilizar el comando: 
npm test -- --watch (o yarn test --watch).


Uso de "mocks" y "spies"


En algunas situaciones, puede ser útil simular o "mockear" ciertos comportamientos o funciones en nuestras pruebas. 

Jest ofrece un conjunto de herramientas para crear "mocks" y "spies" que nos permiten controlar y monitorear el comportamiento de nuestras funciones y componentes.

Definición: Mock

Es una función falsa que podemos utilizar en lugar de la función real en nuestras pruebas. Esto nos permite aislar y probar específicamente la lógica de nuestros componentes, sin preocuparnos por las dependencias externas.

Ejemplo

Podríamos querer "mockear" una función que realiza una llamada a una API para que no se realice la llamada real en nuestras pruebas.

Definición: Spy

Es una función que nos permite rastrear cuántas veces se llamó a una función y con qué argumentos. Esto puede ser útil para asegurarnos de que nuestras funciones se llamen con los parámetros correctos y en el orden correcto.

Ejemplo

Cómo usar "mocks" y "spies" en nuestras pruebas con Jest:

JavaScript Copiar código 
 
                            
import { fetchData } from "./api";

import { processData } from "./utils";



// Mockear la función fetchData

jest.mock("./api", () => ({

    fetchData: jest.fn()

}));



// Spy en la función processData

jest.spyOn(utils, "processData");



test("debe llamar a fetchData y processData correctamente", async () => {

    // Configurar el mock para que devuelva datos de prueba

    fetchData.mockResolvedValue({ data: [1, 2, 3] });

    // Llamar a nuestra función que utiliza fetchData y processData

    const result = await myFunction();

    // Verificar que fetchData se llamó una vez

    expect(fetchData).toHaveBeenCalledTimes(1);

    // Verificar que processData se llamó con los datos devueltos por fetchData

    expect(processData).toHaveBeenCalledWith({ data: [1, 2, 3] });

    // Verificar el resultado de nuestra función

    expect(result).toEqual("Resultado esperado");

});

En este ejemplo:

Estamos probando una función llamada myFunction que utiliza dos funciones externas, fetchData y processData.

Creamos un "mock" para la función fetchData y configuramos el valor devuelto.

Creamos un "spy" en la función processData.

Después de llamar a myFunction verificamos que ambas funciones se hayan llamado correctamente y que myFunction haya devuelto el resultado esperado.


Cobertura de código


Jest incluye una herramienta de cobertura de código llamada Istanbul que nos permite analizar qué partes de nuestro código han sido probadas y cuáles no.

Definición: Cobertura de código

Es una medida de la cantidad de código que nuestras pruebas están verificando.

Para habilitar la cobertura de código en Jest, podemos agregar la opción --coverage a nuestro script de prueba en el archivo package.json:

JSON Copiar código 
 
                            
{

    "scripts": {

        "test": "jest --coverage"

    }

}

Al ejecutar nuestras pruebas con la opción --coverage, Jest generará un informe de cobertura en la carpeta coverage. Podemos abrir el archivo coverage/lcov-report/index.html en nuestro navegador para ver un informe detallado de la cobertura de código en nuestra aplicación.

Importante

Cobertura de código

Aunque una alta cobertura de código puede ser un indicador útil de la calidad de nuestras pruebas, no garantiza que nuestras pruebas sean exhaustivas ni que nuestra aplicación esté libre de errores. La cobertura de código es solo una herramienta que nos puede ayudar a identificar áreas de nuestro código que podrían beneficiarse de más pruebas.


React Testing Library


Con React Testing Library, podemos renderizar componentes, interactuar con ellos y verificar si cumplen con nuestras expectativas.

Definición: React Testing Library

Es una librería que nos facilita la tarea de probar componentes React. Se centra en el comportamiento y la accesibilidad de los componentes en lugar de su estructura interna.


Instalación y configuración

Para comenzar a utilizar React Testing Library, primero necesitamos instalarla como una dependencia de desarrollo en nuestro proyecto:

CLI Copiar código 
 
                            
npm install --save-dev @testing-library/react

O si usamos Yarn:

CLI Copiar código 
 
                            
yarn add --dev @testing-library/react

Escribiendo pruebas


React Testing Library proporciona una serie de funciones útiles para renderizar y manipular componentes React en nuestras pruebas. A continuación, se muestra un ejemplo simple de cómo escribir una prueba para un componente que muestra un mensaje de bienvenida:

JavaScript Copiar código 
 
                            
import React from "react";

import { render, screen } from "@testing-library/react";

En este ejemplo:

Luego, utilizamos la función expect de Jest para verificar que el elemento esté presente en el DOM.


Consultas y eventos


React Testing Library ofrece varios tipos de consultas para buscar elementos en nuestro componente.

Algunas consultas comunes incluyen getByLabelText, getByPlaceholderText, getByRole, getByText, getByTitle y getByTestId. Además de las consultas getBy, también existen las consultas queryBy y findBy, que tienen un comportamiento ligeramente diferente.

Para simular eventos de usuario, como clics y entradas de teclado, podemos utilizar la función fireEvent de React Testing Library.

Ejemplo

Cómo simular un clic en un botón

JavaScript Copiar código 
 
                            
import React from "react";

import { render, fireEvent, screen } from "@testing-library/react";

import Counter from "./Counter";



test("debe incrementar el contador al hacer clic en el botón", () => {

    render(<Counter />);

    const buttonElement = screen.getByRole("button", { name: /incrementar/i });

    fireEvent.click(buttonElement);

    const counterElement = screen.getByText(/Contador: 1/i);

    expect(counterElement).toBeInTheDocument();

});

En este ejemplo:

Buscamos el botón utilizando la consulta getByRole y luego simulamos un clic en el botón utilizando la función fireEvent.click.

Después de simular el clic, buscamos un elemento que contenga el texto "Contador: 1" y verificamos que esté presente en el DOM.



Asíncrono y efectos secundarios


A menudo, nuestros componentes pueden tener comportamientos asíncronos o efectos secundarios, como llamadas a API o cambios en el estado global. React Testing Library proporciona herramientas para manejar estas situaciones y garantizar que nuestras pruebas sean sólidas y efectivas.

Para esperar que un elemento aparezca en el DOM, podemos utilizar la función waitFor de React Testing Library. waitFor acepta una función y espera hasta que dicha función deje de lanzar errores.

Ejemplo

Cómo utilizar waitFor para esperar a que se muestre un mensaje después de hacer clic en un botón.

JavaScript Copiar código 
 
                            
import React from "react";

import { render, fireEvent, screen, waitFor } from "@testing-library/react";

import AsyncComponent from "./AsyncComponent";



test("debe mostrar un mensaje después de hacer clic en el botón", async () => {

    render(<AsyncComponent />);

    const buttonElement = screen.getByRole("button", {

        name: /mostrar mensaje/i

    });

    fireEvent.click(buttonElement);

    await waitFor(() => screen.getByText(/Mensaje cargado/i));

    const messageElement = screen.getByText(/Mensaje cargado/i);

    expect(messageElement).toBeInTheDocument();

});

En este ejemplo, simulamos un clic en el botón y luego utilizamos waitFor para esperar a que aparezca el elemento con el texto "Mensaje cargado". La función waitFor devuelve una promesa, por lo que podemos utilizar la palabra clave await para esperar a que se resuelva antes de continuar con nuestra prueba.

Cuando trabajamos con componentes que realizan llamadas a API o tienen otros efectos secundarios, es importante "mockear" estos comportamientos para que nuestras pruebas sean más rápidas y predecibles.


Podemos utilizar las funciones de "mock" proporcionadas por Jest junto con React Testing Library para aislar y probar nuestros componentes.

Siguiendo estos conceptos y utilizando las herramientas que ofrece React Testing Library, podremos escribir pruebas efectivas y mantenibles para nuestros componentes React. Estas pruebas nos ayudarán a garantizar que nuestros componentes se comporten correctamente y proporcionen una experiencia de usuario de alta calidad.


Ejemplo práctico


Imaginemos que estamos desarrollando un componente sencillo en React, un botón que incrementa un contador. Nuestro objetivo es escribir pruebas para verificar su correcto funcionamiento.

Ejemplo

Cómo podríamos estructurar nuestras pruebas

JavaScript Copiar código 
 
                            
import React from "react";

import { render, fireEvent } from "@testing-library/react";

import CounterButton from "./CounterButton";



describe("CounterButton", () => {

    test("debe mostrar el contador inicialmente en cero", () => {

        expect(getByTestId("counter")).toHaveTextContent("1");

        fireEvent.click(button);

        expect(getByTestId("counter")).toHaveTextContent("2");

    });

});

En este ejemplo:

Utilizamos la librería React Testing Library para renderizar nuestro componente CounterButton y realizar pruebas sobre él.

Verificamos que el contador se muestre inicialmente en cero.

Probamos que el contador se incremente correctamente al hacer clic en el botón.


Testing Unitario de Componentes React


El testing unitario es una metodología de pruebas que se centra en probar pequeñas unidades de código de forma aislada.

En el caso de las aplicaciones React, esto implica probar componentes individuales y sus funciones relacionadas. El objetivo del testing unitario es garantizar que cada componente funcione correctamente de manera independiente, lo que facilita la identificación y solución de problemas en el código.

Es importante aplicar el testing unitario cuando estamos desarrollando aplicaciones complejas en las que necesitamos estar seguros de que todo funciona correctamente.

También es importante para detectar errores en etapas tempranas del desarrollo, antes de que se conviertan en problemas más grandes.

A continuación aprenderemos a escribir pruebas unitarias para componentes React utilizando Jest y React Testing Library.


Configuración: Entorno de pruebas


Antes de comenzar a escribir pruebas unitarias para nuestros componentes React, debemos asegurarnos de que nuestro entorno de pruebas esté configurado correctamente. Para ello, necesitamos instalar Jest y React Testing Library como dependencias de desarrollo:

CLI Copiar código 
 
                            
npm install --save-dev jest @testing-library/react @testing-library/jest-dom

Luego, agregamos un script de prueba a nuestro archivo package.json:

JSON Copiar código 
 
                            
{

    "scripts": {

        "test": "jest"

    }

}

Pruebas unitarias


Para escribir pruebas unitarias para nuestros componentes React, utilizamos Jest junto con React Testing Library. Jest nos proporciona un marco de testing, mientras que React Testing Library nos permite interactuar con nuestros componentes de manera similar a como lo haría un usuario real.

Ejemplo

Un componente de contador con un botón para incrementar el contador.

JavaScript Copiar código 
 
                            
import React, { useState } from "react";



const Counter = () => {

    const [count, setCount] = useState(0);

    return (

        <div>

            <p>Contador: {count}</p>

            <button onClick={() => setCount(count + 1)}>Incrementar</button>

        </div>

    );

};



export default Counter;

Los componentes funcionales son aquellos que no mantienen su propio estado y dependen únicamente de las propiedades que reciben de sus componentes padres. Dado que estos componentes son simples y carecen de estado, las pruebas unitarias para ellos son más sencillas.

Podemos escribir una prueba unitaria para este componente utilizando Jest y React Testing Library de la siguiente manera:

JavaScript Copiar código 
 
                            
import React from "react";

import { render, fireEvent, screen } from "@testing-library/react";

import "@testing-library/jest-dom/extend-expect";

import Counter from "./Counter";



test("debe incrementar el contador al hacer clic en el botón", () => {

    render(<Counter />);

    const buttonElement = screen.getByRole("button", { name: /incrementar/i });

    fireEvent.click(buttonElement);

    const counterElement = screen.getByText(/Contador: 1/i);

    expect(counterElement).toBeInTheDocument();

});

En esta prueba:

Renderizamos el componente Counter.

Buscamos el botón utilizando la función getByRole de React Testing Library.

Simulamos un clic en el botón utilizando la función fireEvent.click.

Verificamos que el contador se haya incrementado correctamente, utilizando la función getByText para obtener el elemento que contiene el texto Contador: 1.


Propiedades y estados


React Testing Library nos proporciona funciones específicas para acceder a los elementos del DOM que representan propiedades y estados de un componente. De esta manera, podemos realizar pruebas más específicas y exhaustivas.

Ejemplo

Para probar propiedades de un componente, podemos utilizar la función getByProp o queryByProp.

JavaScript Copiar código 
 
                            
import React from "react";

import { render, screen } from "@testing-library/react";

import MyComponent from "./MyComponent";



test("renders component with correct props", () => {

    const props = { name: "John" };

    render(<MyComponent {...props} />);

    const nameProp = screen.getByProp("name");

    expect(nameProp).toHaveTextContent("John");

});

En este ejemplo, estamos utilizando la función getByProp para obtener la propiedad name del componente MyComponent y comprobando que tenga el valor 'John' utilizando la función toHaveTextContent.

Ejemplo

Para probar estados de un componente, podemos utilizar la función findByState o queryByState.

JavaScript Copiar código 
 
                            
import React from "react";

import { render, screen } from "@testing-library/react";

import MyComponent from "./MyComponent";



test("changes state when button is clicked", async () => {

    render(<MyComponent />);

    const button = screen.getByRole("button");

    const stateBeforeClick = await screen.findByState({ count: 0 });

    expect(stateBeforeClick).toBeInTheDocument();

    fireEvent.click(button);

    const stateAfterClick = await screen.findByState({ count: 1 });

    expect(stateAfterClick).toBeInTheDocument();

});

En este ejemplo:

Estamos utilizando la función findByState para obtener el estado del componente MyComponent antes y después de hacer clic en un botón.

Utilizamos la función fireEvent.click para simular un clic en el botón y finalmente comprobamos que el estado después del clic sea el que esperamos.


Simulación de eventos y acciones del usuario


En el ejemplo anterior, hemos utilizado la función fireEvent.click para simular un clic en el botón de nuestro componente. 

De manera similar, podemos utilizar otras funciones de React Testing Library para simular otros eventos y acciones del usuario, como por ejemplo fireEvent.change para simular cambios en un input.

Ejemplo

JavaScript Copiar código 
 
                            
import React from "react";

import { render, screen, fireEvent } from "@testing-library/react";

import MyComponent from "./MyComponent";



test("handles input change", () => {

    render(<MyComponent />);

    const input = screen.getByLabelText("Name");

    fireEvent.change(input, { target: { value: "John Doe" } });

    expect(input.value).toBe("John Doe");

});

En este ejemplo:

Estamos probando que nuestro componente maneja correctamente los cambios en un input.

Utilizamos la función getByLabelText para obtener el input que queremos probar.

Utilizamos la función fireEvent.change para simular un cambio en el valor del input.

Finalmente comprobamos que el valor del input sea el que esperamos utilizando la función toBe.


Testing de integración


En el contexto de React, se trata de probar cómo interactúan diferentes componentes entre sí y con otras partes de nuestra aplicación, como APIs y servicios externos.

Definición: Testing de integración

Es una técnica que nos permite probar cómo interactúan diferentes partes de nuestra aplicación cuando se combinan.

Es importante aplicar el testing de integración cuando queremos asegurarnos de que nuestra aplicación funciona correctamente en su conjunto, no solo cada parte individualmente.

También es importante para detectar errores en las interacciones entre componentes y otras partes de nuestra aplicación, antes de que se conviertan en problemas más grandes.


Pruebas: Componentes conectados


En React, a menudo utilizamos librerías como Redux o Context API para conectar diferentes componentes entre sí. Por lo tanto, es importante probar estas conexiones para asegurarnos de que los datos y el estado se transmitan correctamente.

Ejemplo

Cómo podemos hacer pruebas de integración para componentes conectados utilizando Context API

JavaScript Copiar código 
 
                            
import React from "react";

import { render, screen } from "@testing-library/react";

import { Router } from "react-router-dom";

import { createMemoryHistory } from "history";

import { MyContextProvider } from "./MyContext";

import MyComponentConnected from "./MyComponentConnected";



test("renders component with correct data from context", () => {

    const history = createMemoryHistory();

    const route = "/my-component";

    history.push(route);

    render(

        <MyContextProvider value={{ data: "Hello world!" }}>

            <Router history={history}>

                <MyComponentConnected />

            </Router>

        </MyContextProvider>

    );

    const dataFromContext = screen.getByText("Hello world!");

    expect(dataFromContext).toBeInTheDocument();

});

En este ejemplo:

Estamos utilizando el componente Router de React Router v6 para proveer la navegación a nuestro componente MyComponentConnected.

Estamos utilizando el componente MyContextProvider para proveer el contexto con los datos que queremos probar.

Utilizamos la función getByText de React Testing Library para obtener el texto que esperamos del componente.


Mocking: API y servicios externos


Como vimos en la clase anterior, en React a menudo utilizamos APIs y servicios externos para obtener y enviar datos. Sin embargo, no queremos depender de la disponibilidad de estos servicios externos durante nuestras pruebas de integración. Por lo tanto, podemos utilizar mocking para simular el comportamiento de estos servicios externos.

Ejemplo

Cómo podemos hacer mocking de una API utilizando Context API

JavaScript Copiar código 
 
                            
import React from "react";

import { render, screen } from "@testing-library/react";

import { MyContextProvider } from "./MyContext";

import MyComponentWithApi from "./MyComponentWithApi";



test("renders component with correct data from API", async () => {

    const mockData = { data: "Hello world!" };

    jest.spyOn(global, "fetch").mockResolvedValue({

        json: async () => mockData

    });

    render(

        <MyContextProvider>

            <MyComponentWithApi />

        </MyContextProvider>

    );

    const dataFromApi = await screen.findByText("Hello world!");

    expect(dataFromApi).toBeInTheDocument();

    global.fetch.mockRestore();

});

En este ejemplo:

Estamos utilizando `jest.spyOn` para crear un mock de la función `fetch` que nos permitirá simular una respuesta de una API ficticia.

Estamos utilizando la función `findByText` de React Testing Library para obtener el texto que esperamos del componente, y comprobando que se encuentra en el documento con la función `toBeInTheDocument`.

Restauramos la función `fetch` original utilizando `global.fetch.mockRestore()` para evitar interferir con otras pruebas.


Cypress


Cypress se ejecuta directamente en el navegador, lo que nos permite interactuar con nuestra aplicación como lo haría un usuario final, automatizando acciones como hacer clic en botones, completar formularios y navegar entre páginas.

Definición: Cypress

Es una herramienta de testing E2E de código abierto que nos permite escribir y ejecutar pruebas en aplicaciones web de manera rápida y sencilla.

Algunas ventajas de Cypress incluyen:

Facilidad de configuración y uso.

Ejecución rápida de pruebas.

Visualización en tiempo real de las pruebas en el navegador.

Depuración interactiva con herramientas del navegador.

Integración con sistemas de integración continua (CI).


Configuración


A continuación, describiremos los pasos para configurar Cypress en un proyecto React:

1

Instalar Cypress como dependencia de desarrollo:

CLI Copiar código 
 
                            
npm install cypress --save-dev
2

Agregar un script en nuestro archivo package.json para ejecutar Cypress:

JSON Copiar código 
 
                            
{

    "scripts": {

        "cypress:open": "cypress open"

    }

}
3

Ejecutar Cypress por primera vez, lo que generará automáticamente la estructura de carpetas necesaria para nuestras pruebas:

CLI Copiar código 
 
                            
npm run cypress:open

Escribiendo pruebas E2E


Ahora que tenemos Cypress configurado en nuestro proyecto, podemos comenzar a escribir pruebas E2E.

Las pruebas de Cypress se escriben utilizando Mocha y Chai, dos populares librerías de testing para JavaScript.

Ejemplo

Vamos a crear un archivo de prueba llamado home.spec.js dentro de la carpeta cypress/integration

JavaScript Copiar código 
 
                            
describe("Pruebas en la página principal", () => {

    beforeEach(() => {

        // Visita la página principal antes de cada prueba

        cy.visit("/");

    });



    it("debe mostrar el título correcto", () => {

        cy.get("h1").contains("Bienvenido a nuestra aplicación React");

    });



    it("debe navegar a la página de contacto al hacer clic en el enlace de contacto", () => {

        cy.get("a").contains("Contacto").click();

        cy.url().should("include", "/contact");

        cy.get("h2").contains("Página de Contacto");

    });

});

En este ejemplo, hemos creado dos pruebas simples:

La primera prueba verifica que el título de nuestra página principal sea correcto.

La segunda prueba verifica que, al hacer clic en el enlace de contacto, se navegue correctamente a la página de contacto y se muestre el título esperado.


Ejecutar pruebas


Para ejecutar nuestras pruebas, podemos utilizar el comando npm run cypress:open (o yarn cypress:open si utilizamos Yarn). Esto abrirá la interfaz gráfica de Cypress, donde podremos ver y ejecutar nuestras pruebas.

Cypress también ofrece varias herramientas útiles para facilitar el proceso de depuración y escritura de pruebas:

Time-traveling: Cypress toma capturas de pantalla automáticamente en cada paso de nuestra prueba. Podemos hacer clic en cada paso en la barra lateral para ver cómo se veía nuestra aplicación en ese momento.

Selector Playground: Esta herramienta nos permite explorar y encontrar selectores para elementos específicos de nuestra aplicación. Para utilizarla, simplemente haga clic en el ícono del Selector Playground en la parte superior de la ventana de Cypress.

Esperas y aserciones: Cypress incluye un conjunto de comandos para esperar a que ocurran ciertas condiciones antes de continuar con la prueba. Por ejemplo, podemos esperar a que una solicitud de red se complete o a que un elemento específico aparezca en la página. También podemos realizar aserciones para verificar que ciertas condiciones se cumplan, como verificar que un elemento tenga un texto específico o una clase CSS.

Ejemplo

Cómo utilizar esperas y aserciones en nuestras pruebas.

JavaScript Copiar código 
 
                            
describe("Pruebas en el formulario de contacto", () => {

    beforeEach(() => {

        // Visita la página de contacto antes de cada prueba

        cy.visit("/contact");

    });



    it("debe enviar el formulario de contacto correctamente", () => {

        cy.get("input[name='name']").type("John Doe");

        cy.get("input[name='email']").type("john.doe@example.com");

        cy.get("textarea[name='message']").type("¡Hola! Me encanta su aplicación.");

        cy.intercept("POST", "/api/contact", {

            statusCode: 200,

            body: { success: true }

        }).as("sendContactForm");

        cy.get("button[type='submit']").click();

        // Esperamos a que se complete la solicitud de red

        cy.wait("@sendContactForm");

        // Verificamos que se muestre el mensaje de éxito

        cy.get(".success-message").contains("Gracias por su mensaje");

    });

});

En este ejemplo, hemos creado una prueba para el formulario de contacto de nuestra aplicación.

La prueba completa los campos del formulario, intercepta la solicitud de red y simula una respuesta exitosa. Luego, espera a que se complete la solicitud de red y verifica que se muestre el mensaje de éxito.


Testing End-to-End (E2E)


En el contexto de React, se trata de probar cómo interactúan diferentes componentes entre sí y con otras partes de nuestra aplicación, incluyendo la navegación, interacción y validación.

Definición: Testing E2E

Es una técnica de testing que nos permite probar el comportamiento completo de una aplicación, simulando la interacción de un usuario real con la misma.

Es importante aplicar el testing E2E cuando queremos asegurarnos de que nuestra aplicación funciona correctamente en su conjunto, incluyendo la interacción con diferentes componentes y otras partes de la aplicación, como APIs y servicios externos.

Como vimos, Cypress es una herramienta popular para el testing E2E en aplicaciones web. Para configurar Cypress en nuestra aplicación React, primero debemos instalarlo utilizando npm o yarn:

CLI Copiar código 
 
                            
npm install cypress --save-dev

Una vez instalado, podemos abrir Cypress ejecutando el comando:

CLI Copiar código 
 
                            
npx cypress open

Esto nos abrirá la interfaz de usuario de Cypress, desde donde podremos crear y ejecutar nuestras pruebas E2E.


Creación de escenarios de prueba E2E


En Cypress, podemos crear escenarios de prueba E2E utilizando archivos de especificación que describen la interacción del usuario con nuestra aplicación.

Ejemplo

Si queremos probar un escenario de login en nuestra aplicación, podríamos crear un archivo de especificación como el siguiente

JavaScript Copiar código 
 
                            
describe("Login", () => {

    it("logs in successfully", () => {

        cy.visit("/login");

        cy.get("#username").type("user123");

        cy.get("#password").type("password123");

        cy.get("#login-btn").click();

        cy.url().should("eq", "https://myapp.com/dashboard");

        cy.contains("Welcome, user123!").should("be.visible");

    });



    it("shows error message on incorrect credentials", () => {

        cy.visit("/login");

        cy.get("#username").type("user123");

        cy.get("#password").type("wrongpassword");

        cy.get("#login-btn").click();

        cy.contains("Incorrect username or password").should("be.visible");

    });

});

En este ejemplo, estamos utilizando las funciones cy.visit, cy.get, cy.type, cy.click, cy.url y cy.contains de Cypress para simular la interacción del usuario con los elementos de la página y verificar que la aplicación se comporta correctamente.


Navegación, interacción y validación


En Cypress, podemos navegar por diferentes páginas de nuestra aplicación utilizando la función cy.visit.

También podemos interactuar con diferentes elementos de la página utilizando las funciones cy.get, cy.type, cy.click y otras funciones similares. Finalmente, podemos verificar que la aplicación se comporta correctamente utilizando las funciones cy.url, cy.contains y otras funciones similares.

Ejemplo

Cómo podemos hacer pruebas E2E para un componente de contador similar al que vimos en el ejemplo de pruebas unitarias:

JavaScript Copiar código 
 
                            
describe("Counter", () => {

    it("increments counter on button click", () => {

        cy.visit("/counter");
 
        cy.contains("Contador: 0").should("be.visible");

        cy.get("button").click();

        cy.contains("Contador: 1").should("...be.visible");

    });

});

En este ejemplo, estamos visitando la página `/counter`, verificando que el valor del contador comienza en 0, simulando un clic en el botón y verificando que el valor del contador se ha incrementado correctamente.

Es importante tener en cuenta que en las pruebas E2E estamos probando la aplicación como un todo, por lo que es importante pensar en los diferentes escenarios de uso y asegurarnos de que nuestra aplicación se comporta correctamente en todos ellos.


Automatización y CI


Es importante integrar las pruebas en el proceso de desarrollo desde el principio. Esto significa que debemos escribir pruebas unitarias, de integración y E2E en paralelo con el desarrollo de nuestra aplicación, y ejecutarlas regularmente para asegurarnos de que nuestra aplicación sigue funcionando correctamente.

Para facilitar la integración de pruebas en el proceso de desarrollo, podemos utilizar herramientas como Jest, React Testing Library y Cypress, que nos permiten escribir y ejecutar pruebas de forma fácil y rápida.


Configuración de un pipeline de CI


Para asegurarnos de que nuestras pruebas se ejecuten de forma regular y automatizada, podemos configurar un pipeline de CI utilizando herramientas como Jenkins, Travis CI o CircleCI.

Definición: Pipeline de CI

Es un conjunto de etapas que se ejecutan automáticamente cada vez que se hace un cambio en nuestro código, y que incluyen la ejecución de nuestras pruebas.

Ejemplo

Podríamos configurar nuestro pipeline de CI para que ejecute nuestras pruebas unitarias y de integración en cada push a la rama develop, y nuestras pruebas E2E en cada push a la rama main.

De esta forma, podemos asegurarnos de que nuestra aplicación sigue funcionando correctamente en todas las etapas del proceso de desarrollo.


Herramientas y servicios de CI


Algunas de las herramientas y servicios de CI populares que podemos utilizar en aplicaciones React son:

Jenkins: Una herramienta de automatización de CI/CD de código abierto que nos permite configurar pipelines de CI para ejecutar nuestras pruebas y desplegar nuestra aplicación.

Travis CI: Un servicio de CI en la nube que nos permite ejecutar nuestras pruebas en diferentes plataformas y configuraciones.

CircleCI: Un servicio de CI en la nube que nos permite ejecutar nuestras pruebas en diferentes plataformas y configuraciones, y que se integra fácilmente con herramientas como GitHub y Bitbucket.

GitLab CI/CD: Una herramienta de CI/CD integrada en GitLab que nos permite configurar pipelines de CI para ejecutar nuestras pruebas y desplegar nuestra aplicación.

Algunas de las mejores prácticas y técnicas avanzadas que podemos utilizar en el testing de aplicaciones React son:

Utilizar diferentes tipos de pruebas en paralelo, incluyendo pruebas unitarias, de integración y E2E, para asegurarnos de que nuestra aplicación funciona correctamente en todas las etapas del proceso de desarrollo.

Utilizar mocking para simular el comportamiento de APIs y servicios externos durante nuestras pruebas, y evitar depender de la disponibilidad de estos servicios externos durante nuestras pruebas.

Utilizar herramientas de profiling y análisis de rendimiento, como React Profiler y Chrome DevTools, para identificar y solucionar problemas de rendimiento en nuestra aplicación.

Utilizar técnicas de optimización de pruebas, como la ejecución paralela de pruebas y la selección selectiva de pruebas, para acelerar el tiempo de ejecución de nuestras pruebas y reducir el costo de mantenimiento.


TDD en aplicaciones React


En el contexto de React, esto significa que primero escribimos las pruebas unitarias para nuestros componentes, y luego escribimos el código de producción para hacer que las pruebas pasen.

Definición: TDD (Test-Driven Development)

Es una técnica de desarrollo de software que consiste en escribir las pruebas antes de escribir el código de producción.

El proceso de TDD se divide en tres pasos principales:

1

Escribir una prueba: Escribimos una prueba unitaria para el comportamiento que queremos implementar en nuestro componente.

2

Ejecutar la prueba: Ejecutamos la prueba y verificamos que falla.

3

Escribir el código: Escribimos el código de producción para hacer que la prueba pase.

Este proceso se repite iterativamente para cada nueva funcionalidad que queremos implementar en nuestra aplicación.

Ejemplo

Cómo podríamos utilizar TDD para desarrollar un componente de contador similar al que hemos visto anteriormente

import React from "react";

import { render, screen } from "@testing-library/react";

import Counter from "./Counter";



describe("Counter", () => {

    it("renders the initial count", () => {

        render(<Counter />);

        expect(screen.getByText("Contador: 0")).toBeInTheDocument();

    });

    it("increments the count on button click", () => {

        render(<Counter />);

        const buttonElement = screen.getByRole("button", { name: /incrementar/i });

        buttonElement.click();

        expect(screen.getByText("Contador: 1")).toBeInTheDocument();

    });

});

En este ejemplo:

Estamos utilizando TDD para desarrollar el componente Counter.

Hemos escrito una prueba para verificar que el contador se inicializa en 0.

Hemos ejecutado la prueba y verificado que falla.

Hemos escrito el código de producción para hacer que la prueba pase.

Hemos escrito otra prueba para verificar que el contador se incrementa correctamente al hacer clic en el botón.

Hemos ejecutado la prueba y verificado que falla.

Hemos escrito el código de producción para hacer que la prueba pase.

El proceso de TDD nos permite asegurarnos de que nuestras pruebas cubren todo el comportamiento de nuestro componente, y nos permite escribir código más robusto y mantenible al mismo tiempo.
