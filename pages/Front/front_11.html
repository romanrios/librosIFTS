Introducción a las APIs


Una API (Application Programming Interface) es un conjunto de reglas y especificaciones que permite a diferentes aplicaciones comunicarse entre sí. En el contexto del desarrollo web frontend, las APIs nos permiten obtener, enviar, actualizar y eliminar datos desde y hacia un servidor o una base de datos remota.

Las APIs son fundamentales para crear aplicaciones web interactivas y dinámicas.


Al utilizarlas, podemos separar la lógica de la interfaz de usuario (frontend) de la lógica del servidor y la base de datos (backend). Esto facilita la colaboración entre desarrolladores y mejora la escalabilidad y mantenimiento del proyecto.

Peticiones HTTP


Un navegador, durante la carga de una página, suele realizar múltiples peticiones HTTP a un servidor para solicitar los archivos que necesita renderizar en la página.

Es el caso de, en primer lugar, el documento .html de la página (donde se hace referencia a múltiples archivos) y luego todos esos archivos relacionados: los ficheros de estilos .css, las imágenes .jpg, .png, .webp u otras, los scripts .js, las tipografías .ttf, .woff o .woff2, etc.

Una petición HTTP es como suele denominarse a la acción por parte del navegador de solicitar a un servidor web un documento o archivo, ya sea un fichero .html, una imagen, una tipografía, un archivo .js, etc.


Gracias a dicha petición, el navegador puede descargar ese archivo, almacenarlo en un caché temporal de archivos del navegador y, finalmente, mostrarlo en la página actual que lo ha solicitado.


Métodos HTTP


Los métodos HTTP definen las acciones que podemos realizar sobre los recursos de una API.

Los más comunes son:

GET: Solicita la representación de un recurso o conjunto de recursos. No modifica ni crea recursos.

Ejemplo

Obtener la lista de todos los usuarios:
GET / users

POST: Crea un nuevo recurso en el servidor. Por lo general, requiere que enviemos datos en el cuerpo de la petición.

Ejemplo

Crear un nuevo usuario:
POST /users

{

    "name": "Juan",

    "email": "juan@example.com"

}

PUT: Actualiza un recurso existente en el servidor. Requiere que enviemos los datos actualizados en el cuerpo de la petición.

Ejemplo

Actualizar el correo electrónico de un usuario específico:
PUT /users/1

JSON Copiar código

    "email": "juan.nuevo@example.com"

}

DELETE: Elimina un recurso del servidor.

Ejemplo

Eliminar un usuario específico:
DELETE /users/1


Existen otros métodos HTTP menos comunes, como PATCH, HEAD y OPTIONS, que también pueden ser útiles en ciertas situaciones.


APIs REST y sus principios


Las APIs que siguen estos principios se denominan "APIs RESTful".

REST

Representational State Transfer

Es un estilo arquitectónico que define un conjunto de principios y restricciones para construir APIs escalables y fáciles de mantener.

A continuación, se describen los principios clave de las APIs RESTful:

Recursos: Las APIs RESTful se centran en los recursos, que pueden ser objetos, colecciones de objetos o cualquier entidad que queramos manipular. Cada recurso debe tener un identificador único, como una URL.

Representación: Un recurso puede tener múltiples representaciones, como JSON, XML o HTML. El cliente y el servidor pueden negociar la representación deseada a través de encabezados HTTP.

Operaciones sin estado: Cada petición a una API RESTful debe contener toda la información necesaria para procesarla. El servidor no debe almacenar información sobre el estado de las peticiones anteriores.

Cacheabilidad: Las respuestas de una API RESTful pueden ser almacenadas en caché para mejorar la eficiencia y reducir la carga en el servidor.

Sistema en capas: Las APIs RESTful pueden estar compuestas por múltiples capas de componentes, cada una con sus responsabilidades y reglas de comunicación.


Peticiones HTTP con fetch


La función fetch es una API moderna y fácil de usar para realizar peticiones HTTP en el navegador.

Es compatible con promesas y async/await, lo que nos permite escribir código asíncrono más limpio y legible.

La sintaxis básica de fetch es la siguiente:

JavaScript Copiar código 
 
                            
fetch(url, opciones)

    .then((respuesta) => respuesta.json())

    .then((datos) => console.log(datos))

    .catch((error) => console.error(error));

Donde url es la dirección del recurso que queremos solicitar y opciones es un objeto opcional que contiene información sobre la petición, como el método HTTP, los encabezados y el cuerpo.


Peticiones: JSONPlaceholder


Para realizar peticiones HTTP con diferentes métodos, simplemente debemos especificar el método en el objeto de opciones.

Utilizaremos la API de JSONPlaceholder, que nos ofrece un conjunto de recursos falsos para probar nuestras peticiones.

Resumen

JSONPlaceholder es una herramienta en línea que proporciona una API de simulación gratuita para desarrolladores, la cual nos permite realizar pruebas de sus aplicaciones sin tener que conectarnos a una API real, ya que se utiliza para simular una API RESTful.


JSONPlaceholder cuenta con una serie de recursos comunes que se pueden utilizar para probar la funcionalidad de una aplicación. Los recursos incluyen usuarios, publicaciones, comentarios, álbumes, fotos y tareas. Cada recurso tiene una serie de rutas que permiten realizar acciones como obtener, crear, actualizar y eliminar datos.

Para acceder a los recursos de JSONPlaceholder, se deben realizar peticiones HTTP a través de la URL proporcionada por la API. La API es compatible con los métodos HTTP GET, POST, PUT, PATCH y DELETE.

Al realizar una solicitud HTTP a la API, se devuelve una respuesta en formato JSON que contiene los datos solicitados.


A continuación, se muestran ejemplos de cómo realizar peticiones GET, POST, PUT y DELETE con la API de JSONPlaceholder.


GET


Para realizar una petición GET, simplemente llamamos al método fetch con la URL del recurso que deseamos obtener.

Las peticiones GET se utilizan para solicitar datos a una API o servidor.

JavaScript Copiar código 
 
                            
fetch("https://jsonplaceholder.typicode.com/posts")

  .then((response) => response.json())

  .then((data) => console.log(data))

  .catch((error) => console.error("Error:", error));

1

Se utiliza fetch(): Para obtener datos de la API pública de JSONPlaceholder, que proporciona datos JSON de prueba.

2

La URL "https://jsonplaceholdertypicode.com/posts" es la dirección a la que se realiza la petición GET: Al enviar la solicitud, el servidor responde con una respuesta HTTP que contiene los datos solicitados en formato JSON.

3

La función then(): Es una función de la Promesa, que se utiliza para manejar el resultado exitoso de la respuesta del servidor. En este caso, la primera función then() toma la respuesta del servidor y llama al método json() en ella. Este método convierte la respuesta en un objeto JavaScript legible que se puede utilizar posteriormente. Veremos el funcionamiento de promesas más adelante.

4

Después de llamar al método json(), el objeto de respuesta se pasa a la segunda función then() como parámetro: En este caso, simplemente estamos registrando los datos en la consola del navegador utilizando console.log().

5

La función catch(): Si se produce un error en la solicitud GET, la función catch() se utiliza para manejarlo y registrar el mensaje de error en la consola utilizando console.error().


POST


Para realizar una petición GET, simplemente llamamos al método fetch con la URL del recurso que deseamos obtener.

Para realizar una petición POST, pasamos un objeto como segundo argumento de fetch, especificando el método, los encabezados y el cuerpo de la petición.

Las peticiones POST se utilizan para enviar datos a una API o servidor y crear un nuevo recurso.

const postData = {

   body: "Este es un nuevo post de prueba.",

   userId: 1

fetch("https://jsonplaceholder.typicode.com/posts", {

    method: "POST",

    headers: {

        "Content-Type": "application/json"

    body: JSON.stringify(postData)

    .then((data) => console.log(data))

    .catch((error) => console.error("Error:", error));

Se crea un objeto llamado postData: Contiene los datos que se van a enviar en la petición HTTP POST. Este objeto contiene tres propiedades: title, body, y userId.

2

Se llama al método fetch(): Para realizar la petición POST a la URL "https://jsonplaceholder.typicode.com/posts". En este caso, se utiliza un objeto de configuración que se pasa como segundo parámetro de fetch() para personalizar la solicitud HTTP.

Uso de encabezados (headers)


Para agregar encabezados a nuestras peticiones, incluimos un objeto headers en las opciones de fetch.

Los encabezados HTTP nos permiten enviar información adicional sobre la petición, como el tipo de contenido, la autenticación o la aceptación de ciertos formatos de respuesta.


Ejemplo

Para enviar un token de autenticación y aceptar respuestas en formato JSON.

JavaScript Copiar código 
 
                            
fetch("https://jsonplaceholder.typicode.com/posts", {

    headers: {

        "Content-Type": "application/json",

        Authorization: "Bearer <TOKEN>"

    }

})

    .then((response) => response.json())

    .then((data) => console.log(data))

    .catch((error) => console.error("Error:", error));

En el objeto de configuración de la solicitud HTTP GET, se establecen dos encabezados.

El primer encabezado es "Content-Type" que indica el tipo de contenido de la solicitud. En este caso, se establece en "application/json" para indicar que la solicitud enviará un objeto JSON.

El segundo encabezado es "Authorization" que se utiliza para agregar el token de autenticación. El valor del encabezado "Authorization" comienza con "Bearer" seguido del token de autenticación.

Definición: Token bearer

Se utiliza para proporcionar una forma segura de autenticar y autorizar solicitudes HTTP, protegiendo los recursos del servidor de accesos no autorizados.



Un token bearer es un tipo de token de autenticación que se utiliza comúnmente en aplicaciones web y móviles. El token bearer se agrega al encabezado "Authorization" de la solicitud HTTP GET.

Cuando el servidor recibe la solicitud HTTP GET, verifica el token bearer en el encabezado "Authorization" para garantizar que el cliente que realiza la solicitud esté autorizado para acceder a los recursos solicitados. Si el token bearer es válido, el servidor procesa la solicitud y devuelve una respuesta con los recursos solicitados.

Si el token bearer es inválido o no existe, el servidor devuelve un error de autenticación y la solicitud falla.

Resumen

El token bearer se utiliza para proporcionar una forma segura de autenticar y autorizar solicitudes HTTP, protegiendo los recursos del servidor de accesos no autorizados.


Existen muchos encabezados (headers) que se pueden utilizar al realizar peticiones HTTP. Algunos de los más comunes y sus usos son:

Accept: Este encabezado indica los tipos de contenido que el cliente puede entender. Permite solicitar una representación específica de un recurso (por ejemplo, JSON, XML, etc.).

JavaScript Copiar código 
 
                            
fetch("https://jsonplaceholder.typicode.com/posts", {

    headers: {

        Accept: "application/json"

    }

})

    .then((response) => response.json())

    .then((data) => console.log(data))

    .catch((error) => console.error("Error:", error));

Accept-Language: Este encabezado indica el idioma preferido del cliente. Puede ser útil si el servidor proporciona contenido en diferentes idiomas.

JavaScript Copiar código 
 
                            
fetch("https://api.example.com/localized-content", {

    headers: {

        "Accept-Language": "es"

    }

})

    .then((response) => response.json())

    .then((data) => console.log(data))

    .catch((error) => console.error("Error:", error));

User-Agent: Este encabezado proporciona información sobre el cliente que realiza la solicitud, como el navegador y el sistema operativo. Algunos servidores pueden utilizar esta información para adaptar la respuesta a las capacidades específicas del cliente.

JavaScript Copiar código 
 
                            
fetch("https://api.example.com/device-specific-content", {

    headers: {

        "User-Agent":

        "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/58.0.3029.110 Safari/537.36"

    }

})

    .then((response) => response.json())

    .then((data) => console.log(data))

    .catch((error) => console.error("Error:", error));

Cache-Control: Este encabezado permite controlar el almacenamiento en caché de la respuesta por parte del navegador o el servidor proxy. Podemos especificar cómo y por cuánto tiempo se debe almacenar la respuesta en caché.

JavaScript Copiar código 
 
                            
fetch("https://api.example.com/cacheable-content", {

    headers: {

        "Cache-Control": "max-age=3600" // Cachea la respuesta durante 1 hora (3600 segundos)

    }

})

    .then((response) => response.json())

    .then((data) => console.log(data))

    .catch((error) => console.error("Error:", error));

Authorization: Este encabezado permite incluir credenciales de autenticación en la petición, como tokens de acceso o credenciales básicas (usuario y contraseña). Es importante para acceder a recursos protegidos o realizar acciones que requieren permisos específicos.

Content-Length: Este encabezado indica el tamaño en bytes del cuerpo de la petición. Aunque algunos navegadores pueden establecer este encabezado automáticamente, es posible que debas establecerlo manualmente en algunos casos.

JavaScript Copiar código 
 
                            
const postData = {

    title: "Nuevo Post",

    body: "Este es un nuevo post de prueba.",

    userId: 1

};



const jsonString = JSON.stringify(postData);



fetch("https://jsonplaceholder.typicode.com/posts", {

    method: "POST",

    headers: {

        "Content-Type": "application/json",

        "Content-Length": jsonString.length

    },

    body: jsonString

})

    .then((response) => response.json())

    .then((data) => console.log(data))

    .catch((error) => console.error("Error:", error));

Estos son solo algunos ejemplos de los encabezados HTTP que podemos utilizar en las peticiones.

Es importante tener en cuenta que los encabezados pueden variar según la API o el servidor con el que estemos trabajando, por lo que siempre es recomendable consultar la documentación correspondiente para saber qué encabezados son compatibles y cómo utilizarlos adecuadamente.


Parámetros en las peticiones


Los parámetros nos permiten personalizar nuestras peticiones, por ejemplo, para filtrar, paginar o buscar datos específicos. Los parámetros se incluyen en la URL de la petición.

Ejemplo

Solicitar los primeros 5 posts de un usuario específico (userId = 1).

JavaScript Copiar código 
 
                            
fetch("https://jsonplaceholder.typicode.com/posts?userId=1&_limit=5")

    .then((response) => response.json())

    .then((data) => console.log(data))

    .catch((error) => console.error("Error:", error));

En este caso, se está solicitando los primeros 5 posts de un usuario en particular, cuyo ID de usuario es 1.

La URL especificada en el primer parámetro de fetch() contiene los parámetros de consulta "userId=1" y "_limit=5". El primero es utilizado para filtrar los posts para el usuario con ID 1, mientras que el segundo parámetro especifica que solo se deben devolver los primeros 5 posts.

También podemos utilizar la clase URLSearchParams para construir la cadena de parámetros de forma más estructurada.

JavaScript Copiar código 
 
                            
const params = new URLSearchParams({

    userId: 1,

    _limit: 5

});



fetch(`https://jsonplaceholder.typicode.com/posts?${params}`)

    .then((response) => response.json())

    .then((data) => console.log(data))

    .catch((error) => console.error("Error:", error));

La URL de la petición se construye utilizando la clase URLSearchParams() para crear los parámetros de consulta. En este caso, se están especificando dos parámetros de consulta: "userId=1" y "_limit=5".

La instancia de URLSearchParams() se inicializa con los parámetros de consulta y se almacena en la variable params. Luego, la URL de la petición se construye utilizando la interpolación de cadenas de texto y la variable params.


Promesas y async/await


Nos permiten trabajar con valores que aún no están disponibles, pero que se espera que lo estén en el futuro.

Definición: Promesas

En JavaScript son objetos que representan el resultado de una operación asíncrona que puede ser completada en algún momento.

Una promesa puede tener tres estados:

Pendiente (pending): El resultado aún no está disponible.

Resuelta (fulfilled): La operación se completó con éxito y el resultado está disponible.

Rechazada (rejected): Ocurrió un error durante la operación.

Ejemplo de una promesa simple:

JavaScript Copiar código 
 
                            
const promesa = new Promise((resolve, reject) => {

    setTimeout(() => {

        resolve("La promesa se resolvió exitosamente");

    }, 1000);

});

then, catch y finally


Para manejar el resultado de una promesa, podemos utilizar los métodos then, catch y finally.

Then: Se ejecuta cuando la promesa se resuelve exitosamente. Recibe una función que será llamada con el valor resuelto.

Catch: Se ejecuta cuando la promesa es rechazada. Recibe una función que será llamada con el motivo del rechazo (generalmente un error).

Finally: Se ejecuta siempre, independientemente de si la promesa fue resuelta o rechazada. Recibe una función sin argumentos.

Ejemplo:

JavaScript Copiar código 
 
                            
fetch("https://jsonplaceholder.typicode.com/posts")

    .then((response) => response.json())

    .then((data) => console.log("Posts:", data))

    .catch((error) => console.error("Error:", error))

    .finally(() => console.log("Petición finalizada"));

1

En la primera llamada a then(): Se toma la respuesta de la petición HTTP y se llama al método json(), el cual convierte la respuesta en un objeto JSON que puede ser manipulado.

2

En la siguiente llamada a then(): Se accede a los datos analizados y se muestran en la consola con el método console.log().

3

Si la petición falla: El método catch() captura el error y muestra un mensaje de error en la consola utilizando el método console.error().

4

Además, se utiliza el método finally(): Para realizar una acción después de que la petición se haya completado, en este caso, simplemente muestra un mensaje en la consola para indicar que la petición ha finalizado.

Definición: Async/await

La sintaxis nos permite simplificar el manejo de promesas, haciendo que nuestro código asíncrono se vea y funcione de manera similar al código síncrono.

Dentro de una función async, podemos utilizar await antes de una promesa para esperar a que se resuelva y obtener su valor.

Interpretación de estados HTTP


Los códigos de estado HTTP son códigos numéricos de tres dígitos que indican el resultado de una petición HTTP.

Estos códigos se agrupan en cinco clases, según su primer dígito:

1xx (Informativo): La solicitud fue recibida, y el proceso continúa.

2xx (Éxito): La solicitud fue recibida, entendida y aceptada correctamente.

3xx (Redirección): Se requieren más acciones para completar la solicitud.

4xx (Error del cliente): La solicitud contiene sintaxis incorrecta o no puede ser cumplida.

5xx (Error del servidor): El servidor falló al cumplir una solicitud válida.


Algunos códigos de estado HTTP comunes son:

200 OK: La solicitud fue exitosa, y el servidor ha devuelto los datos solicitados.

201 Created: La solicitud fue exitosa, y se ha creado un nuevo recurso como resultado.

400 Bad Request: La solicitud es incorrecta o mal formada.

401 Unauthorized: La solicitud requiere autenticación del usuario.

403 Forbidden: El cliente no tiene permisos para acceder al recurso solicitado.

404 Not Found: El recurso solicitado no pudo ser encontrado en el servidor.

500 Internal Server Error: El servidor encontró un error al procesar la solicitud.

Es importante tener cuidado al utilizar useEffect con peticiones HTTP para evitar bucles infinitos.

Bucle infinito

Peticiones HTTP con Axios


Axios es una biblioteca popular de JavaScript que se utiliza para realizar peticiones HTTP en aplicaciones web. Facilita el proceso de envío de peticiones HTTP a servidores y manejo de respuestas, tanto en aplicaciones del lado del cliente como en aplicaciones Node.js del lado del servidor. 

Axios es especialmente útil en aplicaciones basadas en React, aunque también se puede utilizar con otros frameworks y bibliotecas de JavaScript.

Axios es una biblioteca basada en promesas, lo que significa que las peticiones HTTP realizadas con Axios devuelven promesas que pueden ser manejadas usando then(), catch() y finally().

Axios ofrece una API sencilla para realizar peticiones HTTP utilizando métodos como get(), post(), put() y delete().

Axios maneja automáticamente la conversión de datos a JSON y la configuración de encabezados (headers) apropiados, simplificando el proceso de enviar y recibir datos.

Axios permite configurar interceptors, que son funciones que se ejecutan antes de que las peticiones sean enviadas o antes de que las respuestas sean recibidas y procesadas por la aplicación.

Axios proporciona soporte para la cancelación de peticiones, lo que permite detener peticiones en curso bajo ciertas condiciones.

Ventajas

Ventajas de Axios en comparación con Fetch:

Axios es más fácil de usar que Fetch en términos de sintaxis y manejo de errores.

Maneja automáticamente la conversión de datos a JSON, mientras que con Fetch es necesario hacerlo de forma manual.

Permite el uso de interceptors para modificar peticiones y respuestas antes de que sean procesadas.

Admite la cancelación de peticiones, una característica que no está disponible de forma nativa con Fetch.

Axios ofrece una API unificada para el lado del cliente y del servidor, mientras que Fetch solo está disponible en navegadores.

Desventajas de Axios

Desventajas de Axios en comparación con Fetch:

Axios es una biblioteca externa, lo que significa que debe ser instalada y mantenida, mientras que Fetch está integrado en los navegadores modernos y no requiere instalación.

Fetch es una API estándar y nativa en los navegadores, lo que puede ser considerado una ventaja en términos de compatibilidad y soporte a largo plazo.

Fetch puede ser más adecuado para aplicaciones que requieren un enfoque minimalista y no necesitan características adicionales proporcionadas por Axios.

Instalación

APIs gratuitas para utilizar


Para consumir APIs en React, utilizamos la función 'fetch' o bibliotecas como 'axios'. Estas funciones nos permiten realizar solicitudes HTTP a las APIs y recibir datos para mostrar en nuestra aplicación.

APIs

Interfaces de programación de aplicaciones

Actúan como un puente entre su aplicación y un servidor externo. Nos permiten solicitar datos y realizar acciones en nombre del usuario. Al consumir una API, básicamente estamos haciendo una solicitud a un servidor y recibiendo datos en un formato específico, generalmente JSON, que podemos utilizar en nuestras aplicaciones.

Hay muchas APIs gratuitas disponibles en línea para usar en proyectos de desarrollo de aplicaciones. Algunas de las APIs más populares son:

OpenWeatherMap API: Esta API proporciona datos meteorológicos en tiempo real para cualquier ubicación en todo el mundo. Se puede utilizar para mostrar información del tiempo en una aplicación de clima en React. Para utilizar esta API, es necesario registrarse y obtener una clave de API gratuita.

Random User API: Esta API proporciona datos aleatorios de usuarios como nombres, correos electrónicos, fotos de perfil y ubicaciones. Se puede utilizar para crear una lista de usuarios ficticios en una aplicación de redes sociales en React.

NewsAPI: Esta API proporciona noticias de todo el mundo en diferentes categorías. Se puede utilizar para mostrar noticias en tiempo real en una aplicación de noticias en React. Para utilizar esta API, es necesario registrarse y obtener una clave de API gratuita.

Chuck Norris API: Esta API proporciona datos aleatorios sobre Chuck Norris, incluyendo chistes y hechos divertidos sobre el famoso actor. Se puede utilizar para crear una aplicación de trivia en React.

The Dog API: Esta API ofrece imágenes aleatorias de perros y detalles sobre diferentes razas.

REST Countries: Con esta API, podrán obtener información sobre todos los países del mundo, incluyendo la población, el área, la capital y más.


Se pueden buscar más APIs públicas para utilizar en este repositorio: https://github.com/public-apis/public-apis.


Uso de API keys


Si bien algunas APIs nos permiten acceder a sus recursos y realizar peticiones de forma simple y gratuita sin necesidad de registrarnos, otras APIs requieren el uso de una API key.

Definición: API key

Es un identificador único que nos permite autenticarnos en el servidor y acceder a sus recursos. Por lo general, debemos registrarnos en el sitio web del proveedor de la API para obtener nuestra propia API key.


Muchas APIs permiten obtener API keys de forma gratuita, sin necesidad de hacer un pago, pero para obtener esta clave debemos registrarnos en su página. También hay APIs pagas o que ofrecen un límite gratuito de peticiones y luego cobran por uso.

Nunca debemos compartir nuestras API keys públicamente, ya que pueden ser utilizadas por otras personas y generar cargos en nuestra cuenta o violaciones de la política de uso de la API.


Pasos para consumir una API en React:

1

Registrarse en el sitio web de la API (si es necesario) y obtener una API key.

2

Instalar Axios o utilizar la función 'fetch' para realizar solicitudes HTTP.

3

Realizar una solicitud a la API utilizando la URL proporcionada en la documentación.

4

Manejar la respuesta de la API, extrayendo los datos relevantes y mostrándolos en la aplicación.

5

Asegurarse de manejar posibles errores durante la solicitud, como problemas de conexión o errores en la API.

JavaScript Copiar código 
 
                            
import { useEffect, useState } from "react";

import { Container, Pagination } from "react-bootstrap";

import axios from "axios";

import CountryList from "./components/CountryList";

import Header from "./components/Header";



function App() {

  const [countries, setCountries] = useState([]);

  const [currentPage, setCurrentPage] = useState(1);

  const [pageSize, setPageSize] = useState(10);



  const handlePageChange = (pageNumber) => {

    setCurrentPage(pageNumber);

    fetchAllCountries(pageNumber, pageSize);

  };



  const fetchAllCountries = async (pageNumber, pageSize) => {

    try {

      const response = await axios.get(

        `https://restcountries.com/v3.1/all?page=${pageNumber}&pageSize=${pageSize}`

      );

      setCountries(response.data);

    } catch (error) {

      console.error(error);

    }

  };



  const pageNumbers = Array.from(

    { length: Math.ceil(countries.length / pageSize) },

    (_, index) => index + 1

  );



  const searchCountries = async (searchTerm) => {

    try {

      const response = await axios.get(

        `https://restcountries.com/v3.1/name/${searchTerm}`

      );

      setCountries(response.data);

      setCurrentPage(1);

    } catch (error) {

      console.error(error);

    }

  };


  const handleSearch = async (searchTerm, pageNumber = 1, pageSize = 10) => {

    try {

      if (searchTerm) {

        searchCountries(searchTerm);

      } else {

        fetchAllCountries(pageNumber, pageSize);

      }

    } catch (error) {

      console.error(error);

    }

  };



  useEffect(() => {

    fetchAllCountries();

  }, []);



  return (

    <div className="App">

      <Container>

        <Header handleSearch={handleSearch}></Header>

        <CountryList

          countries={countries.slice(

            (currentPage - 1) * pageSize,

            currentPage * pageSize

          )}

        />

        <Container>

          <Pagination className="pagination">

            <Pagination.Prev
 
              onClick={() => handlePageChange(currentPage - 1)}

            />

            {pageNumbers.map((number) => (

              <Pagination.Item

                key={number}

                active={number === currentPage}

                onClick={() => handlePageChange(number)}

              >

                {number}

              </Pagination.Item>

            ))}

            <Pagination.Next

              onClick={() => handlePageChange(currentPage + 1)}

            />

          </Pagination>

        </Container>

      </Container>

    </div>

  );

}



export default App;

Se inicializan tres estados utilizando la función useState de React: countries, currentPage y pageSize. currentPage es el número de página actual y pageSize es la cantidad de elementos por página que se mostrarán.

La función handlePageChange cambia el estado de currentPage y llama a fetchAllCountries con el número de página y tamaño de página actuales.

La función fetchAllCountries se encarga de realizar la solicitud GET a la API con los parámetros pageNumber y pageSize.

Se crea una matriz pageNumbers que representa el número total de páginas necesarias para mostrar todos los países, dividiendo la longitud total de countries por pageSize y redondeando hacia arriba.

La función searchCountries maneja la búsqueda de países por nombre y actualiza el estado de countries con los resultados de la búsqueda y la página actual se establece en 1.

La función handleSearch se encarga de manejar tanto las búsquedas como las solicitudes de todos los países. Si se realiza una búsqueda, llama a searchCountries y si no, llama a fetchAllCountries.

En el return, se muestra el componente Pagination con los botones Previous y Next para cambiar entre páginas y los números de página correspondientes que se generan mediante el método map en la matriz pageNumbers. Cada número de página se muestra como un elemento de Pagination.Item y al hacer clic en él se llama a handlePageChange para cambiar a la página seleccionada.

Manejo del estado global en React


El estado en React es un objeto que contiene información sobre el componente y cómo debería renderizarse. El estado permite a nuestros componentes ser interactivos y dinámicos, respondiendo a eventos del usuario y actualizaciones de datos. La gestión adecuada del estado es esencial para construir aplicaciones eficientes y fáciles de mantener.


Diferencias: Estado local y estado global



El estado local es el estado que está contenido dentro de un componente y sólo es accesible por ese componente.

Cuando un componente necesita compartir datos con sus componentes hijos, se pueden pasar como props. Sin embargo, este enfoque puede volverse complicado a medida que la aplicación crece y los componentes necesitan compartir datos entre sí.


El estado global es el estado que es accesible por varios componentes en toda la aplicación.

Esto es útil cuando tenemos datos que deben compartirse entre componentes que no están directamente relacionados. El manejo del estado global nos ayuda a evitar el prop drilling (pasar datos a través de múltiples niveles de componentes) y facilita la actualización y el seguimiento del estado compartido.


Soluciones para manejar el estado global


Existen varias soluciones para manejar el estado global en React:

Context API

Es una función incorporada en React que permite compartir datos entre componentes sin pasarlos explícitamente a través de props.

Con createContext, useContext y el componente Provider, podemos crear contextos y compartir datos fácilmente entre componentes.

  const [theme, setTheme] = useState("light");



  useEffect(() => {

Redux

Es una librería popular para manejar el estado global en aplicaciones React.

Definición: Redux

Se basa en el patrón de arquitectura Flux y utiliza un store único, acciones y reducers para gestionar el estado.

Con redux y react-redux, podemos conectar nuestros componentes a un store global y manejar fácilmente acciones y cambios en el estado.

MobX

Es otra librería para manejar el estado global que se basa en el concepto de observables. MobX utiliza una forma más declarativa para manejar el estado, lo que puede simplificar el código y hacer que sea más fácil de entender.


Estado global: Ventajas y desventajas


Manejar el estado global en aplicaciones React tiene varias ventajas y desventajas:

Ventajas


Facilita la comunicación de datos entre componentes no relacionados, eliminando la necesidad de prop drilling.

Mejora la organización del código, ya que el estado compartido se gestiona en un solo lugar, en lugar de estar disperso en múltiples componentes.

Facilita el seguimiento y la depuración de cambios en el estado, especialmente cuando se utiliza una solución como Redux, que proporciona herramientas para monitorear el estado y las acciones en tiempo real.

Desventajas


Añade complejidad adicional a la aplicación, especialmente cuando se utilizan soluciones como Redux, que requieren una configuración inicial y un aprendizaje de nuevos conceptos.

Puede afectar el rendimiento si no se maneja correctamente, ya que las actualizaciones del estado global pueden causar re-renderizados innecesarios en componentes que no necesitan actualizarse.

Es importante evaluar si realmente necesitamos manejar el estado global en nuestra aplicación.

Si sólo estamos compartiendo datos entre componentes cercanos en la jerarquía, es posible que podamos manejarlo utilizando el estado local y props.

Sin embargo, si la aplicación crece y nos enfrentamos a desafíos en la gestión del estado compartido, entonces el uso de una solución de estado global puede ser beneficioso.


Context API


Context API es una herramienta incorporada en React que permite compartir datos entre componentes sin pasarlos explícitamente a través de props. Los principales componentes de la Context API son:

CreateContext: Función que crea un nuevo contexto que contendrá los datos compartidos.

UseContext: Hook que permite consumir un contexto en componentes funcionales.

Provider: Componente que envuelve a otros componentes que necesitan acceso a los datos del contexto y proporciona esos datos.

Consumer: Componente que se utiliza para consumir un contexto en componentes de clase (aunque en componentes funcionales, se prefiere el uso de useContext).

Patrones comunes


Compound components: Se refiere a la técnica de utilizar múltiples componentes relacionados que trabajan juntos para lograr una funcionalidad específica. Estos componentes pueden consumir el mismo contexto y funcionar de manera cohesiva.

Higher-order components (HOC): Es una función que toma un componente y devuelve un nuevo componente con algunas propiedades adicionales. Podemos utilizar HOC para inyectar el contexto en nuestros componentes sin tener que utilizar useContext en cada uno de ellos.


Mejores prácticas


Mejores prácticas y consideraciones de rendimiento al usar Context API:

Evitar crear contextos innecesarios: No es necesario utilizar la Context API para todos los datos compartidos en la aplicación. Si los datos sólo se comparten entre componentes cercanos, debemos utilizar props en su lugar.

Usar múltiples contextos para datos no relacionados: Si tenemos datos no relacionados que necesitan ser compartidos, es mejor separar los contextos para evitar re-renderizados innecesarios en los componentes que sólo necesitan parte de los datos.
