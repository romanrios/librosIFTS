Introducción


En el mundo de la automatización web, Selenium WebDriver es una herramienta ampliamente utilizada para interactuar con navegadores y realizar tareas automatizadas. Sin embargo, su funcionamiento es a menudo calificado como bloqueante para las APIs debido a su naturaleza asincrónica en un entorno web. Este hecho radica en el hecho de que Selenium WebDriver opera fuera del proceso normal del navegador, enviando instrucciones a la aplicación web sin llevar un registro en tiempo real del estado del Document Object Model (DOM).

Esto plantea desafíos significativos, especialmente en lo que respecta a las condiciones de desfase entre las instrucciones del usuario y la respuesta del navegador. Estas disparidades pueden dar lugar a errores como no such element al intentar localizar elementos en la página.

En este contexto, exploraremos los desafíos y las estrategias para abordarlos, con un enfoque particular en las esperas explícitas, que son una herramienta clave para superar estos obstáculos y garantizar la sincronización adecuada entre Selenium WebDriver y la aplicación web.



Selenium Web Driver Waits


Normalmente se dice que Selenium Web Driver es bloqueante para las APIs. Esto se debe a que es una librería que está fuera del proceso normal que envía instrucciones al navegador sobre lo que debe hacer. Y como la plataforma web es intrínsecamente asíncrona por naturaleza, Selenium Web Driver no lleva un registro en tiempo real del estado del DOM. Esto trae algunos desafíos que discutiremos a continuación.

La mayoría de los problemas que surgen de utilizar Selenium y Selenium Web Driver están relacionados a las condiciones de desfasaje que existen entre el navegador y las instrucciones del usuario. Un ejemplo puede ser cuando el/la usuario/a le instruye al navegador que navegue a una cierta página, pero se retorna un no such element error cuando se lo intenta localizar.

Waiting o esperar es hacer que se espere un cierto tiempo luego de la ejecución de la tarea automatizada antes de continuar con el paso siguiente.

Para superar el problema de las condiciones de desfasaje entre el navegador y el script del WebDriver mencionado anteriormente, la mayoría de los clientes de Selenium contienen un wait package o paquete de espera. 

Cuando utilizamos este paquete de esperas se lo conoce típicamente como una espera explícita.



Explicit Wait o esperas explícitas


Las esperas explícitas están disponibles para los/las clientes de Selenium para lenguajes imperativos y procedurales. Le permiten a nuestro código detener la ejecución de programas o hilos, hasta que la condición pasada se resuelva. La condición es llamada con cierta frecuencia hasta que transcurra el tiempo de espera. Esto implica que, mientras la condición devuelva un valor falso, seguirá intentando y esperando.

Dado que la espera explícita nos permite esperar a que una condición ocurra, son útiles para sincronizar el estado entre el navegador y su DOM, y nuestro script de WebDriver.

Pasamos la condición como una referencia de función que la espera ejecutará repetidamente hasta que devuelva un valor verdadero. Un valor verdadero es todo aquello que resulte en un booleano verdadero en el lenguaje empleado, como un string, un number, un boolean, un objeto (incluyendo un WebElement), o una lista o secuencia poblada (no vacía). Esto implica que una lista vacía devuelve falso. Cuando la condición es verdadera y se anula la espera del bloqueo, el valor de retorno de la condición se convierte en el valor de retorno de la espera.

Si la condición falla –por ejemplo, un valor verdadero retornado de la condición nunca es alcanzado– la espera lanzará un error o excepción llamado timeout error.

Condiciones esperadas


Debido a que es bastante común tener que sincronizar el DOM y nuestras instrucciones, la mayoría de los/las clientes también incluyen un conjunto de condiciones esperadas predefinidas. Como su nombre lo indica, son condiciones que están predefinidas para operaciones de espera frecuentes.

Las condiciones disponibles en los diferentes lenguajes varían, pero aquí detallaremos algunas de ellas:

La alerta está presente (alert is present).

El elemento existe (element exists).

El elemento es visible (element is visible).

El título contiene (title contains).

El título es (title is).

Desactualización del objeto (element staleness).

Texto visible (visible text).



Implicit Wait o espera implícita


Hay un segundo tipo de espera que se distingue de la espera explícita, llamada implicit wait o espera implícita. Al esperar implícitamente, WebDriver sondea el DOM durante un cierto tiempo cuando intenta encontrar cualquier elemento. Esto puede ser útil cuando ciertos elementos de la página web no están disponibles de forma inmediata y necesitan un tiempo para cargar.

Esperar implícitamente a que los elementos aparezcan está deshabilitado por defecto y requerirá una habilitación manual por sesión. Combinar esperas explícitas e implícitas causará consecuencias no deseadas, principalmente esperas inactivas durante el tiempo máximo, incluso si el elemento está disponible o la condición es verdadera.

Cuidado: no mezclar esperas implícitas y explícitas. Hacerlo puede causar tiempos de espera impredecibles. Por ejemplo, definir una espera implícita de 10 segundos y una explícita de 15 segundos podría causar un timeout luego de 20 segundos.

Una espera implícita es decirle a WebDriver que sondee el DOM durante un cierto período de tiempo cuando intenta encontrar un elemento o elementos si no están disponibles de inmediato. La configuración predeterminada es 0, lo que significa que está deshabilitada. Una vez establecida, la espera implícita se establece durante la vida de la sesión.




FluentWait o espera fluida


La instancia de FluentWait define la máxima cantidad de tiempo que esperar por una condición como la frecuencia con la cual chequear dicha condición.

Los/las usuarios/as pueden configurar la espera para ignorar tipos específicos de excepciones mientras esperan, como NoSuchElementExcepction al buscar un elemento de la página.

Locators o localizadores


Formas de identificar uno o más elementos específicos del DOM.

Un localizador o locator es una forma de identificar elementos en una página. Es el argumento pasado a los métodos de Finding Element.

Localizadores Tradicionales


Selenium provee soporte para las siguientes ocho estrategias de localización en WebDriver:

Localizador Descripción
Nombre de clase (class name) Localiza elementos cuyo nombre de clase contiene el valor buscado (no permite nombres de clase compuestos).

Selector de CSS

(css selector)

 Localiza elementos que coincidan con un selector CSS.
Identificador (id) Localiza elementos cuyo atributo ID coincida con el valor buscado.
Nombre (name) Localiza elementos cuyo atributo NAME coincida con el valor buscado.

Texto del enlace

(link text)

 Localiza elementos de anclaje cuyo texto visible coincide con el valor de búsqueda
Texto parcial del enlace (partial link text) Localiza elementos de anclaje cuyo texto visible contiene el valor de búsqueda. Si coinciden varios elementos, solo se seleccionará el primero.
Nombre de etiqueta (tag name) Localiza elementos cuyo nombre de etiqueta coincida con el valor buscado.
Xpath Localiza elementos que coincidan con una expresión XPath.

Localizadores Relativos


Selenium 4 introduce Relative Locators o localizadores relativos (anteriormente llamados Friendly Locators). Estos localizadores son útiles cuando no es sencillo construir un localizador para el elemento deseado, pero simple de describir espacialmente dónde el elemento está en relación con otro elemento que tiene un localizador de fácil construcción.

Analizaremos a continuación cómo funcionan.



Cómo funciona


Selenium utiliza la función getBoundingClientRect() de JavaScript para determinar el tamaño y posición de los elementos en la página y emplea esta información para localizar elementos vecinos.

Los métodos de localizador relativo pueden tomar como argumento para el punto de origen ya sea una referencia de elemento previamente ubicada u otro localizador. En los siguientes ejemplos estaremos utilizando únicamente localizadores, pero podríamos cambiar el localizador en el método final por un elemento objeto y funcionaría de la misma manera.

Consideremos el siguiente ejemplo para entender los localizadores relativos.

Localizadores Relativos Disponibles

  • Above o Arriba

Si el elemento del campo de texto del correo electrónico no es fácilmente identificable por alguna razón, pero el elemento del campo de texto de la contraseña sí lo es, podemos ubicar el elemento del campo de texto usando el hecho de que es un elemento de "entrada" "arriba" del elemento de la contraseña.

var emailLocator = RelativeBy.WithLocator(By.TagName("input")).Above(By.Id("password"));

  • Below o Debajo

Si el elemento del campo de texto de la contraseña no es fácilmente identificable por alguna razón, pero el elemento del campo de texto del correo electrónico sí lo es, podemos ubicar el elemento del campo de texto usando el hecho de que es un elemento de "entrada" "debajo" del elemento del correo electrónico.

var passwordLocator = RelativeBy.WithLocator(By.TagName("input")).Below(By.Id("email"));

  • Left of o A la izquierda de

Si el botón de cancelación no es fácilmente identificable por alguna razón, pero el elemento del botón de envío sí lo es, podemos ubicar el elemento del botón de cancelación usando el hecho de que es un elemento de "botón" a la "izquierda" del elemento de envío.

var cancelLocator = RelativeBy.WithLocator(By.TagName("button")).LeftOf(By.Id("submit"));

  • Right of o A la derecha de

Si el botón de envío no es fácilmente identificable por algún motivo, pero el elemento del botón de cancelación sí lo es, podemos ubicar el elemento del botón de envío utilizando el hecho de que es un elemento de "botón" "a la derecha" del elemento de cancelación.

var submitLocator = RelativeBy.WithLocator(By.tagName("button")).RightOf(By.Id("cancel"));

  • Near o Cercano

Si el posicionamiento relativo no es obvio o varía según el tamaño de la ventana, puede usar el método cercano para identificar un elemento que esté a una distancia máxima de 50 px del localizador proporcionado. Un gran caso de uso para esto es trabajar con un elemento de formulario que no tiene un localizador fácil de construir, pero su elemento de etiqueta de entrada asociado sí lo tiene.

var emailLocator = 

RelativeBy.WithLocator(By.tagName("input")).Near(By.Id("lbl-email"));

  • Encadenando localizadores relativos

También puede encadenar localizadores si es necesario. A veces, el elemento se identifica más fácilmente si se encuentra arriba/abajo de un elemento y a la derecha/izquierda de otro.

var submitLocator = RelativeBy.WithLocator(By.tagName("button")).Below(By.Id("email")).RightOf(By.Id("cancel"));

Ubicar los elementos en función de los valores de ubicación proporcionados.

Uno de los aspectos fundamentales de usar Selenium es obtener referencias de elementos para trabajar. Selenium ofrece un número de estrategias de localización incluidas para identificar unívocamente a un elemento. Hay varias formas de utilizar localizadores en escenarios avanzados.

Primer elemento coincidente


Muchos localizadores coincidirán con múltiples elementos en la página. El método singular find element retornará una referencia al primer elemento encontrado en un contexto dado.

Evaluando todo el DOM


Cuando el método findElement es llamado en la instancia del controlador, retorna una referencia al primer elemento en el DOM que coincida con el localizador proporcionado. Este valor puede ser almacenado y utilizado para futuras acciones de elementos. En nuestro ejemplo de HTML, hay dos elementos que tienen el nombre de clase “tomatoes”, por tanto el método retornará el elemento en la lista “vegetales”.

var vegetable = driver.FindElement(By.ClassName("tomatoes"));

Evaluando un subconjunto del DOM


En lugar de encontrar un localizador único en todo el DOM, suele ser útil limitar la búsqueda al ámbito de otro elemento localizado. En el ejemplo anterior, hay dos elementos con un nombre de clase de "tomates" y es un poco más difícil obtener la referencia para el segundo.

Una solución es ubicar un elemento con un atributo único que sea un ancestro del elemento deseado y no un ancestro del elemento no deseado, para luego llamar a findElement en ese objeto:

IWebElement fruits = driver.FindElement(By.Id("fruits"));
IWebElement fruit = fruits.FindElement(By.ClassName("tomatoes"));

Localizador optimizado


Es posible que una búsqueda anidada no sea la estrategia de ubicación más efectiva, ya que requiere que se emitan dos comandos separados para el navegador.

Para mejorar ligeramente el rendimiento, podemos usar CSS o XPath para encontrar este elemento en un solo comando. El siguiente ejemplo emplea un selector CSS.

var fruit = driver.FindElement(By.CssSelector("#fruits .tomatoes"));



Todos los elementos coincidentes


Hay varios casos de uso para necesitar obtener referencias a todos los elementos que coincidan con un localizador, en lugar de solo el primero. Los métodos de búsqueda de elementos plurales devuelven una colección de referencias de elementos. Si no hay coincidencias, se devuelve una lista vacía. En este caso, las referencias a todos los elementos de la lista de frutas y verduras se devolverán en una colección.

IReadOnlyList<IWebElement>plants = driver.FindElements(By.TagName("li"));

Obtener elemento


A menudo, obtenemos una colección de elementos pero queremos trabajar con un elemento específico, lo que significa que necesitamos iterar sobre la colección e identificar el que deseamos:

using OpenQA.Selenium;
using OpenQA.Selenium.Firefox;
using System.Collections.Generic;
namespace FindElementsExample {
 class FindElementsExample {
  public static void Main(string[] args) {
   IWebDriver driver = new FirefoxDriver();
   try {
    // Navigate to Url
    driver.Navigate().GoToUrl("https://example.com");
    // Get all the elements available with tag name 'p'
    IList < IWebElement > elements =driver.FindElements(By.TagName("p"));
    foreach(IWebElement e in elements) {
     System.Console.WriteLine(e.Text);
    }
   } finally {
    driver.Quit();
   }
  }
 }
}

Interactions o interacciones


Hay cinco comandos básicos que pueden ser ejecutados en un elemento:

Click (aplica a cualquier elemento).

Send keys (enviar teclas, aplica únicamente a campos de texto y elementos de contenido editable).

Clear (limpiar, aplica únicamente a campos de texto y elementos de contenido editable).

Submit (enviar, aplica a elementos de formulario).

Select (seleccionar).

A continuación explicaremos cada uno de ellos.



Comandos básicos


Click


El comando de click del elemento se ejecuta en el centro del elemento. Si el centro del elemento está oscurecido por algún motivo, Selenium devolverá un error de interceptación de click en el elemento.

Send Keys o Enviar teclas


El comando de envío de teclas del elemento escribe las teclas proporcionadas en un elemento editable. Normalmente, esto significa que un elemento es un elemento de entrada de un formulario con un tipo de texto o un elemento con un atributo de contenido editable. Si no es editable, se devuelve un error de estado de elemento no válido.

using (var driver = new FirefoxDriver())
{
  // Navigate to Url
  driver.Navigate().GoToUrl("https://google.com");
  // Enter "webdriver" text and perform "ENTER" keyboard action
  driver.FindElement(By.Name("q")).SendKeys("webdriver" + Keys.Enter);
}

El comando limpiar elemento restablece el contenido de un elemento. Esto requiere que un elemento sea editable y reiniciable. Normalmente, esto significa que un elemento es un elemento de entrada de un formulario con un tipo de texto o un elemento con un atributo de contenido editable. Information o información


La información de un elemento, o su information como se le conoce en el rubro tecnológico, es lo que podemos aprender de un elemento. Hay una serie de detalles que se pueden consultar sobre un elemento específico.

IsDisplayed


Este método se utiliza para comprobar si el elemento conectado se muestra en una página web. Devuelve un valor booleano, verdadero si el elemento conectado se muestra en el contexto de navegación actual; de lo contrario, devuelve falso.

Esta funcionalidad se menciona, pero no se define en la especificación w3c debido a la imposibilidad de cubrir todas las condiciones potenciales. Como tal, Selenium no puede esperar que los controladores implementen esta funcionalidad directamente y ahora se basa en la ejecución directa de una gran función de JavaScript. Esta función realiza muchas aproximaciones sobre la naturaleza y la relación de un elemento en el árbol para devolver un valor.

//Navigate to the url
driver.Url = "https://www.selenium.dev/selenium/web/inputs.html";
//Get boolean
            value for is element display
Boolean is_email_visible = driver.FindElement(By.Name(
            "email_input")).Displayed;

IsEnabled


Este método se utiliza para verificar si el elemento conectado está habilitado o deshabilitado en una página web. Devuelve un valor booleano, verdadero si el elemento conectado está habilitado en el contexto de navegación actual; de lo contrario, devuelve falso.

// Navigate to Url
driver.Navigate().GoToUrl("https://google.com");
// Store the WebElement
IWebElement element = driver.FindElement(By.Name("btnK"));
// Prints true if element is enabled else returns false
System.Console.WriteLine(element.Enabled);


IsSelected


Este método determina si el elemento al que se hace referencia está seleccionado o no. Este método se usa ampliamente en casillas de verificación, botones radiales, elementos de entrada y elementos de opción.

Devuelve un valor booleano, verdadero si el elemento al que se hace referencia está seleccionado en el contexto de navegación actual; de lo contrario, devuelve falso.

// Navigate to Url
driver.Navigate().GoToUrl("https://the-internet.herokuapp.com/checkboxes");
// Returns true if element ins checked else returns false
bool value = 
driver.FindElement(By.CssSelector("input[type='checkbox']:last-of-type")).Selected;



TagName


Se utiliza para obtener el nombre de etiqueta del elemento al que se hace referencia que tiene el foco en el contexto de navegación actual.

// Navigate to Url
driver.Navigate().GoToUrl("https://www.example.com");
// Returns TagName of the element
string attr = driver.FindElement(By.CssSelector("h1")).TagName;
