Introducción

En la semana 1 vimos que los niveles de pruebas están asociados al qué se prueba, pero considerando el sistema como un conjunto de partes. Cada nivel de pruebas se enfoca en un nivel de composición de partes del sistema. Vimos que existen cuatro niveles de pruebas: unitaria, integración, sistema, aceptación.

Recorré los contenidos de la semana para saber más sobre los niveles superiores de testing.



Pruebas de Integración


Este nivel se enfoca en la combinación de diferentes componentes. Se centra en la interacción de los componentes del software y en buscar errores cuando unimos las partes que hemos desarrollado. Estas pruebas no son tan rápidas como las pruebas de unidad porque requiere construir una versión del sistema que incluye todos los componentes a probar.

Estas pruebas también pueden ser realizadas por los/las desarrolladores/as que están escribiendo el código y también pueden ser automatizadas. Aunque, en equipos de trabajo más grandes suele haber un equipo especializado en pruebas encargado de estas pruebas de integración.

Las pruebas de integración son tan imprescindibles como las unitarias. El hecho de haber comprobado todos los componentes con resultado satisfactorio, no asegura el correcto funcionamiento entre ellos.

Para realizar este tipo de pruebas, se hace uso de objetos que simulen en muchos casos los datos que contenga la aplicación, como por ejemplo: Stubs, drivers, subsistemas, bases de datos, API’s y microservicios.

Es importante también destacar, en este tipo de pruebas, la importancia de la descripción del error detectado, ya que estos defectos se localizan sobre elementos creados por varios/as desarrolladores/as, quienes únicamente conocerán a fondo la parte del código desarrollada por ellos/as mismos/as.



¿Por qué hacemos pruebas de integración?


A pesar de haber probado mediante Unit Tests cada componente del software, los defectos y errores continuarán existiendo por múltiples motivos:

Un módulo o componente, por lo general está diseñado por un/a desarrollador/a de software en particular, cuyo entendimiento del dominio y lógica de programación puede ser distinta a la de otros/as programadores/as. Con las pruebas de integración nos aseguramos que los módulos de software funcionen al ser unidos a pesar de contener lógica diferente.

A la hora de desarrollar un módulo o un componente, es altamente probable encontrarnos con cambios a los requerimientos por parte de nuestros/as clientes o aquellos/as que nos han pedido el desarrollo. Estos nuevos requerimientos puede que no sean probados con Unit Tests por lo que la integración (y por tanto pruebas de integración) se vuelven necesarias.

Las interfaces que el software tiene con la base de datos pueden ser erróneas. Es posible que la conexión en sí o el manejo de las tablas de base de datos esté mal programado y contenga errores. Si no se realizan pruebas de integración nunca se encontrarán estos errores.

Las interfaces con hardware externo podrían estar mal programadas.

Un manejo inadecuado de las excepciones puede traer muchos problemas en la integración. Es normal que un/a desarrollador/a no maneje adecuadamente una excepción porque el problema de no manejarla excede su trabajo o sus tareas, esto quiere decir que el software puede darnos un error mortal en tiempo de ejecución. Con las pruebas de integración podemos encontrar estas excepciones mal manejadas y ver cómo afectan al resto del software.



Objetivos de las pruebas de integración


El objetivo principal de las pruebas de integración es el de encontrar errores en la unión de componentes de software que han sido desarrollados por diferentes programadores/as. Sin embargo, podemos desglosar este gran objetivo en varios objetivos más pequeños:

Encontrar errores en la seguridad de la aplicación, ayudando a reducir el riesgo de seguridad de la misma.

Encontrar errores en las interfaces del software, ayudándonos a verificar que el comportamiento del software entre las distintas interfaces sea el esperado.

Asegurar la confianza en la calidad de las interfaces. Hemos establecido al comenzar la materia que todo software tiene defectos y errores. Muchas veces, lo que decide si un software es usado o no, es la confianza que sus usuarios/as (o posibles usuarios/as) le tienen al software. Si el software se presenta como algo con muchos errores, entonces habrá desconfianza y no será usado. Las pruebas de unidad nos ayudan a elevar esa confianza, mostrando que no tenemos tantos errores visibles en la integración.

Encontrar defectos en las comunicaciones entre servicios y componentes del mismo software.

Prevenir la aparición de defectos en fases posteriores de pruebas.



Desafíos de las pruebas de integración


Existen muchos desafíos en cuanto a las pruebas de integración que harán que nuestra tarea sea más difícil. Es importante considerarlos y ver cómo aplican a cada proyecto de software en el que trabajamos para elegir la estrategia de integración y de pruebas adecuada.

Hacer pruebas de integración quiere decir probar dos o más sistemas o subsistemas integrados para asegurarnos de que el mismo funcione correctamente. No sólo deben probarse las uniones de la integración, sino que también se debe hacer pruebas exhaustivas considerando el entorno, para asegurarnos de que el sistema puede funcionar correctamente. Existen muchos caminos y permutaciones posibles.

Manejar las pruebas de integración se vuelve algo complejo debido a factores involucrados como la base de datos, plataforma de trabajo, entorno de trabajo, etc.

Cuando integramos un nuevo sistema con un sistema de tipo legacy (un sistema viejo que lleva siendo usado por muchos años) se requieren muchos cambios y esfuerzos de pruebas. Lo mismo aplica cuando integramos 2 sistemas de tipo legacy entre sí.

Integrar dos sistemas diferentes de dos organizaciones diferentes es un gran desafío en cuanto al impacto que tendrán entre ellos si algún cambio debe ser realizado y afecta el funcionamiento interno del sistema.



Ejemplo de Casos de prueba de integración

La prioridad se les da a las uniones entre módulos más que a las funciones unitarias que ya han sido probadas.

Consideremos, por ejemplo, un software con 3 módulos: una página de Inicio de sesión, un módulo de Casilla de correos y un módulo para borrar E-mails. Estrategias de pruebas de integración


Las pruebas de integración están divididas en 2 tipos de estrategias de pruebas de integración:

Pruebas de integración incrementales.

Pruebas de integración no incrementales.

Estrategias de pruebas de integración incrementales


En el enfoque de pruebas incrementales, los módulos se van agregando o integrando uno por uno según la necesidad de pruebas. Los módulos seleccionados deben estar relacionados entre sí de alguna manera lógica. Generalmente, dos o más módulos son integrados para determinar qué tan correcto es su funcionamiento en conjunto. Este proceso se continúa de a poco hasta que se han integrado todos los módulos o componentes y se han encontrado todos los errores aparentes.

Esto quiere decir que tenemos una fuerte relación entre módulos dependientes que debemos probar.

Este enfoque tiene 3 estrategias:

De arriba hacia abajo o top-down.

De abajo hacia arriba o bottom-up.

Híbrido o sándwich.

Analizaremos cada una a continuación.



Top-down


En esta estrategia probamos primero los componentes de más alto nivel y vamos avanzando hacia los de más bajo nivel a medida que el código va pasando las pruebas. Esto significa que primero desarrollamos las interfaces entre subsistemas, las que retornan valores que son controlados y posteriormente se reemplaza con el código real, esto permite probar el flujo completo en niveles superiores antes y luego ir a niveles inferiores.

Fallas graves en el diseño y desarrollo pueden ser detectadas y arregladas de manera temprana ya que los módulos críticos se prueban primero. En este método agregaremos los módulos de manera incremental uno a uno y controlaremos el flujo de datos en el mismo orden en que los vamos agregando.

Con esta estrategia debemos asegurarnos de ir agregando, uno a uno, el módulo hijo (o Child) del anterior de la misma forma como en la figura 10.3 Child C es un hijo de Child B y así sucesivamente. La idea de esto es probar la integración entre módulos dependientes entre sí.

Esta estrategia tiene ventajas y desventajas que listamos a continuación:

Ventajas Desventajas
  • La localización de errores y problemas se hace más fácil.
  • Tenemos la posibilidad de obtener prototipos tempranos del software.
  • Los módulos críticos se prueban de manera prioritaria. Es decir que las fallas más graves del desarrollo se arreglan primero.
  • Se requiere una gran cantidad de stubs para simular los componentes hijo que no poseemos, haciendo que las pruebas sean bastante complicadas.
  • Los módulos de más bajo nivel se prueban de manera inadecuada ya que puede haber un acarreo de errores de integraciones previas.



Bottom-up


En esta estrategia probamos primero los componentes de más bajo nivel y vamos avanzando hacia los de más alto nivel a medida que el código va pasando las pruebas. Los módulos críticos de alto nivel se prueban al final, por lo que podemos causar algún defecto. También podemos decir que vamos integrando los módulos desde abajo hacia arriba para revisar y probar el flujo de información en ese mismo orden.

Con esta estrategia debemos asegurarnos de ir agregando, uno a uno, el módulo padre (o Parent) del anterior de la misma forma como en la figura 10.5 donde Child A es un hijo de Child B y así sucesivamente. La idea de esto es probar la integración entre módulos dependientes entre sí.

Esta estrategia también tiene ventajas y desventajas:

Ventajas Desventajas
  • La localización de errores y problemas se hace más fácil.
  • No debemos esperar a que se desarrollen todos los módulos, ahorrándonos tiempo.
  • Los módulos críticos (los de alto nivel) que controlan el flujo de la aplicación se integran y prueban al final, lo que hace que sean propenso a tener más errores.
  • No es posible tener un prototipo temprano en caso de necesitarlo.



Sándwich testing


En esta estrategia combinamos las estrategias Top-Down y Bottom-Up para realizar la integración y las pruebas. Utilizamos ambas estrategias al mismo tiempo para probar los módulos de alto nivel con los módulos de bajo nivel y para probar los módulos de bajo nivel con los módulos de alto nivel de manera simultánea. Con esta estrategia hay menos probabilidad de ocasionar errores porque se prueba cada una de las interfaces de cada módulo.

Las ventajas y desventajas de esta estrategia son:

Ventajas Desventajas
  • Es la estrategia más rápida.
  • Permite realizar pruebas completas a todos los módulos.
  • Provee ventajas de los métodos Top-Down y Bottom-Up.
  • Es un método complicado de usar.
  • Requiere un mayor grado de concentración y atención ya que requiere realizar una integración en ambas direcciones en simultáneo.



Estrategias de pruebas de integración no incrementales


En la estrategia de pruebas no incrementales probamos todo el software junto una vez que ya se la ha integrado. Este enfoque se utiliza cuando tenemos un flujo de datos muy complejo en el cual es difícil ver cuál es el padre y cuál es el hijo. En estos casos crearemos la información en cualquiera de los módulos y vamos probando la integración en todos los otros módulos. La estrategia típicamente usada con el enfoque no incremental es la de Big Bang.

Estrategia Big Bang


En esta estrategia integramos todos los módulos al comienzo de una sola vez. Luego probamos los flujos de información e integraciones de todo el software. Es ideal para softwares pequeños, si lo utilizamos para softwares más grandes la identificación exacta del origen de un problema o defecto puede ser muy difícil y compleja.

Las ventajas y desventajas son:

Ventajas Desventajas
  • Es extremadamente conveniente para sistemas pequeños.
  • Identificar el origen de los defectos que encontramos es difícil ya que no sabemos exactamente qué integración lo causó.
  • Problemas en módulos e integraciones pequeñas son fáciles de ignorar.
  • Se tiene menos tiempo para probar la integración ya que se debe esperar a que todo el software esté completado para comenzar.
  • Es probable que olvidemos probar.



Ejemplo de Big bang testing


Podríamos considerar como ejemplo un software de E-mails, similar a Gmail, en donde hacemos clic en el Inbox o Casilla de correos y la aplicación nos muestra la Casilla de correos. Si consideramos la pantalla de inicio del software como la que vemos en la figura siguiente, notamos que necesitamos hacer pruebas de integración no incrementales ya que no tenemos módulos padres ni módulos hijos.



Pruebas de sistema


En este nivel se prueba el sistema como un todo. Las pruebas de sistema se pueden hacer de forma granular, es decir casos de uso/funcionalidades individuales, o combinando flujos completos de inicio a fin. Si seguimos los principios de las pruebas, este nivel debería hacerlo alguien diferente a quien desarrolló el software. El propósito de estas pruebas es encontrar errores evaluando las especificaciones del sistema de punta a punta. Algunas de estas pruebas pueden ser automatizadas y otras de ellas no.

También se intenta determinar si el sistema en su globalidad opera satisfactoriamente. Con esto no nos referimos solo a la funcionalidad sino también a cuestiones de recuperación de fallas, seguridad, pruebas de estrés y pruebas de performance.

El entorno de pre-producción, en el que se hacen las pruebas de sistema, debe ser lo más parecido posible al entorno en el cual se usará el software en el día a día. Si los entornos son muy diferentes es posible que se encuentren errores que en su uso normal no existirían, o aun peor, que no se encuentren errores que si se encontraran en su entorno normal de uso. Si bien se aspira a esta similitud, muchas veces no es posible anticipar las necesidades ni condiciones de uso reales. 

Es posible hacer un software de ventas online que considere que miles de personas accederán al sitio, pero si el negocio tiene un Hot Sale o descuento por fiestas, el tráfico crecerá de una manera que no se puede anticipar y colapsara el sistema.

Estos casos no necesitan ser tan extremos como el caso de un Hot Sale, muchas veces ocurrirá que no encontraremos defectos que sí o sí encontraremos en producción.



Tipos de pruebas de sistema


Existen más de 50 tipos de pruebas de sistemas. Sin embargo, hoy en día las organizaciones se concentran en realizar 7 tipos puntuales:

Pruebas funcionales.

Pruebas de regresión.

Pruebas de carga.

Pruebas de recuperación

Pruebas de migración.

Pruebas de usabilidad.

Pruebas de hardware/software.



Pruebas funcionales


La prueba funcional es un tipo de prueba que se realiza para encontrar todos los errores de un software a nivel de funcionalidad, teniendo en cuenta desde la interfaz de usuario hasta el impacto en la base de datos. 

Ya hemos explorado la importancia de las pruebas funcionales y las diferentes estrategias y métodos de la misma en las semanas 2 y 3. Debido a que ya se ha explorado la importancia de estas pruebas y su funcionamiento no se repetirán los contenidos esta semana.



Pruebas de regresión


La prueba de regresión es un tipo de prueba que se realiza para verificar que un cambio de código en el software no afecta la funcionalidad existente del producto. Esto es para asegurarse de que el producto funcione bien con nuevas funciones, correcciones de errores o cualquier cambio en la función existente. Los casos de prueba ejecutados previamente se vuelven a ejecutar para verificar el impacto del cambio. Queremos verificar si nuevos errores han sido introducidos ante estos cambios.

Esta prueba se puede realizar en una nueva compilación cuando hay un cambio significativo en la funcionalidad original, incluso en una sola corrección de error.

Regresión significa volver a probar las partes sin cambios de la aplicación.

La mayoría de las organizaciones hoy en día intentan que estos casos de prueba estén automatizados ya que se requiere que los casos de prueba se ejecuten una y otra vez, y ejecutar los mismos casos de prueba una y otra vez de forma manual, también requiere mucho tiempo y es tedioso.

Por ejemplo, consideremos un producto X, en el que una de las funciones es activar la confirmación, la aceptación y los E-Mails enviados cuando se hace clic en los botones Confirmar, Aceptar y Despachar.

Se produce algún problema en el E-mail de confirmación y, para solucionarlo, se realizan algunos cambios de código. En este caso, no sólo se deben probar los E-mails de confirmación, sino que también se deben probar los E-mails de aceptación y envío para asegurarse de que el cambio en el código no los haya afectado.

La prueba de regresión no depende de ningún lenguaje de programación como Java, C ++, C #, etc. Es un método de prueba que se utiliza para probar el producto en busca de modificaciones o actualizaciones. Verifica que cualquier modificación en un producto no afecta los módulos existentes del producto. Verifica que los errores se hayan solucionado y que las funciones recién agregadas no hayan creado ningún problema en la versión de trabajo anterior del software.

Los testers realizan pruebas funcionales cuando hay una nueva construcción disponible para verificación. La intención de esta prueba es verificar los cambios realizados en la funcionalidad existente y también en la funcionalidad recién agregada.

Se listan, a continuación, las ventajas y desventajas de las pruebas de regresión:

Ventajas Desventajas
  • Mejora la calidad del Producto.
  • Garantiza que cualquier corrección de error o mejora que se realice no afecte la funcionalidad existente del Producto.
  • Se pueden utilizar herramientas de automatización para esta prueba.
  • Se asegura de que los problemas que ya están solucionados no vuelvan a ocurrir.
  • También debe hacerse para un pequeño cambio en el código porque incluso un pequeño cambio en el código puede crear problemas en la funcionalidad existente.
  • Si, en caso de que no se utilice la automatización en el Proyecto para esta prueba, será una tarea tediosa y que llevará mucho tiempo ejecutar los casos de prueba una y otra vez.



Herramientas de prueba de regresión automatizadas


La prueba de regresión automatizada es el área de prueba donde podemos automatizar la mayoría de los esfuerzos de prueba. Ejecutamos todos los casos de prueba ejecutados previamente en una nueva compilación.

Esto significa que tenemos un conjunto de casos de prueba disponibles y ejecutar estos casos de prueba manualmente consume mucho tiempo. Conocemos los resultados esperados, por lo que la automatización de estos casos de prueba ahorra tiempo y es un método de prueba de regresión eficiente. El grado de automatización depende del número de casos de prueba que seguirán siendo aplicables con el tiempo.

Si los casos de prueba varían de vez en cuando, el alcance de la aplicación sigue aumentando y la automatización del procedimiento de regresión será una pérdida de tiempo.

La mayoría de las herramientas de prueba de regresión son de tipo grabación y reproducción. Registrará los casos de prueba navegando a través de la interfaz de usuario verificará si los resultados esperados están llegando o no.

Algunas de las herramientas más usadas son:

  • - Selenium
  • - AdventNet QEngine
  • - vTest
  • - agua
  • - actiWate
  • - SilkTest
  • - TimeShiftX

Agregar y actualizar casos de prueba de regresión en un conjunto de pruebas de automatización es una tarea engorrosa. Al seleccionar una herramienta de automatización para las pruebas de regresión, debe verificar si la herramienta le permite agregar o actualizar los casos de prueba fácilmente.

En la mayoría de los casos, necesitamos actualizar los casos de prueba de regresión automatizados con frecuencia debido a los cambios frecuentes en el sistema.



¿Por qué la prueba de regresión?


La regresión se inicia cuando un/a programador/a corrige cualquier error o agrega un nuevo código para una nueva funcionalidad al sistema. Puede haber muchas dependencias en la funcionalidad recién agregada y existente.

Es una medida de calidad comprobar si el nuevo código cumple con el antiguo para que el código no modificado no se vea afectado.

La mayoría de las veces, el equipo de pruebas tiene la tarea de verificar los cambios de última hora en el sistema. En tal situación, es necesario probar sólo el área de aplicación afectada para completar el proceso de prueba a tiempo cubriendo todos los aspectos principales del sistema.

Esta prueba es muy importante cuando se agrega un cambio o mejora en la aplicación. La nueva funcionalidad no debería afectar negativamente al código probado existente. Se requiere una regresión para encontrar los errores que ocurrieron debido a un cambio en el código. 

Si no se realiza esta prueba, el producto puede tener problemas críticos en producción y eso puede causar problemas a los/las usuarios/as.



Técnicas de prueba de regresión


Existen 4 técnicas principales de pruebas de regresión:

Vuelva a probar todo: Como sugiere el propio nombre, todos los casos de prueba en el conjunto de pruebas se vuelven a ejecutar para garantizar que no haya errores que hayan ocurrido debido a un cambio en el código. Este es un método costoso ya que requiere más tiempo y recursos en comparación con las otras técnicas.

Selección de prueba de regresión: En este método, los casos de prueba se seleccionan del conjunto de pruebas para volver a ejecutarlos. No se vuelve a ejecutar toda la suite. La selección de casos de prueba se realiza sobre la base del cambio de código en el módulo. Los casos de prueba se dividen en dos categorías, uno es casos de prueba reutilizables y otro es casos de prueba obsoletos. Los casos de prueba reutilizables se pueden usar en futuros ciclos de regresión, mientras que los obsoletos no se usan en los próximos ciclos de regresión.

Priorización de casos de prueba: Los casos de prueba con prioridad alta se ejecutan primero que los de prioridad media y baja. La prioridad del caso de prueba depende de su criticidad y su impacto en el producto y también de la funcionalidad del producto que se utiliza con más frecuencia.

Híbrido: La técnica híbrida es una combinación de selección de prueba de regresión y priorización de casos de prueba. En lugar de seleccionar todo el conjunto de pruebas, seleccione sólo los casos de prueba que se vuelven a ejecutar según su prioridad. 



Pruebas de carga


Las pruebas de carga y rendimiento son los procesos en los cuales introducimos cargas de trabajo a un sistema para determinar su comportamiento en condiciones de carga normal o carga máxima. 

Estas pruebas ayudan a identificar la capacidad máxima de operaciones de una solicitud, detectar cualquier cuello de botella y determinar qué elementos están causando la degradación del sistema. También pueden servir para investigar, medir, validar o verificar otros atributos de calidad del sistema, como la escalabilidad, la confiabilidad y el uso de los recursos.

Se realizan normalmente como una serie de pasos en los que el sistema de pruebas inicia una cantidad de usuarios/as simultáneos/as que se sabe que son compatibles con la infraestructura. Esto establece un conjunto de líneas base de datos de rendimiento a los que hacer referencia a medida que aumenta el número de usuarios/as simultáneos/as a lo largo de la prueba. La prueba de rendimiento puede ayudar a determinar varias líneas de base diferentes, como la velocidad de conexión promedio, la latencia promedio y el tiempo promedio para descargar un archivo de tamaño fijo y más.

Una vez que se conocen los valores de rendimiento de línea base, el número de usuarios/as aumenta a un número que se espera de forma realista para visitar el software durante un período de muestra. La prueba a menudo se ejecutará en ese número estático de usuarios/as durante varios minutos para verificar la estabilidad del software después que el sistema se estabilice en el nuevo nivel de carga.

Si al hacer las pruebas de carga se está generando carga esperada dentro de la capacidad del sistema, los tiempos de respuesta de la aplicación deben permanecer dentro de las directrices generadas. Si empuja el sistema más allá de estos límites, se mueve al ámbito de las pruebas de estrés, causando presión deliberada en el sistema para identificar los componentes que fallan.



Pruebas de recuperación


Las pruebas de recuperación de un sistema se realizan para confirmar la fiabilidad, confianza y responsabilidad del software así como también la capacidad para confiar en las medidas de recuperación del sistema.

El software debería poder recuperarse sin problemas de los posibles errores o problemas que puedan surgir.



Pruebas de migración


La prueba de migración es un proceso de verificación de la migración del sistema heredado al nuevo sistema con un mínimo de interrupción/tiempo de inactividad, con integridad de los datos y sin pérdida de datos, al tiempo que garantiza que todos los aspectos funcionales y no funcionales especificados de la aplicación se cumplan después de migración.

La migración de la aplicación a un nuevo sistema puede ser por diversas razones, consolidación del sistema, tecnología obsoleta, optimización o cualquier otra razón.

Por lo tanto, si bien el sistema en uso debe migrar a un nuevo sistema, es esencial garantizar los siguientes puntos:

Cualquier tipo de interrupción/inconveniente causado al/la usuario/a debido a la migración debe evitarse/minimizarse. Por ejemplo: tiempo de inactividad, pérdida de datos.

Es necesario asegurarse de que el/la usuario/a pueda seguir utilizando todas las funciones del software causando un daño mínimo o nulo durante la migración. Por ejemplo: cambio en la funcionalidad, eliminación de una funcionalidad en particular.

También es importante anticipar y descartar todos los posibles fallos / obstáculos que puedan ocurrir durante la migración real del sistema en producción.

Para garantizar una migración fluida del sistema en producción mediante la eliminación de esos defectos, es esencial realizar pruebas de migración de forma previa a la misma.



¿Cuándo se requiere esta prueba?


Las pruebas deben realizarse antes y después de la migración. Podemos decir que hay 3 fases en cualquier migración de datos:

Pruebas previas a la migración.

Prueba de migración.

Pruebas posteriores a la migración.

Además de lo anterior, también se ejecutan las siguientes pruebas como parte de toda la actividad de migración.

Verificación de compatibilidad con versiones anteriores.

Prueba de reversión.

Antes de realizar esta prueba, es esencial que cualquier tester comprenda claramente los siguientes puntos:

Los cambios que ocurren como parte del nuevo sistema (servidor, interfaz, base de datos, esquema, flujo de datos, funcionalidad, etc.)

Comprender la estrategia de migración real establecida por el equipo. Cómo ocurre la migración, los cambios paso a paso que ocurren en el backend del sistema y los scripts responsables de estos cambios.


Por lo tanto, es esencial hacer un estudio exhaustivo del sistema antiguo y el nuevo y, en consecuencia, planificar y diseñar los casos de prueba y los escenarios de prueba que se cubrirán como parte de las fases de prueba anteriores y preparar la estrategia de prueba.



Fase 1 de la migración


Analicemos las 3 fases de migración de datos, mencionadas en el título anterior.

Fase 1: Pruebas previas a la migración


Antes de migrar los datos, se realiza un conjunto de actividades de prueba como parte de la fase de prueba previa a la migración. Esto puede ignorarse en aplicaciones más simples. Pero cuando se van a migrar aplicaciones complejas, las actividades previas a la migración son imprescindibles.

Algunas de las acciones más comunes a realizar durante esta fase (pero no las únicas) son:

Establecer un alcance claro de los datos: qué datos deben incluirse, qué datos deben excluirse, qué datos necesitan transformaciones / conversiones, etc.

Realizar el mapeo de datos entre la aplicación heredada y la nueva; para cada tipo de datos en la aplicación heredada, compare su tipo relevante en la nueva aplicación y luego asignarlos: mapeo de nivel superior.

Si la nueva aplicación tiene el campo que es obligatorio en ella, pero no es el caso en el legado, asegurarse que el software viejo no tenga ese campo como nulo. - Mapeo de nivel inferior.

Estudiar el esquema de datos de la nueva aplicación: nombres de campo, tipos, valores mínimos y máximos, longitud, campos obligatorios, validaciones de nivel de campo, etc., claramente.

Estudiar las interfaces en la nueva aplicación y sus conexiones. Los datos que fluyen en la interfaz deben estar altamente protegidos y no romperse.

Preparar casos de prueba, escenarios de prueba y casos de uso para nuevas condiciones en las nuevas aplicaciones.

Ejecutar un conjunto de casos de prueba, escenarios con un conjunto de usuarios/as y mantener los resultados almacenados. Lo mismo debe verificarse después de la migración para garantizar que los datos y la funcionalidad heredados estén intactos.



Fase 2: Prueba de migración


La Guía de migración que es elaborada por el equipo de Migración debe seguirse estrictamente para llevar a cabo la actividad de migración.

Idealmente, la actividad de migración comienza con la copia de seguridad de los datos en la cinta, de modo que, en cualquier momento, se pueda restaurar el sistema heredado.

Los scripts de migración, la guía y otra información relacionada con la migración real deben recopilarse del repositorio de control de versiones para su ejecución.

Anotar el tiempo real necesario para la migración desde el punto de inicio de la migración hasta la restauración exitosa del sistema, es uno de los casos de prueba a ejecutar y, por lo tanto, el 'Tiempo necesario para migrar el sistema' debe registrarse en el informe de prueba final que se entregará como parte de los resultados de la prueba de migración y esta información será útil durante el lanzamiento de la producción. El tiempo de inactividad registrado en el entorno de prueba se extrapola para calcular el tiempo de inactividad aproximado en el sistema activo.

Durante esta prueba, todos los componentes del entorno normalmente serán desactivados y eliminados de la red para llevar a cabo las actividades de migración. Por tanto, es necesario señalar la 'Falta del tiempo' requerido para la prueba de migración. Idealmente, será el mismo que el del tiempo de Migración.



Fase 3: Pruebas posteriores a la migración


Una vez que la aplicación se ha migrado correctamente, las pruebas posteriores a la migración entran en escena.

Aquí, la prueba del sistema de un extremo a otro se realiza en el entorno de prueba. Los testers ejecutan casos de prueba identificados, escenarios de prueba, casos de uso con datos heredados, así como un nuevo conjunto de datos.

Dado que el alcance de las pruebas posteriores a la migración se vuelve muy grande, es ideal segregar las pruebas importantes que deben realizarse primero para calificar que la migración es exitosa y luego realizar las restantes más tarde.

También es aconsejable automatizar los casos de prueba funcionales de extremo a extremo y otros posibles casos de prueba para que el tiempo de prueba pueda reducirse y los resultados estén disponibles rápidamente.



Pruebas de usabilidad


Las pruebas de usabilidad son un tipo de prueba no funcional que intenta evaluar la comprensibilidad, capacidad de aprendizaje, operabilidad, atractivo y cumplimiento del software.

Las pruebas de usabilidad son para determinar hasta qué punto entendemos el producto de software, es fácil de aprender, fácil de operar y atractivo para los usuarios bajo condiciones y requisitos específicos.

Este tipo de pruebas generalmente lo ejecutan usuarios/as reales y no el equipo de desarrollo. El equipo de desarrollo es quien ha creado el producto y, por lo tanto, no logra encontrar menos defectos relacionados con la experiencia del/la usuario/a.

Las ventajas y desventajas de las pruebas de usabilidad son las siguientes:

Ventajas Desventajas
  • Al ejecutar una prueba de usabilidad, podemos mejorar las fallas del producto, si las encontramos, y rectificarlas antes del lanzamiento del producto. Por lo tanto, la ejecución de pruebas ayudará a mejorar la calidad del producto.
  • Muchas veces las discusiones internas se dirigen hacia debates que pueden resolverse después de realizar pruebas de usabilidad, lo que ayuda a comprender mejor la experiencia del/la usuario/a.
  • A veces se observa que el equipo de desarrollo no detecta algunos defectos que se pueden encontrar ejecutando pruebas de usabilidad.
  • Dado que la retroalimentación proviene directamente del lado del/la usuario/a, resulta beneficioso para la organización para mejorar el producto de acuerdo con el público objetivo.
  • Las pruebas de usabilidad pueden resultar costosas para la organización o equipo que desea implementarlas.



Métodos de prueba de usabilidad


Existen 5 métodos para probar la usabilidad:

Pruebas de pasillo: Este método es uno de los métodos disponibles más eficaces y rentables en comparación con otros. En este método, pocas personas al azar reciben el sitio web o el producto para probar en lugar de profesionales capacitados/as. A medida que personas al azar prueban el servicio sin conocimiento previo del producto, lo prueban de manera más efectiva y brindan resultados más precisos y comentarios honestos para mejorar, si corresponde.

Prueba de usabilidad remota: Como su nombre indica, las pruebas de usabilidad remota las llevan a cabo personas que se encuentran en ubicaciones remotas, es decir, que se encuentran en varias provincias o, a veces, en varios países con fines de prueba. Este tipo de prueba se realiza de forma remota y los problemas se pueden informar si se encuentran. Los comentarios pueden ser registrados y enviados por personas al azar y no por un/a experto/a en tecnología. A veces, las pruebas remotas se ejecutan mediante videoconferencia.

Revisión de expertos: Se le pide al/la experto/a en el campo que pruebe el producto o servicio y proporcione comentarios y luego envíe los resultados. Este tipo de pruebas suele ser caro, ya que la organización necesita contratar a un/a experto/a en tecnología. También se puede realizar una revisión de expertos de forma remota para enviar los resultados. Este tipo de prueba de usabilidad se ejecuta rápidamente y lleva menos tiempo. El/la experto/a encuentra rápidamente las lagunas y detecta las fallas en el producto o servicio. Esta prueba es confiable debido a la experiencia que posee el/la experto/a. La revisión de un/a experto/a vale la pena cuando el producto tiene una característica crítica.

Prueba de prototipos de papel: Este método implica un ensayo de ejecución de la prueba, un boceto a mano, un dibujo de los modelos o prototipos. Discutir el flujo, dibujarlos en papel y considerar todas las posibles entradas, escenarios y condiciones, es el objetivo de este tipo de pruebas. Este es uno de los tipos principales de pruebas que se observa en los proyectos de desarrollo para eliminar los problemas principales. Al realizar pruebas de prototipos en papel, se puede tener más claridad sobre el proceso de ejecución. La prueba del prototipo en papel generalmente se realiza dentro del equipo de trabajo. Por lo tanto, se considera en las primeras etapas del proceso de prueba. Este es un método relativamente más económico de prueba de usabilidad, pero no es la forma más eficaz de probar, ya que a veces lleva más tiempo y existe una mayor posibilidad de que incluso después de la prueba, perdamos algunos problemas.

Pruebas de usabilidad automatizadas: Este método consiste en escribir scripts para la automatización y cuando se activan los scripts, se ejecutan los casos de prueba. Después de la ejecución de la prueba, los resultados se registran y se envían. Para este método de prueba, una organización necesita contratar un recurso que esté familiarizado con la escritura de scripts y la construcción de un marco de automatización. Este es uno de los métodos de prueba más utilizados, ya que no es tan caro como el tipo de prueba de revisión de expertos y no es tan básico como el método de prueba de prototipo de papel.



Pruebas de hardware/software


Estas pruebas se refieren a probar tanto el hardware como el software. Las configuraciones de hardware deben ser compatibles con el software que debe ejecutar para evitar problemas en un entorno de producción.



Pruebas de aceptación


Una vez el proceso de prueba del sistema es completado por el equipo de prueba, todo el software se entrega a los/las clientes y/o usuarios/as para probar su aceptabilidad, es decir, el software debe ser impecable para cumplir con los requisitos funcionales. Además, los flujos de negocios de un extremo a otro se verifican de manera similar a los escenarios en tiempo real. En estas pruebas los/las clientes y usuarios/as nos dirán si están dispuestos a utilizar el software.

Tipos de pruebas de aceptación


Hay varios tipos distintos de pruebas de aceptación:

Prueba de aceptación del usuario (UAT): Quiere evaluar si el Producto está funcionando correctamente para su uso según el/la usuario/a. Los requisitos específicos que los/las usuarios/as finales utilizan con bastante frecuencia se seleccionan principalmente con fines de prueba. Esto también se denomina prueba de usuario final. El término 'Usuario' aquí significa los/las usuarios/as finales a los/las que está destinado el software y, por lo tanto, las pruebas se realizan desde la perspectiva del/la usuario/a final y desde su punto de vista.

Prueba de aceptación comercial (BAT): Esto es para evaluar si el Producto cumple con los objetivos y propósitos comerciales o no. BAT se enfoca principalmente en los beneficios comerciales (finanzas) que son bastante desafiantes debido a las condiciones cambiantes del mercado y tecnologías avanzadas, por lo que la implementación actual puede tener que sufrir cambios que resultan en presupuestos adicionales. Incluso el Producto que pasa los requisitos técnicos puede fallar BAT debido a estas razones.

Prueba de aceptación de contrato (CAT): Este es un contrato que especifica que una vez que el Producto entre en funcionamiento, dentro de un período predeterminado, se debe realizar la prueba de aceptación y debe pasar todos los casos de uso de aceptación. El contrato firmado aquí se denomina Acuerdo de Nivel de Servicio (SLA), que incluye los términos en los que se realizará el pago solo si los servicios del Producto cumplen con todos los requisitos, lo que significa que se cumple el contrato. A veces, este contrato puede suceder antes de que el Producto entre en funcionamiento.

Regulaciones/pruebas de cumplimiento de regulaciones (RAT): Esto es para evaluar si el Producto viola las reglas y regulaciones definidas por el gobierno del país donde se lanza. Esto puede no ser intencional, pero tendrá un impacto negativo en el negocio. Por lo general, el software desarrollado que está destinado a ser lanzado en todo el mundo, tiene que someterse a RAT, ya que diferentes países o regiones tienen diferentes reglas y regulaciones definidas por sus órganos de gobierno. Si se viola alguna de las reglas y regulaciones de cualquier país, entonces ese país o la región específica de ese país no podrá usar el Producto y se considerará una falla. Los/las proveedores/as del producto serán directamente responsables si el producto se libera aunque haya una infracción.

Prueba de aceptación operativa (OAT): Esto es para evaluar la disponibilidad operativa del Producto y es una prueba no funcional. Incluye principalmente pruebas de recuperación, compatibilidad, mantenibilidad, disponibilidad de soporte técnico, confiabilidad, conmutación por error, localización, etc. OAT principalmente asegura la estabilidad del Producto antes de lanzarlo a producción.

Prueba alfa: Esto es para evaluar el Producto en el entorno de prueba por un equipo de testers especializados, generalmente llamados alfa testers. Aquí, los comentarios de los/las testers y sus sugerencias ayudan a mejorar el uso del Producto y también a corregir ciertos errores. Las pruebas se hacen en un entorno controlado.

Prueba beta / Prueba de campo: Esto es para evaluar el Producto exponiéndose a los/las usuarios/as finales reales, generalmente llamados/as beta testers o usuarios/as beta, en su entorno. Se recopilan comentarios continuos de los/las usuarios/as y se solucionan los problemas. Además, esto ayuda a mejorar el Producto para brindar una experiencia de usuario/a más rica. Las pruebas se realizan de manera no controlada, lo que significa que un/a usuario/a no tiene restricciones sobre la forma en que se utiliza el Producto.
