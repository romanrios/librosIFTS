Distribución y despliegue


Nos adentraremos en los complejos procesos de llevar una aplicación desde la fase de desarrollo hasta su pleno funcionamiento en el entorno de producción. Desde la compilación y empaquetado hasta la publicación en las tiendas de aplicaciones, exploraremos cada paso crucial del despliegue de aplicaciones.

No nos detendremos ahí; nos sumergiremos en el poderoso mundo de Git y el control de versiones. Descubriremos la importancia de mantener un historial preciso de los cambios, exploraremos comandos fundamentales de Git y entenderemos la gestión efectiva de ramas con Gitflow.

Además, exploraremos plataformas de almacenamiento clave como GitHub, GitLab y Bitbucket, comprendiendo cómo estas herramientas transforman la colaboración en el desarrollo de software.


¡Preparate para desentrañar los secretos de la implementación en la Play Store de Android, desde la creación de cuentas de desarrollador hasta la generación de archivos APK!

Implementación de Android Play Store


Para implementar una aplicación en la tienda de aplicaciones de Android (Play Store), debemos cumplir previamente con ciertos requisitos:

    Tener una cuenta de desarrollador de Google Play, para crear una cuenta, es necesario pagar una tarifa anual de $... (a determinar).

    Tener un certificado de firma de aplicaciones válido. Este certificado se utiliza para firmar la aplicación y garantizar su autenticidad.

    Tener activo un archivo APK de la aplicación. El archivo APK es el paquete de instalación de la aplicación, que contiene todos los recursos y código necesarios para ejecutarla en un dispositivo Android.

    Proporcionar una descripción detallada y una imagen de portada para la aplicación. Estos elementos se mostrarán en la tienda de aplicaciones y ayudarán a los usuarios a entender qué hace la aplicación y por qué deberían descargarla.

    Elegir una categoría y una serie de etiquetas para la aplicación. Esto ayudará a Play Store a organizar la aplicación en la tienda de aplicaciones y a que los usuarios la encuentren fácilmente.

¿Qué hacer una vez que se cumplen con estos requisitos?

Se puede seguir el proceso de publicación de Google Play para desplegar la aplicación en la tienda de aplicaciones, debemos tener en cuenta que la aprobación de la aplicación puede llevar algún tiempo, por lo que debemos tener paciencia y asegurarnos de proporcionar toda la información necesaria para que sea aprobada.

A continuación, veremos cómo cumplir con estos requisitos y subir la aplicación en caso de que queramos hacerlo.



Alojar el código de la aplicación en un repositorio GIT


Definición

Un servicio de alojamiento de repositorio es una herramienta de gestión organizativa que ofrece una visión transparente en el proceso de flujo de trabajo (que suele ser opaco) del desarrollo de software, el servicio ofrecerá herramientas para medir, monitorizar, debatir y gestionar la eficiencia y la precisión del desarrollo de software.


El objetivo principal de estas herramientas es almacenar conjuntos de archivos de datos y realizar un seguimiento de los cambios, revisiones y control de versiones que se han producido en ellos.

¿Cómo se realiza el control de versiones?



Control de versiones


Definición

El control de versiones, también conocido como "control de código fuente", es la práctica de rastrear y gestionar los cambios en el código de software y gestiona los cambios en el código fuente a lo largo del tiempo.


Cuando varias personas colaboran en un proyecto, es difícil mantener el seguimiento de las revisiones: quién cambió qué, cuándo, y dónde están almacenados esos archivos.

Los sistemas de control de versiones

se ocupan de este problema manteniendo un seguimiento de todos los cambios que se han enviado al repositorio, al igual que los documentos de Google Drive, podemos tener un historial de las versiones del código, de manera que las versiones previas no se pierden con cada iteración permitiendo regresar a una versión previa del código.

Pero te preguntarás, ¿cuál es la diferencia entre el repositorio y el control de versiones?



Diferencia entre control de versiones y repositorio


Es esencial comprender que los servicios de alojamiento de repositorios y los sistemas de control de versiones son dos entidades separadas.

    Los sistemas de control de versiones son herramientas de línea de comandos de bajo nivel que se usan para administrar los cambios en el ciclo de vida de desarrollo de software en un conjunto de archivos de código fuente.

    Los servicios de alojamiento de repositorios son aplicaciones de terceros que encapsulan y mejoran un sistema de control de versiones. No se puede utilizar por completo un servicio de alojamiento de repositorios sin tener que usar un sistema de control de versiones subyacente.

Y ¿qué es GIT?


Control de versiones GIT


Git

Es un sistema de control de versiones de código de forma distribuida, esto significa que Git te ayuda a gestionar los archivos de un proyecto y mantiene un registro de todo el historial de las cosas en las que vamos trabajando


Características 

  • - Es software libre.
  • - Nos permite mantener un historial completo de versiones.
  • - Nos ofrece un sistema de trabajo con ramas, las cuales nos permiten crear proyectos divergentes de un proyecto principal, para hacer experimentos o para probar nuevas funcionalidades.
  • - Es el sistema de control de versiones más utilizado actualmente y uno de los más modernos.


Una de las grandes ventajas de Git, que lo convierten en un factor diferencial en cuanto a la competencia, es su arquitectura distribuida o DVCS (Sistema de control de versiones distribuido), la cual permite que haya una copia del trabajo del código de cada desarrollador y, al mismo tiempo, es un repositorio local que almacena el historial completo de todos los cambios hechos dentro del código, sin importar que un desarrollador tenga una versión de código más avanzada o atrasada que otro.

Esto resulta una ventaja muy grande en comparación a la arquitectura centralizada, como CVS, la cual depende completamente del servidor central que maneja todo el sistema de control de versiones, ya que, en caso de que esta fallara o los servidores se cayeran, no habría forma de recuperar los avances en el código hasta que todo el sistema centralizado sea reparado.


En conclusión

Git es una herramienta poderosa que nos permite rastrear y gestionar cambios en un proyecto de forma segura y eficiente.


 

 


Repositorio GIT


Un repositorio de Git es un almacenamiento virtual de nuestro proyecto. Nos permite guardar versiones del código a las que podemos acceder cuando lo necesitemos.


Características

  • - Los repositorios pueden ser públicos o privados.
    - La elección de uno u otro dependerá del tamaño del equipo de trabajo y del grado de privacidad requerido por los desarrolladores u organización en función de sus objetivos de negocio.

  • - En un repositorio podemos guardar los archivos necesarios para nuestros proyectos, pudiéndose encontrar en servidores que proveen este tipo de servicios (como son Github, Bitbucket, Gitlab, etc.), siendo un tipo de repositorio los repositorios remotos .


Repositorios remotos

Los repositorios remotos de Git son copias del repositorio de un proyecto que se encuentran almacenadas en un servidor en línea, en lugar de una computadora local, se utilizan para compartir los cambios en un proyecto con otros colaboradores y para mantener una copia de seguridad en línea del mismo.

Cuando trabajamos en un proyecto en Git, podemos enviar nuestros cambios a un repositorio remoto con el comando git push, y podemos obtener los cambios que otros colaboradores han realizado con el comando git pull, esto nos trabajar con otros desarrolladores en un proyecto colaborativo, ya que Git maneja la fusión de cambios de manera automática y permite revertir cualquier cambio que no deseemos.



Fundamentos GIT


Cuando trabajamos en un proyecto en Git, podemos realizar cambios en los archivos del proyecto en nuestra computadora local, y luego guardar esos cambios en el repositorio local de Git de nuestra computadora.

Cada vez que guardamos un conjunto de cambios en el repositorio local de Git, lo llamamos commit.


Al hacer un commit, estamos guardando una "foto" de los archivos del proyecto en su estado actual, junto con un mensaje que describe los cambios que hayamos realizado, los commits nos permiten revertir cualquier cambio que hayamos realizado en el proyecto en cualquier momento, ya que Git almacena un historial completo de todos los commits que hemos realizado en el repositorio, y podemos compartir nuestros cambios con otros colaboradores a través de repositorios remotos, como GitHub o GitLab.

El estado de los archivos de un proyecto en GIT

Cada archivo de un repositorio puede tener dos estados: rastreado y sin rastrear.


    Los archivos rastreados (tracked files):son todos aquellos archivos que estaban en la última instantánea del proyecto; pueden ser archivos sin modificar, modificados o preparados.

    Los archivos sin rastrear: son todos los demás, es decir, cualquier otro archivo de nuestro directorio de trabajo que no estaba en la última instantánea y que no está en el área de preparación ( staging area).

Cuando clonamos por primera vez un repositorio, todos nuestros archivos estarán rastreados y sin modificar ya que aún no han sido editados.

Crear un archivo APK


Las siglas APK corresponden a los términos Android Application Package.

Archivo APK

Es el paquete de instalación de una aplicación para dispositivos Android, contiene todos los recursos y código necesarios para instalar y ejecutar la aplicación en un dispositivo Android, así como información adicional como el nombre de la aplicación, el icono y las autorizaciones necesarias para acceder a diferentes funcionalidades del dispositivo, estos archivo se usan para distribuir e instalar aplicaciones en dispositivos Android, y pueden descargarse desde la tienda de aplicaciones de Google Play o desde otras fuentes en línea.


Crear compilación para Android en Appflow

Para crear un archivo APK de la aplicación, primero debemos crear una compilación para Android, dentro de Appflow.

    Vamos a Build > Builds y hacemos click en el botón “New Build”. 

    A continuación debemos seleccionar el commit del repositorio de github del cual queramos obtener los archivos para crear la compilación. En este caso elegimos el último commit realizado.

    Luego debemos seleccionar para qué tipo de plataforma realizaremos la compilación. En este caso, seleccionaremos Android.

     A continuación, debemos elegir el tipo de compilación. Hay algunos tipos diferentes de compilaciones que podemos elegir para cada plataforma (iOS, Android) que tienen diferentes características y requisitos. En Android, tenemos 2 tipos de compilaciones principales:

         a) debug: las compilaciones de Android de tipo debug o depuración no requieren ningún certificado de firma y se pueden descargar e instalar directamente en dispositivos Android.

         b) release: para aplicaciones que se quieren subir a la Play Store, requieren crear certificados de firma de Android y cargarlos en un certificado de firma de tipo “Production”.

    En caso de que queramos subir la aplicación a la Play Store, debemos elegir “Release”, si simplemente queremos generar el archivo APK para instalarlo en un celular Android, entonces podemos simplemente elegir “debug” y no será necesario el certificado de firma.

    Finalmente, hacemos click en el botón “Build” para generar la compilación de nuestra app.

    Se abrirá una consola dónde podremos ver logs mientras se realiza la compilación.     


Una vez finalizado este proceso, podremos descargar el archivo APK correspondiente haciendo click en el botón “Download APK” que nos saldrá a la derecha de la consola.





Instalar APK en un dispositivo Android


Una de las principales ventajas de Android respecto a otros sistemas operativos móviles es la libertad, ya que podemos, por ejemplo, instalar aplicaciones de cualquier sitio más allá de la tienda oficial de Google, por supuesto esto conlleva ciertos riesgos de seguridad, por lo que no se recomienda instalar aplicaciones fuera de la Play Store, sin embargo, como esta aplicación la hemos desarrollado y compilado nosotros mismos, podemos instalarla para probarla en nuestros dispositivos Android.

¿Qué tener en cuenta antes de instalar aplicaciones fuera de la Play Store?

En la mayoría de los casos Android tiene deshabilitada la opción de instalar aplicaciones de orígenes desconocidos, que son todas las que no se instalan a través de Google Play.

Para poder instalar aplicaciones fuera de la Play Store, debemos ir a los Ajustes de Android, y una vez en ellos buscar la opción de “Seguridad”, activamos la opción “Orígenes desconocidos”.


¿Cómo instalar nuestra app?

Para instalar nuestra app, vamos a enviar el archivo APK que descargamos anteriormente al dispositivo dónde queramos instalar la aplicación (puede ser vía email o utilizando algún sistema de transferencia de archivos), una vez allí, seleccionamos el archivo APK y Android lo abrirá con su instalador de paquetes.

Es probable que Google nos advierta al realizar la instalación, ya que es una app con autor (desarrollador) desconocido.


Una vez instalada la app, podremos abrirla y utilizarla como cualquier otra app de nuestro teléfono.



Subir APK firmado a Play Store


Android requiere que todos los APK que se suben a la Play Store estén firmados de manera digital con un certificado para poder instalarse o actualizarse en un dispositivo.

¿Qué es un APK firmado?

Un archivo APK firmado es un APK que ha sido firmado con un certificado digital válido, la firma digital se usa para garantizar la autenticidad y la integridad del archivo APK. 

Cuando una aplicación se firma digitalmente, se genera un hash único del archivo APK que se incluye en la firma de esta forma, cuando un dispositivo Android intenta instalar la aplicación, comprueba la firma y calcula el hash del archivo APK para verificar que no se ha modificado con el fin de asegurarse de que la aplicación que se está instalando es la misma que se ha firmado y que no ha sido alterada de alguna manera.


Generación de certificado Android


El almacén de claves de Android, que se utiliza para firmar aplicaciones, se puede generar mediante keytool, una herramienta que se incluye al instalar Android Studio.


Keytool se usa comúnmente para generar certificados de seguridad, importar y exportar claves y certificados, y para gestionar las entradas del almacén de claves (keystore).

keytool -genkey -v -keystore MY-RELEASE-KEY.keystore -alias MY_ALIAS_NAME -keyalg RSA -keysize 2048 -validity 10000 -storetype jks

Debemos cambiar “MY-RELEASE-KEY” y “MY_ALIAS_NAME” por el nombre que deseemos, la herramienta nos pedirá que ingresemos una contraseña de almacén de claves y una contraseña de clave, además, nos solicitará que completemos otros datos como nuestro nombre, organización y ciudad.

Una vez finalizado, podremos encontrar un archivo MY-RELEASE-KEY.keystore en la carpeta dónde ejecutamos el comando.


Nota

  • Es sumamente importante almacenar de forma segura una copia de seguridad de la contraseña del almacén de claves, la contraseña de la clave, el alias y el archivo del almacén de claves ya que estos no son recuperables y pueden evitar que podamos actualizar una aplicación en Play Store.

Subir el certificado a Appflow

Una vez que generamos el keystore o almacén de claves, vamos a subirlo a Appflow, para eso, vamos a la pantalla Build > Signing Certificate, y hacemos clic en el botón “Add certificate”.

Agregamos un nombre para el certificado, y elegimos la plataforma, en este caso Android, se nos solicitará que adjuntemos el archivo keystore que generamos en el paso anterior, y completemos con el alias de la clave y las contraseñas, tanto del archivo como de la clave. Hacemos clic en “Add certificate” y podremos ver el certificado agregado en nuestra aplicación de Appflow.

Generar APK firmado en Appflow

 Para crear un archivo APK firmado, debemos crear una compilación para Android dentro de Appflow, como hicimos anteriormente, sólo que en lugar de hacer una compilación del tipo “debug” elegiremos “release”.


     Vamos a Build > Builds y hacemos click en el botón “New Build”.

     A continuación, debemos seleccionar el commit del repositorio de github del cual queramos obtener los archivos para crear la compilación. En este caso elegimos el último commit realizado.

    Seleccionamos la plataforma Android.

     A continuación, debemos elegir el tipo de compilación “release”, ya que nuestro objetivo es obtener un APK firmado para subir a la Play Store.

    Como elegimos la compilación “release”, debemos elegir un “Signing certificate”. Elegimos el certificado que cargamos previamente.

    Además, debemos elegir la opción “Google Play Store”, esto nos solicitará que agreguemos un “destino” en Appflow.

    Hacemos click en el botón “Set up your first destination to store”. Se abrirá un cuadro de diálogo. Debemos completar las siguientes opciones:
    a)   Nombre de destino. 
    b)   Tipo: elegimos Google Play.
    c)   Track: Google Play Developer nos permite subir APK nuevos para nuestras apps y publicarlos en diferentes segmentos, permite implementar versiones alfa y beta de una app, las cuales estarán disponibles sólo para usuarios aprobados e implementar una versión de lanzamiento en etapas, que estará disponible automáticamente para una pequeña cantidad de usuarios de la app. Tenemos varias opciones:
          i)   Segmentos de pruebas, como "alpha" y "beta". Las versiones alfa y beta de la app se implementan para los usuarios que asignamos a los grupos de prueba alfa y beta, podemos asignar usuarios a estos grupos mediante Google Play Console.
         ii)   Segmento de pruebas internas: "internal": para pruebas internas. Elegiremos esta opción.
        iii)   Segmento de producción: "production": se implementa para todos los usuarios.

    d)   Nombre del paquete (package name): en formato de nombre de dominio inverso, por ejemplo, com.tudominio.tunombredeaplicación.
    e)   Formato de publicación: el tipo de artefacto de compilación.

          i)   APK: el formato que ya hemos visto anteriormente.
         ii)   Paquetes de aplicaciones de Android (AAB): Google exige que las aplicaciones que se suban a Google Play deben hacerlo con formato ABB (Android App Bundle) en lugar de APK.

    Un AAB es un archivo de paquete de Android, similar a un archivo APK pero es más eficiente que un APK ya que permite a los/las desarrolladores/as distribuir aplicaciones más grandes y mejor optimizadas.

    La ventaja de ABB es que el tamaño de las aplicaciones se reducen un 15% frente a un APK universal, ya que cuando el usuario va a descargar una aplicación en formato ABB se genera un APKM con únicamente los recursos necesarios para ese dispositivo tales como: la versión optimizada para su procesador, su idioma y recursos gráficos adaptados a la resolución de su pantalla, evitando así instalar recursos que el usuario no va a usar. Por lo tanto, debemos elegir esta opción en lugar de APK.
    e)   Clave JSON: para cargar la app en Google Play Store, debemos crear una cuenta de servicio y descargar el archivo JSON que contiene información de seguridad para la cuenta de servicio, podemos generarlo siguiendo las instrucciones de  la  siguiente página:

    https://ionic.io/docs/appflow/destinations/google

    Una vez que creamos un “destino” en Appflow, podemos seleccionarlo y crear la compilación para Google Play.

    Finalmente, hacemos click en el botón “Build” para generar la compilación de nuestra app.

    Se abrirá una consola dónde podremos ver logs mientras se realiza la compilación.

    Una vez finalizado este proceso, podremos descargar el archivo AAB correspondiente haciendo click en el botón “Download AAB” que nos saldrá a la derecha de la consola.

    Finalmente, este archivo AAB que descarguemos podemos subirlo a la Play Console para publicar la aplicación en la Play Store, luego debemos esperar a la revisión de la app para que esté disponible en la tienda.

    Si es la primera aplicación que subimos, la validación puede tardar un par de días


Despliegue de la aplicación


El despliegue, se refiere al proceso de poner en producción una aplicación, esto puede incluir tareas como:

    La compilación y empaquetado de la aplicación.

    La creación de un perfil de distribución..

    La publicación en la tienda de aplicaciones correspondiente.

Implica preparar la aplicación para su funcionamiento en un entorno de producción y hacerlo accesible a los usuarios finales, además de las tareas de instalación de la aplicación en un servidor, la configuración de la base de datos y otros componentes necesarios, así como la realización de pruebas para asegurarse de que el sistema esté funcionando correctamente.

 

¿Qué relevancia tiene el despliegue?

El despliegue es una parte importante del ciclo de vida de desarrollo de una aplicación, ya que asegura que la aplicación esté lista para su uso en un entorno real y permite que los usuarios finales puedan acceder a ella y utilizarla, también puede incluir la actualización de la aplicación con nuevas funcionalidades y correcciones de errores, lo que permite mantenerla en buen estado y mejorar su calidad y rendimiento.

Conlleva la subida del código de la aplicación a un repositorio de control de versiones como Git, lo que permite tener un historial de cambios y facilitar el seguimiento y la gestión del código.

Te preguntarás, ¿cómo se aloja el código de la aplicación? Te lo contamos a continuación.



Comandos básicos de GIT


Comandos Git

Conjunto de instrucciones que se utilizan para interactuar con un repositorio de Git, con estos comandos, podemos administrar y hacer seguimiento de los cambios en los archivos de nuestra aplicación, podemos realizar tareas como hacer commit de cambios, crear ramas, fusionar cambios, entre otras tareas.




Comando Descripción
git init
  • Con este comando le indicaremos a Git que queremos crear un nuevo repositorio para utilizar su sistema de control de versiones, sólo debemos posicionarnos en la carpeta raíz de nuestro proyecto y ejecutar el comando.
git config

  • Este comando se utiliza para establecer o cambiar la configuración de Git en nuestra computadora, podemos definir nuestro nombre y dirección de correo electrónico, que se utilizarán como identidad en los commits que realicemos.

    Por ejemplo, para configurar nuestro nombre y email, podemos ejecutar: 
    git config --global user.email "tu@email.com" 
    git config  --global user. name  "Tu Nombre"

    Existen muchas otras configuraciones de Git que podemos encontrar ejecutando el comando git config --list (o sólo git config para ver una explicación más detallada).
git add

  • Este comando se usa para agregar archivos a la "zona de preparación" (staging area) en nuestro repositorio local, los archivos en la zona de preparación están listos para ser incluidos en un commit.
  • El comando git add puede recibir tanto la ruta de un archivo como la de una carpeta; si es de una carpeta, el comando añade recursivamente todos los archivos que están dentro de la misma.

  • Por ejemplo, para agregar un único archivo:

                       git add  nombre_archivo.js    

    Para agregar una carpeta “pages”.

                    git add pages 

    Para incluir todos los archivos modificados:

                    git add.

git commit

  • Una vez que hayamos agregado los archivos que queremos incluir en el commit (con “git add”), podemos ejecutar el comando “git commit” para guardar los cambios en el repositorio local de Git. Cuando ejecutamos git commit, estamos creando un punto en el historial del repositorio que incluye los cambios que hayamos agregado a la zona de preparación ( staging) previamente con “git add”, al hacer un commit, debemos proporcionar un mensaje que describa los cambios que hayamos realizado en el proyecto, utilizando el siguiente formato:

                        git commit -m "Mensaje del commit"                                                 

  • El mensaje del commit debe ser conciso y descriptivo, y debe indicar de manera clara qué cambios se han realizado en el proyecto, una vez que hayamos ejecutado git commit, los cambios se habrán guardado de forma permanente en el repositorio local de Git y estarán listos para ser compartidos con otros colaboradores o para ser respaldados en un repositorio remoto.
git pull
  • Se utiliza para descargar y fusionar cambios del repositorio remoto en nuestro repositorio local, cuando ejecutamos git pull, Git descarga los cambios del repositorio remoto y los fusiona automáticamente en el repositorio local, esto nos permite obtener los cambios que hayan realizado otros colaboradores en el proyecto y fusionarlos con los nuestros.
git clone
  • Se utiliza para crear una copia local de un repositorio remoto, cuando ejecutamos “git clone”, Git descarga todos los archivos del repositorio remoto y los guarda en una nueva carpeta en nuestra computadora y crea una conexión entre el repositorio local y el repositorio remoto, lo que te permite enviar nuestros cambios al repositorio remoto con el comando “git push” y obtener los cambios de otros colaboradores con el comando “git pull”.


        


Sistema de Ramas y Gitflow


Todo repositorio va a contener una línea de tiempo sobre la cual se marcarán los cambios que se hagan sobre los archivos, ésta sería una rama.

Los repositorios traen consigo una rama llamada “master” o “main” y es la principal en todos los proyectos, a partir de esta rama se pueden crear otras ramas y, por lo tanto, otras líneas de tiempo.

Cualquier sistema de control de versiones moderno tiene algún mecanismo para soportar distintas ramas (branches). Cuando hablamos de branching o trabajo con ramas, significa que un usuario ha tomado la rama principal de desarrollo “master” o “main” y a partir de ahí ha continuado trabajando sin seguir la rama principal de desarrollo.

¿Para qué crear y configurar nuevas ramas?


    Para solucionar bugs  o errores en nuestra aplicación, creando una rama a partir del momento en que estos surgen, y para poder resolverlos más rápidamente.

    Para desarrollar y probar nuevas funcionalidades en la aplicación antes de integrarlas a la rama principal, para una posterior entrega de valor.

    Para experimentar nuevas ideas en nuestras aplicaciones de forma segura, ya que podremos tener siempre la rama principal inalterada.

Flujo de trabajo con ramas

Para establecer un manejo adecuado de las ramas dentro de los repositorios de Git existen diferentes flujos de trabajo a seguir para generar menos conflictos entre cambios según las mejores prácticas y la lógica de trabajo del equipo del proyecto.

Gitflow es uno de esos flujos de trabajo y se basa en cinco palabras clave durante la construcción de una aplicación y sus respectivas entregas de valor.

Master


Es la rama principal de proyecto, donde va el código para generar la aplicación en entornos de producción. Sobre ella no se debería desarrollar.

Hotfix


Este tipo de rama se crea para la solución de un incidente o error. Debe ser generada a partir de Master. Hotfix se utiliza en ambientes en producción para resolver errores.

Develop


En esta rama se tienen las nuevas funcionalidades de la aplicación. No es aconsejable hacer commit (guardar cambios) directamente sobre ella, excepto si son cambios pequeños y que no afecten la lógica; por ejemplo, cambiar un texto.

Feature


Rama generada a partir de la rama develop. Sobre ella se harán los desarrollos de nuevas funcionalidades, una vez que esté desarrollada la funcionalidad, la rama feature podrá ser mezclada con Develop para que los cambios queden allí sincronizados, y finalmente se procede a borrar la rama feature. Es aconsejable tener esta rama sincronizada con los cambios que se agreguen a la rama develop para evitar conflictos.

Release


Esta rama se utiliza para las entregas a producción o ambiente real. Su propósito es habilitar que en ella se hagan las pruebas del cliente, una vez que se termine con las pruebas y sean exitosas se mezcla con la rama master y se sale a producción.


Plataformas de almacenamiento más conocidas

Github


Es una de las plataformas de alojamiento de código fuente más conocidas. Propiedad de Microsoft, ofrece a los/las desarrolladores/as la posibilidad de crear repositorios de código y guardarlos en la nube de forma segura usando el sistema de control de versiones Git.

Gitlab


Es otra plataforma popular de hospedaje de código fuente que ofrece muchas funciones similares a Github, ya que ambas controlan la versión de los proyectos hospedados usando Git.

Gitlab proporciona de forma nativa herramientas de integración y entrega continua o CI/CD, así como métricas para el seguimiento de la calidad, el rendimiento y el rendimiento del código así como Pruebas de usabilidad.

Bitbucket


Es una plataforma para alojar y administrar código fuente para proyectos escritos en Python. Se integra con Jira y Trello permitiendo gestionar repositorios desde estas herramientas. BitBucket admite la integración continua mediante Pipelines.
