Introducción


La semana pasada escribimos nuestras primeras pruebas unitarias utilizando NUnit y exploramos algunos atributos de pruebas. 

Estas primeras pruebas eran para un sistema simple donde teníamos que revisar el resultado de algunos valores de retorno del software. Sin embargo, en nuestra vida laboral muchas veces nos encontraremos con conexiones a sistemas externos. 


Esto quiere decir que, en ocasiones, el resultado de nuestras pruebas dependerá de un objeto sobre el cual no tenemos control o incluso, que aún no funciona. Este objeto puede ser un web service, una API externa, la hora del día, un servicio de nuestro mismo software que está desarrollando alguien más, etc. Lo importante, es que no tenemos ningún tipo de control sobre lo que nos retornará esta dependencia. Para esto, utilizamos stubs .


Continuaremos usando el ejemplo del programa LogAn, que estudiamos la semana anterior, para comprender cómo funcionan los stubs, así que te invitamos a que tengas a mano el archivo con el código del programa, para revisarlo a medida que avancemos con las explicaciones.


Comencemos...

Uso de stubs


Antes de avanzar, pongamos en común algunas definiciones.

Definición 1

Una dependencia externa es un objeto de nuestro sistema con el que nuestra unidad de prueba interactúa, pero sobre el cual no tiene control alguno.

Controlar estas dependencias externas es lo que analizaremos esta semana. Para esto, utilizamos stubs.

Definición 2

Un stub es un reemplazo controlable para una dependencia existente en el sistema. Cuando utilizamos un stub podemos probar nuestro código sin tener que lidiar con las dependencias de una manera directa.

Los stubs son una herramienta valiosa para mejorar la calidad de las pruebas unitarias. Más adelante estaremos hablando sobre mocks, que son similares a los stubs, pero con la principal diferencia que realizamos los asserts contra los mocks, pero no así con los stubs.

Continuaremos utilizando el ejemplo del programa LogAn, que conocimos la semana anterior, para comprender cómo funcionan los stubs. Utilizaremos un stub para librarnos de una dependencia del sistema de archivos. Recordemos que el programa LogAn lee archivos del Logs del sistema de archivos y los revisa buscando ciertas anomalías.


A modo de ejemplo

Imaginemos que la aplicación LogAnalyzer puede ser configurada para recibir múltiples archivos de log con múltiples extensiones diferentes, utilizando adaptadores distintos para cada tipo de archivo. Para simplificar las cosas, vamos a asumir que los archivos se guardan en algún lugar del disco y tenemos una configuración que nos indica a donde. Podríamos decir entonces que una versión simplificada del método se ve de la siguiente manera:


El problema con el que nos encontramos es que una vez que la prueba depende del sistema de archivos, estamos haciendo una prueba de integración, y por tanto nos encontramos con todos los problemas asociados con pruebas de integración: Son más lentos de correr, necesitan una configuración extra, prueban múltiples cosas, puede haber fallas no relacionadas al código que queremos probar.

Aquí identificamos la esencia del “diseño que inhibe pruebas”: El código a probar tiene una dependencia a un recurso externo, que puede romper la prueba incluso si la lógica del código es perfecta y sin defectos. 

En sistemas legacy (sistemas viejos que heredamos y sobre los que debemos trabajar) es común que cada unidad de trabajo tenga muchas dependencias a recursos externos sobre los que el código a probar tiene poco o nada de control.

A continuación, veamos cómo probar y escribir fácilmente la prueba.

Cómo probar fácilmente LogAnalyzer


Aquí es donde vemos la parte del arte de las pruebas unitarias. Este arte se trata de encontrar el lugar adecuado para agregar o utilizar una capa de abstracción para probar mejor el código.

Si hay algo que no logremos probar, entonces podemos agregar una capa que contenga las llamadas a ese algo, y luego imitamos esta capa en las pruebas. El arte de probar también involucra descubrir cuándo una capa de abstracción ya existe en vez de tener que inventarla, y también incluye el saber cuándo no agregar una porque complica demasiado el código.

Volviendo al LogAnalyzer, la única forma como podemos escribir una prueba para este código, es tener un archivo de configuración en el sistema de archivos. Como queremos que nuestro código sea testeable sin tener que recurrir a pruebas de integración, estamos intentando evitar este tipo de dependencias.

Si lo miramos con cuidado, notamos que hay un patrón para descomponer la dependencia:

Encontrar la interfaz o la API con la cual trabaja el objeto que estamos probando. En este caso no usamos la palabra interfaz en el sentido más puro de la programación orientada a objetos, sino que hablamos de la clase u objeto con el cual estamos trabajando. 

Reemplazar la implementación subyacente de esa interfaz con algo sobre lo que sí tenemos control.

Transferir este patrón al código con el que estamos trabajando requiere unos pasos extra:

Primero vamos a encontrar la interfaz. En nuestro caso particular del proyecto LogAn, estamos hablando del archivo de configuración que tenemos en el sistema de archivos.

Si la interfaz está conectada de forma directa a la unidad de trabajo bajo prueba (como es nuestro caso, ya que estamos ingresando directamente al sistema de archivos), podemos hacer el código testeable, agregando una capa de abstracción y escondiendo la interfaz. La figura 7.1 muestra como quedará el diseño luego de este paso.

Por último, reemplazar la implementación subyacente de esa interfaz interactiva con algo sobre lo que tengamos control. En nuestro caso, reemplazamos la instancia de la clase que nuestro método llama (FileExtensionManager) con una clase stub que sí podemos controlar (StubExtensionManager), dándonos control sobre las dependencias externas.

Nuestro reemplazo no se comunicará con el sistema de archivos en absoluto, lo que rompe la dependencia con el sistema de archivos. Como no estamos probando la clase que si se comunica con el sistema de archivos (eso lo haremos cuando hagamos pruebas de integración) sino el código que llama a esa clase; está bien que la clase stub no haga nada realmente, sino que responda felizmente cuando estamos corriendo una prueba. Refactorizar el diseño para que sea testeable


Volvemos a las definiciones para introducir dos conceptos que continuaremos utilizando a lo largo de la semana: refactoring y seams.

Definición 1

Refactoring es el acto de cambiar el código sin cambiar su funcionalidad. Es decir que hace exactamente lo mismo que hacía antes. Ni más, ni menos. Simplemente se ve diferente. Un ejemplo de refactoring puede ser cambiar el nombre de un método por otro más representativo, o dividir un método muy largo en otros más pequeños para que sea más legible. El código puede refactorizarse por muchos motivos, para optimizar el código, hacerlo más legible, más testeable (como develaremos en esta semana), etc.

Definición 2

Las seams son lugares en nuestro código donde podemos enchufar una funcionalidad diferente. Como, por ejemplo, las clases de stub, agregar un parámetro a un constructor, agregar una propiedad de tipo public, hacer un método sin implementación para que se le pueda realizar un override. Las seams son lo que obtenemos al implementar el principio “Abierto Cerrado”, donde la funcionalidad de una clase está abierta para extensión, pero su código está cerrado para modificaciones directas.

Podemos refactorizar nuestro código agregando una nueva seam sin cambiar la funcionalidad original del código. Esto es exactamente lo que hicimos en el apartado anterior, agregando la interfaz IExtensionManager.

Antes de comenzar a refactorizar, es muy importante recordar que es una acción muy peligrosa si no tenemos ningún tipo de pruebas automatizadas que verifiquen sus funcionalidades. Siempre debemos tener algún tipo de pruebas de interacción cuidando nuestra espalda antes de hacerle algo al código existente que tenemos. Si no tenemos esas pruebas de integración, al menos debemos poseer un plan de escape, es decir, una copia del código intacta. De esa manera, si todo sale mal, podemos tirar a la basura el código refactorizado y quedarnos con el código sin cambiar. Hablaremos sobre las pruebas de integración más adelante en la materia, así que por ahora se recomienda tener el plan de escape listo mientras aprendemos cómo refactorizar un código y hacerlo más testeable.

Para romper la dependencia entre nuestro código y la dependencia que tenemos con el sistema de archivos, utilizaremos seams en nuestro código. Lo único que debemos hacer, es asegurarnos de que el código resultante haga exactamente lo que debe hacer una vez que terminamos. Existen 2 tipos de refactories que rompen dependencias:


Tipo A: Abstracción de objetos concretos a interfaces o delegados.

- Técnica común de este tipo: Extraer una interfaz para reemplazar la implementación subyacente.

Tipo B: Refactorizar para permitir una inyección de implementaciones falsas en esas interfaces o delegados. Algunas técnicas comunes de este tipo: 

- Inyectar un stub en la clase de prueba.

- Inyectar un objeto falso (o fake) a nivel de constructor.

- Inyectar un objeto falso (o fake) como propiedad de get o set. 

- Inyectar un objeto falso (o fake) justo antes de la llamada a un método.

Te invitamos a profundizar en estas técnicas de refactoring en el documento S7. Tipos de refactories que rompen dependencias



Interacción utilizando mock objects


Hasta ahora hemos estudiado cómo resolver el problema de probar el código que depende de otros objetos para ser probado correctamente. Hemos usado stubs para asegurarnos que el código bajo prueba recibe las entradas necesarias para probar su lógica de forma independiente.

Ahora analizaremos el funcionamiento de los mock objects. Estos objetos nos permitirán revisar si un objeto está haciendo llamadas correctas a otro objeto. Es posible que el método que queremos probar no tenga un valor de retorno, pero sí tenga algún tipo de lógica compleja que queremos revisar y que está retornando correctamente.

La comunicación entre objetos

Los mocks son interacciones entre objetos, ¿Qué significa que haya interacciones entre objetos?  En pocas palabras: un objeto envía un mensaje a otro. Es importante destacar que esta práctica no se trata de pruebas de integración en sí misma; más bien, se enfoca en examinar estas interacciones cuando constituyen el objetivo o resultado específico de la unidad de trabajo bajo prueba.

Definición

Un mock object es un fake u objeto falso en el sistema que decide si una prueba unitaria se ha pasado o no. Lo hace verificando si el objeto bajo prueba hizo una llamada al objeto falso como se esperaba. No suele haber más de un mock por prueba.

Podríamos decir que un mock no suena muy diferente a un stub, pero la diferencia es lo suficientemente grande como para distinguirla.

Ahora que sabemos lo que son los mocks y los stubs podemos definir lo que es un fake. Es un término que ya utilizamos bastante, y podemos imaginar qué quiere decir. 

Pero ahora observemos una definición formal:

Definición

Un fake es un término genérico que puede utilizarse para describir a un stub o a un mock, porque ambos parecen (pero no son) el objeto real. 

Si un fake es un stub o un mock depende de cómo está siendo usado en la prueba actual. Si es usado para revisar una interacción (es decir, hacemos un assert contra ese fake) entonces es un mock. Si no, es un stub.


A continuación, profundicemos las diferencias entre los objetos.

Diferencia entre los mocks y los stubs


Los stubs reemplazan a un objeto para que podamos probar otro objeto sin problemas. La figura muestra la interacción entre los objetos:


A primera vista, la diferencia entre los mocks y los stubs puede parecer minúscula o incluso inexistente. La distinción es sutil pero importante, porque muchos de los frameworks de mocks que existen usan estos términos para describir los comportamientos diferentes del framework. La diferencia básica es que un stub no puede fallar una prueba. Los mocks si.

La forma más fácil de darnos cuenta que estamos trabajando con un stub es notar que un stub nunca puede fallar una prueba. El assert que hacemos al final de la prueba siempre se dará contra la unidad de trabajo a probar. Por otro lado, una prueba está utilizando un mock para verificar si la prueba fallará o no. Primer ejemplo de mock object


Crear y usar un mock es bastante similar a utilizar un stub, excepto que el mock hará un poco más de cosas: guardará su historial de comunicación para luego verificarlo contra sus expectativas.


Vamos a agregar un nuevo requerimiento a nuestro LogAnalyzer. Ahora debe interactuar con un web service externo que recibirá un mensaje de error cada vez que LogAnalyzer encuentre un nombre de archivo que sea demasiado corto.

Desafortunadamente, el web service con el que queremos trabajar aun no esta 100% funcional, y aun si lo estuviera, tomaría mucho tiempo utilizarlo como parte de las pruebas. Debido a eso, vamos a refactorizar nuestro diseño y crear una nueva interfaz para la que podemos crear un mock después. La interfaz tendrá métodos que necesitaremos llamar para acceder al web service.

En primer lugar, vamos a extraer una interfaz simple para utilizar en el código bajo prueba en vez de comunicarnos directamente con el web service:



Esta interfaz nos servirá para crear mocks, pero también stubs. Nos permitirá evitar la dependencia externa sobre la que no tenemos control.

A continuación, crearemos el mock object que vamos a utilizar. Puede parecer un stub, pero contiene una porción extra de código que hace que sea un mock object.


Esta clase implementa la interfaz que definimos anteriormente, de la misma manera que lo hace un stub. Pero además de eso guarda el estado resultante para después. De esta manera nuestra prueba puede realizar un assert directamente contra este mock para ver si fue llamado correctamente. Esto aún no es un mock object. Se convertirá en uno cuando efectivamente lo usemos para probar, tal como analizaremos a continuación.



Prueba con mock object


Observemos en detalle el código:

[Test]
public void Analyze_TooShortFileName_CallsWebService()
{
FakeWebService mockService = new FakeWebService(); 
LogAnalyzer log = new LogAnalyzer(mockService); 
string tooShortFileName="abc.ext"; 
log.Analyze(tooShortFileName); 
StringAssert.Contains("Filename too short:abc.ext", mockService.LastError); 
}
public class LogAnalyzer
{
private IWebService service;
public LogAnalyzer(IWebService service)
{
this.service = service;
} 
public void Analyze(string fileName)
{
if(fileName.Length<8)
{
service.LogError("Filename too short:" + fileName); 
}
} 
} 

Marcado en verde podemos observar cómo estamos comparando el mock object con el resultado que esperamos en vez de hacerlo contra la clase que estamos probando. 

Esto es porque estamos probando la interacción entre el LogAnalyzer y el web service. Utilizamos las mismas técnicas de inversión de dependencia y refactoring que aprendimos cuando hablamos de los stubs, pero esta vez el mock también puede fallar la prueba.

También es importante notar que no estamos escribiendo la prueba contra el mock. Hay un par de razones para esto:

Queremos reutilizar el mock en otras pruebas con otros mensajes de assert.

Si dejamos el assert dentro del mock, quien lea la prueba no tendrá idea de que estamos comparando. Estaríamos ocultando información esencial de la prueba, disminuyendo la legibilidad y mantenibilidad.

Es posible que necesitemos reemplazar más de un solo objeto en una prueba, es por eso que podemos combinar stubs y mocks para armar pruebas. Esta perfecto utilizar múltiples stubs en una prueba, pero es incorrecto utilizar más de un mock por prueba. Esto se debe a que estaríamos probando más de una cosa, y una prueba unitaria debe probar una sola cosa.


Te invitamos a profundizar en ésta posibilidad, en el documento S7. Utilizando mocks y stubs juntos .





El problema con los fakes manuales


Hay muchos problemas que vienen de escribir de forma manual mocks y stubs en nuestro código:


Toma mucho tiempo escribir los mocks y stubs.

Es difícil escribir mocks y stubs para clases e interfaces con muchos métodos, propiedades y eventos.

Para guardar el estado de múltiples llamadas a un método de un mock, debemos escribir mucho código duplicado en el fake.

Si queremos verificar que todos los parámetros en una llamada a un método se enviaron correctamente, necesitamos escribir múltiples asserts.

Es difícil rehusar un mock o un stub manual para otras pruebas. Con fakes básicos funciona, pero cuando son más complejos con más de 3 métodos, todo se vuelve tedioso de mantener.

¿Realmente tendremos espacio para que un fake sea un stub a veces y un mock otras? Eso será algo muy raro. Eso será una o dos veces por proyecto, si es que el proyecto es lo suficientemente grande.


¿Por qué usamos mocking frameworks?


Definición

Un mocking framework es un conjunto de APIs programables que hace que crear fakes sea mucho más simple, rápido en menos líneas de código que hacerlo de forma manual.

Cuando estos frameworks se diseñan bien, pueden ahorrarle mucho tiempo al/la desarrollador/a de escribir código repetitivo para realizar validaciones y comparaciones. Y si están muy bien hechos, pueden hacer que las pruebas unitarias de nuestro código duren muchos años sin que el/la desarrollador/a tenga que volver a arreglarlos por cada cambio al código de producción.

Estos frameworks existen para la mayoría de los lenguajes que tienen frameworks de pruebas unitarias. Por ejemplo, para Java tenemos Mockito o jMock. Es importante estudiar que tan nuevo o viejo es framework antes de usarlo, ya que algunos frameworks viejos pueden carecer de funcionalidades interesantes que los más nuevos sí tendrán.


Estos mocking frameworks nos permiten escribir mocks y stubs de forma más elegante y permite hacer que pruebas más complejas sean más simples. También hace que nuestras pruebas sean menos propensas a errores.



Creando fakes de forma dinámica


Un fake dinámico es un stub o mock que se ha creado en tiempo de ejecución sin la necesidad de haber sido escrito a mano. Estudiaremos cómo implementar estos fakes utilizando NSubstitute. 


NSubstitute es un software open source y puede descargarse gratuitamente. Puede hacerse fácilmente utilizando NuGet, el manejador de paquetes de .NET. 

Analizaremos este framework ya que tiene una excelente documentación, pero otro framework que puede aprenderse, y es muy bueno, es FakeItEasy.

A partir de ahora nos referiremos a NSubstitute como NSub

Debemos descargar NSub con NuGet de la misma forma que lo hicimos con NUnit y lo importamos del mismo modo como importamos cualquier otra librería o API.

NSub soporta el modelo arrange-act-assert con el que venimos trabajando. La idea es crear los fakes y configurarlos en la porción de arrange, utilizarlos para probar en la parte de act y realizar los asserts en la parte de assert

NSub tiene una clase llamada Substitute que utilizaremos para crear los fakes en tiempo de ejecución. Esta clase tiene un método genérico, y es la forma principal de crear fakes en nuestra aplicación. Llamaremos a este método For(type) siendo el tipo de objeto del cual queremos crear un fake.

Este método crea y retorna dinámicamente un fake que se adhiere a ese tipo o interfaz en tiempo de ejecución. No necesitamos implementar ese objeto a nivel código.

A continuación, se muestra entonces una prueba que utiliza su respectivo fake escrito a mano, como veníamos haciendo hasta ahora.



Prueba con fake dinámico


Analicemos la prueba a continuación:

[TestFixture]
class LogAnalyzerTests
{
[Test]
public void Analyze_TooShortFileName_CallLogger()
{
FakeLogger logger = new FakeLogger();
LogAnalyzer analyzer = new LogAnalyzer(logger);
analyzer.MinNameLength= 6; 
analyzer.Analyze("a.txt"); 
StringAssert.Contains("too short",logger.LastError); 
}
}
class FakeLogger: ILogger
{
public string LastError;
public void LogError(string message)
{
LastError = message;
}
} 

Resaltado en verde creamos nuestro fake para ser usado. Resaltado en amarillo vemos cómo usamos ese fake como un mock para realizar un assert. Las partes resaltadas son las que cambiarán una vez que comencemos a utilizar NSub. Ahora empezaremos a crear fakes de forma dinámica. 

[Test]
public void Analyze_TooShortFileName_CallLogger()
{
ILogger logger = Substitute.For<ILogger>(); 
LogAnalyzer analyzer = new LogAnalyzer(logger); 
analyzer.MinNameLength = 6; 
analyzer.Analyze("a.txt"); 
logger.Received().LogError("Filename too short: a.txt");

Verificamos en este código que con un par de líneas de código nos libramos enteramente de los stubs o mocks escritos a mano, esto se debe a que se generan de forma dinámica. Podemos apreciar, resaltado en verde, que el objeto logger es generado dinámicamente implementando la interfaz ILogger, pero no hay ninguna implementación de los métodos de ILogger.

Desde este momento hasta la ultima línea de la prueba, todas las llamadas al fake son registradas de forma automática, es decir, guarda para su uso después. Esto ocurre en la ultima línea (la resaltada con amarillo).

En la última línea, en vez de una llamada tradicional de assert, podemos notar que usamos un método especial de NSub. La interfaz ILogger no tiene ningún método especial llamado Recieved() sino que esa es una extensión especial que realiza NSub. Este método es nuestra forma de revisar que un método ha sido invocado en nuestro fake.

Recieved() pareciera funcionar por arte de magia. Retorna el mismo tipo de objeto que lo invocó. Llamando al método Recieved() antes del método LogError(), le hacemos saber a NSub que estamos preguntándole al fake si ese método se llamó o no. Hablando en términos de legibilidad, le estamos diciendo al lector “Algo recibió (recived) una llamada al método, o esta prueba fallará”.

Si nunca se le hubiera enviado ese mensaje al fake, entonces, los logs de prueba nos mostrarían un mensaje como este:

NSubstitute.Exceptions.ReceivedCallsException : Expected to receive a call matching: LogError("Filename too short: a.txt") Actually received no matching calls.


Simulando valores falsos o stubs


A continuación, comprobamos cómo retornar un valor de un fake cuando la interfaz tiene un retorno no nulo. Para nuestro ejemplo, agregaremos una interfaz que busca reglas de los archivos permitidos en el sistema de archivos.

[Test]
public void Returns_ByDefault_WorksForHardCodedArgument()
{
IFileNameRules fakeRules = Substitute.For<IFileNameRules>(); 
fakeRules.IsValidLogFileName(Arg.Any<String>()).Returns(true); 
Assert.IsTrue(fakeRules.IsValidLogFileName("strict.txt")); 
} 

En este código apreciamos resaltado con verde, que estamos forzando al método del sustituto a retornar true. Esta línea resaltada en verde muestra que, sin importar cuál sea el parámetro que se le envía al método que estamos simulando, siempre retornará true. Esto nos permite tener una alta mantenibilidad, ya que si el código en producción cambia y hace múltiples llamados al método, no importa cuál sea ese parámetro, su respuesta siempre será true. También ayuda a la legibilidad del código, ya que resta importancia al nombre del archivo a enviar. Si utilizamos un String específico para ese parámetro, alguien que lea el código podría pensar que ese String en particular es importante.

La clase Arg que utilizamos en esa misma línea indica que no nos importa la entrada que se requiere para que este fake haga valer true el retorno de su método. Esto se llama un argument matcher y se utiliza en los frameworks de mocking para mejorar la legibilidad.

También podemos simular el lanzado de excepciones de la siguiente manera:

[ Test]
public void Returns_ArgAny_Throws()
{
IFileNameRules fakeRules = Substitute.For<IFileNameRules>(); 
fakeRules.When(x => x.IsValidLogFileName(Arg.Any<string>())).Do(context =>{ throw new Exception("fake exception"); }); 
Assert.Throws<Exception>(() => fakeRules.IsValidLogFileName("anything")); 
}

Podemos notar que en la última línea utilizamos Assert.Throws para revisar que efectivamente se lance la excepción que deseamos probar. La sintaxis de NSub en este caso no es excelente, de hecho otros frameworks como FakeItEasy tienen una sintaxis más simple para este tipo de casos.
