Introducción a las pruebas automatizadas


Las pruebas automatizadas tienen amplios beneficios como parte de una estrategia de prueba más grande diseñada para aumentar la calidad de la aplicación mientras se mantiene la velocidad de entrega. Lo hace facilitando el descubrimiento de errores dentro de una aplicación en puntos anteriores del ciclo de vida de desarrollo. Un aspecto de dicha estrategia son las pruebas unitarias que se escriben a medida que se desarrollan los componentes, ejecutados por el/la desarrollador/a dentro del proceso de construcción. Otro implica la integración de pruebas automatizadas más complejas en el proceso de Integración Continua. Entonces, a medida que los/las desarrolladores/as escriben y confirman cambios en el código de la aplicación, la funcionalidad nueva y existente se valida continuamente.

Cuando los problemas en la base del código se identifican anteriormente en el proceso de desarrollo, a menudo son menos costosos de resolver. Probablemente, son el resultado de un cambio reciente en el código, y uno de los/las desarrolladores/as del proyecto puede corregir el error rápidamente. Esto significa una menor probabilidad de que los errores de detención del programa lleguen al final del proceso de desarrollo y, en consecuencia, se presente un menor riesgo para el cronograma de entrega.

Errores comunes de las pruebas automatizadas


Con todos los beneficios que vienen con una implementación efectiva de pruebas automatizadas, está claro que vale la pena exprimir el jugo. Es decir, el esfuerzo que sin duda será necesario para superar los desafíos asociados con el desarrollo de un proceso de automatización de pruebas valdrá la pena.

Hablaremos de algunos de los desafíos o problemas más comunes al comenzar una estrategia de prueba automatizada (cliqueá sobre el título para desplegar el texto):



Selenium


Selenium es un conjunto de utilidades que facilita la labor de obtener juegos de pruebas para aplicaciones web. Para ello nos permite grabar, editar y depurar casos de prueba, que podrán ser ejecutados de forma automática e iterativa posteriormente.

Además de ser una herramienta para registrar acciones, permite editarlas manualmente o crearlas desde cero. Las acciones se basan en el uso de diferentes API's en diferentes lenguajes (PHP, Ruby, JAVA, JavaScript, etc.) Entre su principales características podemos nombrar:

Facilidad de registro y ejecución de los test.

Referencia a objetos DOM en base al ID, nombre o a través de XPath.

Auto-completado para todos los comandos.

Las acciones pueden ser ejecutadas paso a paso.

Herramientas de depuración y puntos de ruptura (breakpoints).

Los test pueden ser almacenados en diferentes formatos.

El potencial de esta herramienta es que puede ser utilizado para la grabación de las pruebas funcionales durante la Generación de pruebas de regresión. Con este servicio se consigue obtener una batería de pruebas automatizadas que podrán ser utilizadas cuando sea necesario repetir las pruebas.

Los componentes más importantes para realizar este proceso de pruebas son los siguientes:

Selenium Core: el módulo del núcleo contiene la funcionalidad básica del marco de trabajo, incluyendo el JavaScriptTestRunner (el código base sobre el cual trabaja Selenium) y la API de comandos de prueba básicos.

Selenium IDE: es un entorno de desarrollo integrado para pruebas. Se encuentra implementado como una extensión de Firefox y Chrome. Esta permite grabar, editar y depurar pruebas. Originalmente se le conoció como Selenium Recorder. Se pueden desarrollar scripts de pruebas al crear una grabación y de esa manera se puede editar manualmente con sentencias y comandos para que la reproducción de la grabación sea correcta. Los scripts se generan en Selanese, un lenguaje especial para Selenium que provee comandos que ejecutan acciones sobre objetos en el navegador, como hacer clic en un botón, seleccionar de una lista de opciones, verificar la presencia de un texto, así como para tomar la totalidad de la página producto de las acciones.

Selenium WebDriver:  es el componente más utilizado de Selenium. El mismo permite a los/las usuarios/as escribir su código en distintos lenguajes de programación e interactuar con el navegador seleccionado. WebDriver se compone de un conjunto de APIs que permiten al tester controlar el navegador a nivel del sistema operativo, las páginas web y sus elementos HTML. Al ejecutar Selenium WebDriver se levanta una instancia del navegador y ejecuta el script de prueba. Dicha instancia es específica del navegador, por ejemplo, Google ha desarrollado y mantiene ChromeDriver con el objetivo de integrarse a Selenium y dar soporte a la automatización en Chromium/Chrome. Con Selenium WebDriver también es posible la integración con diferentes librerías, como Cucumber, TestNG o JUnit. Actualmente, Selenium 4 es la última versión de Selenium, que está disponible para los lenguajes de programación: Java, .Net, Python, Ruby y JavaScript. Esta última versión tiene varias mejoras y características nuevas, como la introducción de los localizadores relativos. En el caso de utilizar algún navegador de Chromium o Firefox se han incluido formas de manejar la autenticación, la intersección de la red y tareas como esperar un cambio en el DOM, o proporcionar una forma de ver los errores de JavaScript, entre otras. Selenium permite siete localizadores de objetos diferentes como forma de interactuar con la aplicación web. Esto es importante para la creación de las pruebas automatizadas, siendo posible hacer clic, deslizar e ingresar texto a través de los localizadores.

Selenium Grid: es un servidor que permite usar instancias de navegador habilitando la ejecución en forma paralela de varias pruebas simultáneas en varias máquinas. Es un entorno con múltiples instancias, que contiene un hub y múltiples nodos. El hub es un servidor “master” que controla y se comunica con nodos. Y los nodos son dispositivos remotos donde básicamente se ejecutan los script de pruebas de Selenium. Desde el hub, se envían las instrucciones de ejecución y se puede verificar si todas las aplicaciones de los diferentes sistemas se están ejecutando correctamente. Un ejemplo es la herramienta LambdaTest, que dentro de Selenium Grid se utiliza para realizar pruebas en la nube.



Características de Selenium


Presentamos en este apartado, las ventajas y desventajas de Selenium.

Al ser un conjunto de herramientas, Selenium ofrece muchas posibilidades para las pruebas automatizadas, lo que hace posible que encuentre algo que se adapte a su necesidad y forma de trabajo.

Como se mencionó anteriormente, Selenium se puede utilizar en todos los navegadores modernos, como Chrome, Firefox, Safari, Internet Explorer, Opera y Edge. Es compatible con diferentes sistemas operativos como Windows, Linux, Mac OS, UNIX y funciona con múltiples lenguajes de programación.

El objetivo principal de las pruebas automatizadas es ahorrar tiempo y esfuerzos. Con la ayuda de Selenium Grid, se pueden ejecutar pruebas en paralelo, lo que reduce el tiempo de ejecución de las pruebas.

Además, al ser de código abierto, las herramientas se pueden adaptar según las necesidades de cada usuario/a y existe una gran cantidad de usuarios/as que lo utilizan, lo que facilita el aprendizaje y compartir mejores prácticas.

Por último, también se destacan como otras ventajas:

Es gratis, no incluye costos de licenciamiento.

Simple de instalar.

Tiene mucha capacidad de integración, lo que permite realizar distintas pruebas y reportes en paralelo.

Reutilización de código, los script de pruebas se pueden utilizar en diferentes escenarios y navegadores.

Cuenta con mucha documentación, tanto oficial como de la comunidad Selenium.

Desventajas de Selenium


Más allá de todas los beneficios y ventajas, también es oportuno mencionar que carece de algunas funcionalidades.

En el caso de automatización de pruebas, tanto en aplicaciones de escritorio como en aplicaciones mobile, se requiere de otros drivers o componentes adicionales. Por ejemplo, se puede utilizar Appium para aplicaciones que se ejecutan en dispositivos móviles.

Las funcionalidades vinculadas a informes y reportes de datos de las pruebas, solo son posibles mediante integraciones con otras herramientas.

A su vez, se debe considerar el mantenimiento de la infraestructura. La configuración de varios componentes y de Selenium Grid es una tarea que requiere tiempo. Implica un seguimiento continuo, así como las actualizaciones y parches de las mismas.

Otra de sus desventajas, es que se requiere conocimiento de programación para realizar las pruebas. A pesar de ser una ventaja su utilización en varios lenguajes de programación, muchas veces los tester no necesariamente tienen conocimientos de desarrollo, lo que limita su uso.



Tipos de pruebas con Selenium


A continuación, se detallan los diferentes tipos de pruebas que se pueden automatizar con Selenium:

Dentro de las pruebas funcionales, se encuentran las pruebas de UIn y para automatizar las pruebas, Selenium controla de forma remota las instancias de los navegadores, obteniendo los elementos de la interfaz gráfica de usuario y así poder simular tareas o interacciones, como completar campos, seleccionar una opción de campos desplegables, hacer clic en botones, entre otros.

Las pruebas de regresión aseguran que no se introduzcan efectos secundarios en las funcionalidades de una aplicación cuando el código cambia, o cuando se introducen nuevas funcionalidades en la aplicación. Las pruebas automatizadas facilitan mucho este tipo de pruebas, ahorrando tiempos en la liberación de nuevas versiones.

Las pruebas de compatibilidad garantizan que la aplicación web cumpla con los puntos de rendimiento en las diferentes combinaciones de sistemas operativos y navegadores. Las pruebas automatizadas con Selenium facilitan mucho este tipo de pruebas debido a que tiene una gran soporte con diferentes navegadores y sistemas operativos.

El objetivo de las pruebas end-to-end es verificar que todos los puntos de contacto con la aplicación web funcionen. Por ejemplo, el tester podría escribir una serie de casos de prueba para verificar que la búsqueda, el registro, el pago y la consulta de un producto, funcionen según lo previsto (y fallen cuando se ingresen valores no válidos en los campos correspondientes).



¿Por qué automatizar pruebas con Selenium?


Cuando se realizan pruebas con frecuencia y de manera repetitiva, es un buen momento para comenzar a evaluar la necesidad de realizar pruebas automatizadas. También, cuando comienza a consumir mucho tiempo realizarlas de forma manual. Es allí cuando se debería de evaluar su automatización con Selenium.

Como se ha mencionado anteriormente, son muchas las ventajas para utilizar Selenium a la hora de implementar pruebas automatizadas: no requiere una licencia ni costos para utilizarlo, imita acciones de usuario, fácil de implementar, cuenta con diferentes herramientas según cada escenario, soporta diferentes lenguajes de programación, es compatible con otros framework, el código puede ser reutilizable y posee el soporte de una comunidad bastante grande.



¿Cómo elegir la herramienta correcta de Selenium para trabajar?


Debido a que hay varias herramientas o componentes de Selenium, puede que la decisión sobre qué herramienta utilizar no sea tarea sencilla para nuevos/as usuarios/as. A continuación, se detalla una breve guía para poder elegir correctamente los componentes de Selenium a utilizar para cada necesidad:

Selenium IDE

  • Si se tiene poco conocimiento previo de programación y de pruebas automatizadas. 
  • Para crear casos de pruebas simples y suites de pruebas, que luego se pueden exportar a Selenium WebDriver. 
  • Ejecutar código JavaScript personalizado utilizando runScript. 
  • Se puede probar una aplicación web sólo en Firefox y Chrome. Se puede exportar el script de pruebas en varios lenguajes de programación.


Selenium WebDriver

  • Si se necesita utilizar un lenguaje de programación específico para sus casos de prueba automatizados. 
  • Para probar aplicaciones y generar reportes personalizados con formato HTML. 
  • Si se desea probar aplicaciones web con datos dinámicos. 
  • Para probar aplicaciones con funcionalidades basadas en AJAX. 
  • Si se desea probar aplicaciones en diferentes plataformas usando Selenium Grid o probar aplicaciones en CI/CD.


Selenium Grid

  • Si se necesita ejecutar casos de prueba automatizados en diferentes navegadores y sistemas operativos simultáneamente. 
  • Si se necesita ejecutar una suite de pruebas muy grande y en el menor tiempo posible.



Selenium WebDriver


Lo primero que debemos hacer para comenzar a trabajar con Selenium, es instalar las librerías correspondientes que nos permitirán automatizar las pruebas. Para ello tenemos varias opciones según como prefiramos realizar las instalaciones.

Si utilizamos el Packet Manager podemos utilizar el comando:

Install-Package Selenium.WebDriver

Si utilizamos el .NET CLI podemos utilizar el siguiente comando:

dotnet add package Selenium.WebDriver

Si queremos especificarlo como una dependencia por referencia en el CSProj debemos agregar la siguiente dependencia como un PackageReference en un ItemGroup:

<PackageReference Include="Selenium.WebDriver" Version="4.7.0" />



Instalar los drivers de los navegadores


A través de WebDriver, Selenium da soporte a los navegadores más grandes del mercado como por ejemplo Google Chrome, Chromium, Firefox, Internet Explorer, Safari y Edge. Estos drivers no vienen por defecto con la versión de Selenium que acabamos de instalar ya que cada navegador provee su propia distribución del mismo.

Existen 3 formas estables de instalar los drivers. Existe también una forma provista por Selenium utilizando el Selenium Manager, pero no será vista en esta materia debido a que aún está en Beta y puede no funcionar correctamente. Se puede consultar la documentación de Selenium si se desea aprender al respecto.

Driver Management Software


La mayoría de las computadoras actualizan los navegadores de forma automática, pero no necesariamente los drivers de cada navegador. Para asegurarnos que tenemos la versión correcta del driver para nuestro navegador tenemos varias librerías de terceros que pueden ayudarnos.

Podemos importar el WebDriver Manager Package (https://github.com/rosolko/WebDriverManager.Net ) agregando las siguientes dependencias en nuestro código:

using WebDriverManager; 

using WebDriverManager.DriverConfigs.Impl;

Luego usamos el método SetUpDriver() para configurarlo como corresponde:

new DriverManager().SetUpDriver(new ChromeConfig()); 

var driver = new ChromeDriver();

La variable de entorno PATH


Esta opción requiere que descarguemos de manera manual el Driver, que podemos descargarla de la página oficial de Selenium haciendo clic aquí. Esta opción nos da la flexibilidad de cambiar de lugar el driver en nuestra computadora sin la necesidad de actualizar nuestro código. También funcionará en múltiples computadoras sin necesidad que cada una de ellas tenga el driver almacenado en exactamente el mismo lugar. Si vamos a trabajar con código que luego saldrá a producción, esta es la mejor opción.

Podemos agregar los drivers en un directorio que ya está listado en el PATH o podemos agregarlo a un directorio cualquiera y luego agregarlo al PATH.

Si estamos programando en Windows, podemos ver los directorios que ya se encuentran en PATH abriendo el Command Prompt y ejecutando el comando:

echo %PATH%

Si la ubicación del driver no está aún en el PATH, podemos agregarlo al PATH con el comando:

setx PATH "%PATH%;C:\WebDriver\bin"

Podemos probar que hayamos agregado correctamente el driver al PATH ejecutando el comando (este es un ejemplo para el Driver de Chrome/Chromium):

chromedriver.exe

Si el PATH se configuró correctamente veremos la siguiente salida en la terminal:

Starting ChromeDriver 95.0.4638.54 (d31a821ec901f68d0d34ccdbaea45b4c86ce543e-refs/branch-heads/4638@{#871}) on port 9515 Only local connections are allowed. Please see https://chromedriver.chromium.org/security-considerations for suggestions on keeping ChromeDriver safe. ChromeDriver was started successfully.



Hardcodear la ubicación del Driver


Ya hemos visto en clases anterior que hardcodear ubicaciones en el código no siempre es lo mejor ya que puede traernos problemas a la hora de utilizar nuestro software en distintas máquinas, pero si sabemos que el driver estará siempre en la misma ubicación, es una opción fácil de utilizar y no necesitamos meternos en problemas con los archivos de configuración del Sistema Operativo.

Sólo debemos establecer una variable global con la ubicación del Driver que hemos descargado manualmente:

var driver = new ChromeDriver(@"C:\WebDriver\bin");

Este ejemplo muestra el driver de Chrome/Chromium, pero puede hacerse con todos los drivers.



Escribir nuestro primer script de Selenium


Ahora que hemos instalado Selenium y nos aseguramos que tenemos los Drivers instalados, podemos comenzar a escribir código de Selenium para comenzar a automatizar nuestras pruebas.

Básicamente, lo único que hace Selenium es enviarle comandos al navegador para que haga algo o le envía pedidos de información. La mayoría de lo que hace Selenium es una combinación de 8 comandos básicos que deberemos conocer para trabajar.

Comenzar la sesión: Comenzar y detener una sesión en Selenium sirve, básicamente, para abrir y cerrar el navegador que vamos a utilizar. Antes de comenzar a probar debemos abrir una sesión para habilitar el navegador en el que vamos a probar nuestro software. Aprenderemos mas sobre las sesiones más adelante, pero la forma básica de abrir una sesión es el siguiente comando:

IWebDriver driver = new ChromeDriver();

Realizar acciones del navegador: Podemos realizar muchas acciones del navegador con Selenium para ejecutar el código en el que estamos trabajando. En el ejemplo a continuación se muestra una navegación a una página web:

driver.Navigate().GoToUrl("https://www.selenium.dev/selenium/web/web-form.html");

Solicitar información del navegador: El navegador propiamente dicho tiene mucha meta información respecto a lo que está mostrando. No solo tiene la información de nuestro software sino que tiene también información de título, tamaño del browser, posición del browser, cookies, alerts, etc. Con Selenium podemos pedir toda la información relacionada al navegador que se nos ocurra. Por ejemplo:

var title = driver.Title;

Establecer una estrategia de espera: Sincronizar el código con el estado actual del navegador es uno de los mayores desafíos con Selenium, y hacerlo bien es un tema avanzado.

Esencialmente queremos asegurarnos que el elemento que necesitamos se encuentra en la página que tenemos abierta antes de intentar localizarlo, así como también que ese elemento está en un estado en el cual podemos interactuar con él.

Una espera implícita suele ser una mala solución a este problema, ya que dependemos de que las cosas se hagan solas. Sin embargo es la más fácil de demostrar, así que es el ejemplo que damos a continuación:

driver.Manage().Timeouts().ImplicitWait = TimeSpan.FromMilliseconds(500);

Más adelante en la materia estudiaremos otras estrategias de espera.

Encontrar un elemento: La mayoría de los comandos que se utilizan en la mayoría de las sesiones de Selenium están relacionados a los elementos. Pero no podemos interactuar con un elemento si no lo encontramos primero. Por ejemplo:

var textBox = driver.FindElement(By.Name("my-text"));

var submitButton = driver.FindElement(By.TagName("button"));

Realizar una acción sobre un elemento: Para realizar pruebas con Selenium, ejecutaremos acciones sobre los elementos. La cantidad de acciones existentes no son muchas pero si se usan de manera frecuente. Más adelante en la materia aprenderemos más sobre este componente de Selenium.

textBox.SendKeys("Selenium");

submitButton.Click();

Solicitar información del elemento: Los elementos de una página web pueden almacenar mucha información a la que necesitaremos acceder. Tenemos muchas formas de acceso, un ejemplo es:

var value = message.Text;

Finalizar la sesión: Así como abrimos una sesión para comenzar a trabajar, debemos finalizar la sesión para liberar recursos. Una vez que finalizamos la sesión se cierra el navegador abierto, por lo que no se le puede enviar ningún comando.

driver.Quit();

Driver Sessions


Como ya hemos mencionado anteriormente, comenzamos y finalizamos una sesión para abrir y cerrar un navegador. Una sesión se crea de forma automática cuando inicializamos un nuevo objeto de tipo Driver.

Cada lenguaje de programación tiene distintas opciones de constructor para describir el tipo de sesión que queremos. La sesión por defecto es de tipo local. Las sesiones pueden ser de tipo local (es decir que el navegador se abre en nuestra propia computadora) o de tipo remota (es decir que el navegador se abre en otra computadora distinta).

Estudiaremos ahora sobre algunas opciones compartidas por todos los navegadores. Cada navegador tiene, además de las genéricas, opciones específicas. Se recomienda consultar la documentación de Selenium (haciendo clic aquí) para encontrar las opciones específicas al navegador con el que se está trabajando.



Opciones del navegador


Veamos ahora algunas opciones compartidas por todos los navegadores.

browserName


Esta opción se utiliza para definir el nombre del navegador que estamos utilizando para una sesión dada. Si el navegador especificado no está instalado en la máquina remota (en el caso de una sesión remota), entonces fallará la creación de la sesión.

browserVersion


Esta opción no es obligatoria, se utiliza para definir la versión del navegador a utilizar en el escritorio remoto. Es importante notar, que si se especifica la versión, debemos asegurarnos que esa versión específica sea la que está instalada en la computadora remota en la que estaremos haciendo pruebas. Si esto no es así, entonces la creación de la sesión fallará.

pageLoadStrategy


Existen 3 estrategias para la carga de la página. Esta opción se refiere al estado de carga de la pagina y son alguno de los 3 que veremos a continuación.

Las distintas estrategias de carga de la página se utilizan, más que nada, para gestionar la velocidad de carga de recursos. Puede ocurrir que una página sea muy pesada y tarde mucho en cargar, es por eso que debemos gestionar la estrategia de carga.

normal (default)


La página se carga de manera completa. Es la opción por defecto y espera a que carguen todos los recursos. El WebDriver espera a que cargue toda la página para permitir interactuar sobre la página.




Otras opciones


Continuamos mostrando opciones compartidas por todos los navegadores.

eager


La carga es interactiva. El acceso al DOM está listo, pero otros recursos como imágenes o videos puede que aún estén cargando cuando se habilite el uso del WebDriver.


none


No se espera a que se cargue ningún recurso de la página web para permitir el uso del WebDriver.



platformName


Se utiliza para identificar el sistema operativo de la máquina remota. Buscar el atributo platformName retorna el nombre de dicho sistema operativo.

acceptInsecureCerts


Esta opción revisa si se utiliza un Certificado TLS expirado o inválido al navegar en la sesión abierta. Si la opción está setteada en False, un certificado inseguro lanzará un error en cualquier encuentro con cualquier dominio con un certificado problemático. Si está setteado en True, un certificado inválido sería aceptado por el navegador sin problemas.


Timeouts


Una sesión de WebDriver viene con un session timeout predeterminado. Un timeout es el tiempo de espera que tiene el navegador hasta dar una respuesta, si la petición toma más tiempo que el definido en el timeout entonces el navegador hace que salte un error.

Hay 3 tipos de timeouts posibles.

Script Timeout


Especifica cuándo interrumpir un script que se está ejecutando en el navegador actualmente. El tiempo por defecto es de 30000 y se asigna cuando se crea una nueva sesión del WebDriver.

Page Load Timeout


Especifica el tiempo en el que una página debe cargarse en el contexto de navegación actual. El tiempo por defecto de este timeout es de 300000 y se asigna también cuando se crea una nueva sesión del WebDriver. Si una página aún no se termina de cargar y se llega al límite del Timeout, la carga será interrumpida y se lanzará una TimeoutException.

Implicit Wait Timeout


Con esto especificamos el tiempo a esperar para la estrategia de ubicación de elementos implícita (que estudiaremos la semana que viene). El tiempo por defecto es de 0 y se asigna cuando se crea una sesión.



Proxy


Un servidor proxy actúa como intermediario para las peticiones entre un/a cliente y un servidor. Básicamente, el tráfico fluye a través del servidor proxy cuando intenta viajar entre el/la cliente y el servidor.

Un servidor proxy para la automatización de scripts con Selenium puede ser útil para:

Capturar el tráfico en la red.

Mockear llamadas al backend hechas por el sitio web.

Acceder al sitio web requerido bajo topologías de red complejas o políticas organizacionales estrictas.

Remote WebDriver


Podemos utilizar el WebDriver de Selenium de forma remota de la misma manera que lo utilizamos localmente. La principal diferencia es que el WebDriver remoto debe ser configurado para que pueda correr las pruebas en una máquina separada.

Un WebDriver remoto se compone de 2 partes: un/a cliente y un servidor. El/la cliente son nuestras pruebas de WebDriver y el servidor es simplemente un servlet de Java que nos permitirá hostear cualquier navegador moderno.

Para ejecutar un WebDriver remoto necesitamos conectarnos a un RemoteWebDriver. Hacemos esto apuntando a la URL del servidor a donde estaremos corriendo las pruebas. Para poder configurar la conexión, utilizamos las opciones que vimos en el apartado anterior.

Aquí podemos ver un ejemplo de instanciar un WebDriver remoto apuntando a un web server remoto con la URL www.example.com, corriendo nuestras pruebas en Firefox.
